场景场景引入

你正在写一个 Python 脚本，希望调用一个 AI 来帮用户总结文章。你选择了 Ollama，因为它免费、本地运行、不需要联网。但在写代码之前，你需要理解这个"黑盒"里面到底在发生什么——这样当输出结果不理想时，你才知道该调哪个参数。

起源从 RNN 到 Transformer：一场 NLP 的革命

要理解大模型，我们得先回到 2017 年之前，看看那时的自然语言处理（NLP）是什么样子。

在 2017 年以前，处理序列数据（如一句话、一段文字）的主流技术是循环神经网络（RNN）和它的改进版 LSTM/GRU。RNN 的核心思想是"逐个读、逐个记"：从第一个字开始，读一个字更新一次记忆，再读下一个字，再更新……就像一个人读文章，只能一个字一个字地读，而且记忆力有限，读到后面就忘了前面。

RNN 有个致命问题：长距离依赖困难。假设文章开头提到"小明"，结尾问"他去了哪里？"，RNN 很可能已经忘了"他"指的是"小明"。LSTM 通过"门控机制"缓解了这个问题，但没有从根本上解决。

2017 年 6 月，Google 机器翻译团队的 Ashish Vaswani 等人在论文《Attention Is All You Need》中提出了 Transformer 架构。这篇论文的标题本身就极具宣言性——它宣称：以前人们用各种复杂的循环结构来建模序列，其实根本没必要，只要一个"注意力机制"就够了。

Transformer 发布后，迅速横扫了 NLP 领域的各项基准测试。2018 年的 BERT（Google）和 GPT-1（OpenAI），2019 年的 GPT-2，2020 年的 GPT-3……它们都是基于 Transformer 的变体。大语言模型（LLM）时代就此开启。

思考为什么 Transformer 能解决"遗忘"问题？

想象你在读一篇很长的论文。RNN 就像一个只能逐行阅读且记忆力有限的人，读到第 50 页时，第 1 页的内容已经模糊了。而 Transformer 的做法是：直接翻到任意一页，同时看所有页，并根据当前问题决定哪些页最重要。

具体来说，Transformer 的核心洞察是：每个词的位置都应该能直接"看到"其他所有词的位置，而不是像 RNN 那样只能看到前面的词。这叫做"全局自注意力"（Global Self-Attention）。

举个例子，对于句子"猫坐在垫子上，因为它很温暖"：

  • RNN 读到"它"时，需要一层一层地把信息从"垫子"传过来，传得越远信息越失真。
  • Transformer 读到"它"时，直接计算"它"和句子中每个其他词的"关联分数"。分数最高的可能是"垫子"，于是模型立刻知道"它"指的是"垫子"。

这种"全局可见"的设计，让 Transformer 在处理长文本时具有天然优势。上下文窗口从早期的 512 Token，发展到今天的 128K 甚至 1M Token。

但 Transformer 也有代价：计算量巨大。RNN 逐个处理，内存占用稳定；Transformer 需要同时计算所有词之间的关联，内存和计算复杂度随序列长度平方增长。这个 trade-off（权衡）是理解大模型硬件需求的关键。

原理核心原理

Transformer：大模型的骨架

2017 年 Google 提出的 Transformer 架构，是目前几乎所有大语言模型的基础。它的核心思想是：把输入的每一句话拆成小块，然后让每个小块都能看到其他所有小块，根据相关性决定应该关注谁。

想象你在读一句话："猫坐在垫子上，因为它很温暖。" 当读到"它"的时候，你需要知道"它"指的是"垫子"而不是"猫"。Transformer 的注意力机制就是干这个的——它让每个词都能"看到"上下文中的所有其他词，并计算彼此之间的关联度。

注意力机制的本质：给每个词一个"聚光灯"，让它能照亮上下文中最重要的其他词。关联度越高，光线越强。

Token：模型的"单词"

模型并不直接处理人类眼中的"字"或"词"。它把文本拆成更小的单位，叫做 Token。

  • 英文：一个 Token 大约相当于 0.75 个单词，"Hello world" ≈ 2~3 个 Token
  • 中文：一个汉字通常占 1~2 个 Token，"你好世界" ≈ 4~6 个 Token

Token 决定了模型的"视野范围"。一个模型的"上下文窗口"（比如 4096 Token）意味着它一次最多能同时"记住"这么多 Token。超出这个范围，前面的内容就会被遗忘。

记忆边界：上下文窗口就像一张固定大小的纸，写满了就得擦除最上面的内容。所以长文档需要分段处理，不能直接丢给模型。

参数规模：模型的"脑容量"

你常听到"70 亿参数"、"700 亿参数"，这些数字是什么意思？

参数就是模型内部可调节的"旋钮"数量。每个旋钮存储一个浮点数，表示某种连接的强度。参数越多，模型能学到的模式越复杂，但同时需要的计算资源和内存也越大。

规模 典型模型 显存需求 适用场景
7B Llama 3 8B、Qwen 2.5 7B ~8 GB 个人电脑、快速原型
13~14B Qwen 2.5 14B ~16 GB Mac M 系列、中高端 PC
70B+ Llama 3 70B、DeepSeek V3 ~80 GB+ 服务器、专业工作站

Ollama 支持量化（Quantization），可以把模型压缩到更小的体积。比如一个 70B 模型经过 4-bit 量化后，显存需求从 140GB 降到约 40GB，代价是精度轻微下降。

生成过程：一个 Token 一个 Token 地"写"

当你向模型提问时，它并不是一瞬间输出整段答案。它的内部过程是：

  1. 把你的输入拆成 Token，送入模型
  2. 模型计算出下一个最可能的 Token 是什么
  3. 把这个 Token 追加到输出末尾
  4. 重复步骤 2~3，直到遇到结束标记或达到最大长度

这个过程叫做自回归生成——每次只预测下一个字，然后把预测结果作为新的输入继续预测。就像一个人写文章，每写一个字就要停下来想一想下一个字该写什么。

自注意力机制的矩阵运算详解

注意力机制的核心，是将输入序列中的每个 Token 映射为三个向量：查询 Query（Q）、键 Key（K）、值 Value（V）。假设输入是一个序列的嵌入矩阵 $X \in \mathbb{R}^{n \times d}$，其中 $n$ 为序列长度，$d$ 为嵌入维度：

$$Q = X W_Q, \quad K = X W_K, \quad V = X W_V$$

$W_Q, W_K, W_V \in \mathbb{R}^{d \times d_k}$ 是可学习的投影矩阵。注意力分数通过 Q 和 K 的点积计算，再经过 softmax 归一化，得到注意力权重矩阵 $A$：

$$A = \text{softmax}\left( \frac{Q K^\top}{\sqrt{d_k}} \right)$$

除以 $\sqrt{d_k}$ 是为了防止点积值过大导致 softmax 梯度消失。最终输出是权重矩阵与 V 的乘积：

$$\text{Attention}(Q,K,V) = A V = \text{softmax}\left( \frac{Q K^\top}{\sqrt{d_k}} \right) V$$

$A \in \mathbb{R}^{n \times n}$ 是一个方阵，$A_{ij}$ 表示第 $i$ 个 Token 对第 $j$ 个 Token 的"关注程度"。自注意力的关键特性：任意两个 Token 的交互距离都是常数 O(1)，不受序列长度影响——这正是 Transformer 解决长距离依赖的数学基础。

位置编码的物理直觉

自注意力机制本身对 Token 的顺序是"失明"的：$A_{ij}$ 的计算完全不依赖于 $i$ 和 $j$ 的位置。为了让模型感知序列顺序，Transformer 引入了位置编码（Positional Encoding）。

原始 Transformer 使用正弦/余弦函数生成位置编码向量 $PE_{(pos, 2i)} = \sin(pos / 10000^{2i/d})$，$PE_{(pos, 2i+1)} = \cos(pos / 10000^{2i/d})$。这种设计的物理直觉是：

  • 不同频率的"时钟"：每个维度对应一个不同频率的正弦波。低频维度感知"远距离"位置关系，高频维度感知"近距离"位置差异。
  • 相对位置可线性表达：利用三角恒等式，$\sin(\alpha+\beta)$ 可以表示为 $\sin\alpha$ 和 $\cos\beta$ 的线性组合，这使得模型能自然学习"相对位置"信息。
  • 外推性：正弦编码是连续函数，对训练时未见过的更长序列具有一定的外推能力。

现代大模型（如 GPT、Llama）则改用可学习的位置嵌入（Learnable Positional Embeddings）或旋转位置编码（RoPE），但核心目标一致：让模型知道"谁在前面、谁在后面"。

论证为什么 Transformer 是"对的"架构？

你可能会问：既然 Transformer 计算量这么大，为什么它还能取代 RNN 成为主流？我们需要从多个维度来论证。

维度 1：并行计算能力

RNN 必须逐个处理序列中的每个元素，第 10 个词的计算必须等第 9 个词算完。这种"串行"特性让 RNN 很难利用现代 GPU/TPU 的并行计算能力。而 Transformer 的自注意力机制可以同时计算所有词对之间的关联——整个序列一次性处理，完美契合 GPU 的大规模并行架构。在硬件层面，Transformer 是"GPU 友好"的。

维度 2：长距离依赖

RNN 中信息传递的距离越远，衰减越严重。LSTM 虽然有所缓解，但在实践中超过 100 个词的依赖关系就很难捕捉了。Transformer 的自注意力让任意两个词之间的距离都是"一步直达"——无论相隔多远，关联分数的计算方式完全相同。这使得 Transformer 能轻松处理数百甚至数千 Token 的长文本。

维度 3：可扩展性

大量研究表明，Transformer 的性能随着模型规模（参数数量）和数据量的增加而持续提升，呈现出可预测的"缩放定律"（Scaling Laws）。RNN 则很快遇到瓶颈：增大网络深度会导致梯度消失/爆炸，训练变得极不稳定。Transformer 的残差连接和层归一化让深层网络的训练变得可行，于是才有了今天百亿、千亿参数的"大"模型。

核心论证：Transformer 不是"完美"的架构（它的计算复杂度确实很高），但它在并行性、长距离建模和可扩展性三个关键维度上取得了最佳平衡。这三点恰好与现代硬件（GPU）和大规模数据训练的需求完美匹配，因此它成为了大语言模型的标准骨架。

对比总结：

特性RNN/LSTMTransformer
并行计算差（串行）好（完全并行）
长距离依赖弱（信息衰减）强（一步直达）
训练稳定性差（梯度消失）好（残差连接）
扩展性有限优秀（Scaling Law）
计算复杂度O(n) 每步O(n²) 整体

要点核心要点

  • Transformer 的核心是注意力机制，它让模型理解词与词之间的关系
  • Token是模型处理文本的最小单位，不是人类眼中的"字"
  • 上下文窗口限制了模型一次能"记住"的内容长度
  • 参数规模决定了模型的复杂度和硬件需求，量化可以降低门槛
  • 模型生成是逐 Token 预测的过程，不是一次性输出整段答案

实验动手体验 Token 和注意力

这些实验不需要写代码，用在线工具就能完成，帮助你直观感受大模型的内部工作方式。

实验 1：观察 Token 的分割方式

  1. 打开 OpenAI 的 Tokenizer 工具（搜索 "OpenAI Tokenizer"）。
  2. 输入一段中英文混合的文本，例如："Hello world，这是一个测试。"
  3. 观察每个 Token 的边界。你会发现：英文单词被拆成子词（如 "testing" → "test" + "ing"），中文汉字通常每个字一个 Token。
  4. 尝试输入 "苹果" 和 "Apple"，观察它们被编码为几个 Token。

预期现象：不同的词被编码为不同数量的 Token。常见英文词可能是 1 个 Token，生僻词或拼写错误可能被拆成多个。中文通常 1~2 个 Token 一个汉字。

实验 2：体验上下文窗口的限制

  1. 打开任意一个在线大模型对话界面（如 ChatGPT、Claude 等）。
  2. 复制粘贴一段很长的文章（例如 5000 字），然后问模型："请总结文章开头的第一个观点。"
  3. 如果文章超过了模型的上下文窗口，观察模型的反应（可能拒绝处理，或者只处理了后半部分）。

预期现象：如果文本超过上下文窗口，模型会"截断"内容。这就是为什么长文档需要先分段或提取关键部分再输入。

实验 3：感受模型的"逐字生成"

  1. 在任意大模型界面中，问一个需要推理的问题，例如："请详细解释为什么天空是蓝色的。"
  2. 仔细观察输出过程——模型是一个字一个字（实际上是 Token 一个 Token）地"吐"出来的。
  3. 注意看：有时候模型生成到一半会"反悔"，删掉刚写的半句话重新写。这就是自回归生成的特点。

预期现象：模型不会一次性写出完整答案。每个新 Token 的生成依赖于之前所有已生成的 Token。这种"一边想一边说"的特性，也是模型偶尔产生逻辑跳跃的原因。

实验 4：用 NumPy 手动实现单头注意力

通过编写简化版的自注意力代码，输入一个短句的嵌入矩阵，观察注意力权重矩阵 $A$ 的实际数值分布：

import numpy as np

def softmax(x):
    exp_x = np.exp(x - np.max(x, axis=-1, keepdims=True))
    return exp_x / np.sum(exp_x, axis=-1, keepdims=True)

def single_head_attention(X, W_Q, W_K, W_V):
    """
    X: 输入嵌入矩阵, shape (n_tokens, d_model)
    W_Q, W_K, W_V: 投影矩阵, shape (d_model, d_k)
    返回: 输出矩阵 O (n_tokens, d_k), 注意力权重 A (n_tokens, n_tokens)
    """
    Q = X @ W_Q      # (n, d_k)
    K = X @ W_K      # (n, d_k)
    V = X @ W_V      # (n, d_k)

    scores = Q @ K.T              # (n, n)
    scores = scores / np.sqrt(Q.shape[1])  # 缩放
    A = softmax(scores)           # (n, n) 注意力权重矩阵

    O = A @ V                     # (n, d_k)
    return O, A

# ========== 实验输入 ==========
# 假设我们有 4 个 token，每个 token 的嵌入维度为 3
# 用随机数初始化（实际中应来自词嵌入层）
np.random.seed(42)
n_tokens, d_model, d_k = 4, 3, 2
X = np.random.randn(n_tokens, d_model)

# 随机初始化投影权重（实际中通过训练学习）
W_Q = np.random.randn(d_model, d_k)
W_K = np.random.randn(d_model, d_k)
W_V = np.random.randn(d_model, d_k)

O, A = single_head_attention(X, W_Q, W_K, W_V)

print("注意力权重矩阵 A:")
print(np.round(A, 3))
print("\n每行的和应为 1.0:", A.sum(axis=1))

# 观察：对角线元素是否最大？这表示 token 最"关注"自己
print("\n对角线元素:", np.round(np.diag(A), 3))

实验任务：

  1. 运行上述代码，观察注意力权重矩阵 $A$ 的形状和数值。每一行的和是否接近 1？为什么？
  2. 将 X 的第 0 行和第 2 行设置为完全相同的向量，重新运行。$A_{0,2}$ 和 $A_{2,0}$ 是否变大？这说明什么？

错误常见错误

错误 1：以为模型"理解"了内容

模型并不真正"理解"语言，它只是在统计意义上学会了 Token 之间的关联模式。它可能会给出听起来很对但实际上错误的答案（幻觉）。

错误 2：忽略上下文窗口限制

把一本 10 万字的小说直接丢给上下文窗口只有 4096 Token 的模型，前面的内容会被截断。需要先分段或总结。

错误 3：认为参数越多越好

70B 模型在消费级电脑上跑不动。对于日常任务，7B 或 14B 的量化模型往往足够，速度还快得多。

示例代码示例

用 Python 大致估算一段文本的 Token 数量（简化版）：

def rough_token_count(text):
    """粗略估算 Token 数：中文按 1 字 ≈ 1.5 Token，英文按 1 词 ≈ 1.3 Token"""
    import re
    chinese_chars = len(re.findall(r'[\u4e00-\u9fff]', text))
    english_words = len(re.findall(r'[a-zA-Z]+', text))
    return int(chinese_chars * 1.5 + english_words * 1.3)

text = "Hello world，这是一个测试。"
print(f"估算 Token 数: {rough_token_count(text)}")
# 输出: 估算 Token 数: 9

实际 Token 数因分词器不同而有差异。Ollama 使用的模型有各自的分词器（如 Llama 用 SentencePiece，GPT 用 BPE）。
