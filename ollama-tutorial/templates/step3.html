Step 3 — 基础调用方式

Python requests / ollama 库、生成与聊天接口

场景

Ollama 已经安装好了，模型也拉下来了。现在你要写一个 Python 脚本，让它自动调用模型来回答用户的问题。你可以用两种方式：直接用 HTTP 请求，或者用 Ollama 官方提供的 Python 库。

起源REST API：互联网通信的通用语言

在理解 Ollama 的 API 之前，让我们先了解一下 REST API 的历史。

2000 年，计算机科学家 Roy Fielding 在他的博士论文中提出了 REST（Representational State Transfer，表述性状态转移）架构风格。Fielding 也是 HTTP 协议的主要设计者之一。REST 的核心思想是：互联网上的一切资源（如用户、文章、模型）都可以通过统一的 URL 来标识，通过标准的 HTTP 方法（GET、POST、PUT、DELETE）来操作。

REST API 之所以成为现代互联网服务的标准，是因为它具有简单、通用、无状态的特性。无论你是用 Python、JavaScript、Java 还是任何其他语言，只要你能发送 HTTP 请求，就能调用 REST API。这使得不同的系统之间可以方便地互相通信。

Ollama 选择 REST API 作为接口，是一个深思熟虑的设计决策：它让 Ollama 不仅是一个命令行工具，更是一个可以被任何编程语言调用的"语言模型服务"。

思考封装还是不封装？

当你面对一个 HTTP API 时，有两个选择：直接用底层的 HTTP 库（如 Python 的 requests），或者用官方提供的高层封装库（如 ollama Python 库）。怎么选？

直接用 requests 的理由：

  • 透明：你能看到每一个 HTTP 头、每一个状态码、每一个 JSON 字段。
  • 灵活：你可以自由添加代理、自定义超时、处理重试逻辑、打印调试信息。
  • 无依赖：不需要安装额外的库，减少项目的依赖 footprint。

用官方库的理由：

  • 简洁：几行代码就能完成调用，不需要处理 URL、JSON 解析等重复劳动。
  • 一致性：API 命名与 Ollama CLI 保持一致，降低学习成本。
  • 内置功能：流式输出的解析、错误类型的封装，官方库已经帮你做好了。

这其实是软件工程中一个经典的权衡：控制 vs 便利。当你需要快速原型验证时，官方库更高效；当你需要深度定制或构建生产级系统时，直接用 HTTP 请求更可控。很多经验丰富的开发者会在项目中同时保留两种能力——用官方库快速开发，用 requests 处理特殊场景。

原理

方式一：直接用 requests

Ollama 的 API 就是普通的 HTTP REST 接口，任何能发 HTTP 请求的库都能调用。这是最底层、最灵活的方式。

import requests

url = "http://localhost:11434/api/generate"

data = {
    "model": "qwen2.5:7b",
    "prompt": "请用中文解释什么是 HTTP 请求",
    "stream": False
}

response = requests.post(url, json=data)
result = response.json()
print(result["response"])

关键点：

  • json=data 会自动把字典转成 JSON 并设置 Content-Type: application/json
  • stream: False 表示一次性返回完整结果，适合快速测试
  • 响应中的 response 字段就是模型生成的文本

方式二：用 ollama 官方库

Ollama 提供了一个 Python 库，封装了 HTTP 调用的细节，代码更简洁。

# 安装
pip install ollama

# 调用
import ollama

response = ollama.generate(
    model="qwen2.5:7b",
    prompt="请用中文解释什么是 HTTP 请求"
)
print(response["response"])

官方库的优势：

  • 不需要自己处理 URL、HTTP 状态码、JSON 解析
  • API 与 Ollama CLI 的命令命名一致，好记
  • 自动处理流式输出的解析（后面会讲）

劣势：如果你需要自定义 HTTP 行为（比如代理、超时、重试），官方库反而不如 requests 灵活。

generate vs chat：两个核心接口

Ollama 提供两个主要的生成接口，用途不同：

接口 适用场景 特点
/api/generate 单轮文本补全、续写、翻译 只传一个 prompt，模型补全后面的内容
/api/chat 多轮对话、聊天机器人 传消息列表，模型理解对话历史

/api/chat 的消息格式（OpenAI 兼容）：

messages = [
    {"role": "system", "content": "你是一个乐于助人的助手"},
    {"role": "user", "content": "你好"},
    {"role": "assistant", "content": "你好！很高兴见到你。"},
    {"role": "user", "content": "今天天气怎么样？"}
]

response = ollama.chat(
    model="qwen2.5:7b",
    messages=messages
)
print(response["message"]["content"])

每条消息有三个角色：

  • system：给模型设定全局行为规则（比如"你是一个严谨的程序员"）
  • user：用户的输入
  • assistant：模型的回复（用于多轮对话时提供历史上下文）

要点

  • 两种调用方式：底层 requests（灵活）和高层 ollama 库（简洁）
  • /api/generate 适合单轮补全，/api/chat 适合多轮对话
  • 消息列表中的 role 决定这段话的"身份"：system、user、assistant
  • system 提示是控制模型行为的有力工具，后面会详细讲

论证为什么 generate 和 chat 要分开？

Ollama 提供了两个核心接口——/api/generate 和 /api/chat。为什么需要两个接口，而不是一个"万能"接口？这背后是对不同应用场景的精准划分。

generate 的设计哲学：简单补全

generate 的本质是"文本补全"（completion）。你给它一个开头，它帮你续写。这种模式源自最早的 GPT 模型——它就是一个强大的"自动补全引擎"。generate 的场景包括：续写文章、翻译单句、代码补全、生成摘要。这些任务不需要"记忆"之前的对话，只需要根据当前 prompt 生成内容。

chat 的设计哲学：状态ful 的对话

chat 的本质是"对话管理"。它接收一个消息列表，每个消息有角色（system/user/assistant），模型需要理解整个对话的上下文，并给出合适的回复。这种模式模拟了人类的对话方式：你说的每句话都建立在之前所有对话的基础上。

为什么不能合并？

技术上当然可以合并，但分开设计有几个好处：

  1. 语义清晰：generate 是"补全"，chat 是"对话"。接口名称本身就告诉开发者该怎么用。
  2. 协议兼容：chat 的消息格式与 OpenAI 的 API 兼容，这让迁移成本极低。如果你之前用 OpenAI 的 API，切换到 Ollama 只需要改一个 base URL。
  3. 性能优化：generate 不需要解析消息列表，内部处理更简单。对于纯补全任务，它比 chat 稍微高效一点。

核心论证：接口设计不是技术问题，而是用户体验问题。好的 API 设计让开发者一眼就能知道该用哪个接口，减少犯错的可能。generate 和 chat 的分离，正是这种设计哲学的体现。

实验动手对比两种调用方式

这个实验将让你亲身体验 requests 和 ollama 库的差异，并理解 HTTP 请求的具体过程。

实验 1：用 requests 发送原始 HTTP 请求

  1. 打开 Python 交互式环境（python3）或 Jupyter Notebook。
  2. 运行以下代码：
import requests, json

url = "http://localhost:11434/api/generate"
data = {
    "model": "qwen2.5:1.5b",
    "prompt": "用一句话解释 HTTP 请求",
    "stream": False
}

response = requests.post(url, json=data)
print("状态码:", response.status_code)
print("响应头:", dict(response.headers))
print("响应体:", json.dumps(response.json(), indent=2, ensure_ascii=False))
  1. 观察状态码（应该是 200）、响应头中的 Content-Type（应该是 application/json），以及响应体中的各个字段。

实验 2：用 ollama 库调用同一接口

  1. 安装 ollama 库：pip install ollama。
  2. 运行以下代码：
import ollama

response = ollama.generate(
    model="qwen2.5:1.5b",
    prompt="用一句话解释 HTTP 请求"
)
print(type(response))
print(response["response"])
  1. 对比两种方式的代码量、可读性和返回数据的结构差异。

实验 3：观察 chat 接口的消息处理

  1. 运行以下代码：
import ollama

messages = [
    {"role": "system", "content": "你是一个幽默的助手"},
    {"role": "user", "content": "讲个程序员笑话"}
]

response = ollama.chat(model="qwen2.5:1.5b", messages=messages)
print(response["message"]["content"])
  1. 尝试修改 system 消息的内容（如"你是一个严肃的教授"），再次运行，观察输出风格的变化。
  2. 在 messages 列表中追加一轮对话历史，再发送新消息，验证模型是否能记住上下文。

常见错误

错误 1：忘记设置 stream=False，结果被流式响应搞混

Ollama 默认是流式输出（stream=True）。如果直接用 response.json() 解析，会报错。要么显式设置 stream=False，要么用循环读取流。

错误 2：把 generate 和 chat 混用

generate 只接受 prompt 字段；chat 只接受 messages 字段。传给 generate 一个 messages 列表会报错。

错误 3：模型名写错或模型未下载

如果收到 model not found，先运行 ollama list 确认本地有这个模型，再检查名称拼写和大小写。

示例：封装一个调用函数

实际项目中，你不会每次都写一遍完整的请求代码。封装一个函数是更好的做法：

import ollama

def ask_model(prompt, model="qwen2.5:7b"):
    """向本地 Ollama 模型提问，返回生成的文本"""
    try:
        response = ollama.generate(
            model=model,
            prompt=prompt,
            stream=False
        )
        return response["response"]
    except ollama.ResponseError as e:
        return f"调用失败: {e.error}"
    except Exception as e:
        return f"未知错误: {str(e)}"

# 使用
answer = ask_model("用一句话总结 Transformer 的核心思想")
print(answer)

封装的好处：

  • 调用方只需要关心 prompt 和返回的文本
  • 错误处理集中在一处
  • 以后如果要切换模型或加参数，只需要改这个函数
