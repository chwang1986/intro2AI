场景场景引入

你已经掌握了 Ollama 的所有核心用法。现在的问题是：怎么把这些能力变成真正有用的功能？比如自动从一段文字里提取关键信息、帮用户写代码、或者让模型返回可以被程序直接解析的数据结构。

起源从 Prompt 到 Agent：大模型应用的演进

大模型应用的发展，经历了三个主要阶段。

阶段 1：直接 Prompt（2020~2022）

最早的用法很简单：直接把问题丢给模型，看它怎么回答。这种"零样本"（zero-shot）的方式对于通用问题效果不错，但对于专业任务，输出往往不稳定、格式不一致。

阶段 2：结构化 Prompt 与 Few-Shot（2022~2023）

研究者们发现，如果在 prompt 中给出示例（few-shot learning），模型的表现会显著提升。同时，通过在 prompt 中明确要求输出格式（如"请用 JSON 返回"），可以让模型的输出更容易被程序解析。这就是我们今天要学习的"结构化输出"技术的前身。

阶段 3：RAG 与 Agent（2023~至今）

当模型需要回答基于私有数据的问题时，prompt 里放不下整个数据库。于是出现了 RAG（检索增强生成）：先把问题向量化，去数据库里检索最相关的文档片段，再把片段和问题一起送给模型。更进一步的 Agent 架构，让模型可以调用外部工具（如搜索引擎、计算器、代码执行器），实现真正的"自主任务执行"。

本教程的前五步，帮你完成了阶段 1 和 2 的基础能力建设。下一步，你可以探索 RAG 和 Agent——但那需要向量数据库、工具调用框架等更多知识，留待后续教程。

思考怎么把"聊天"变成"工具"？

聊天很有趣，但真正的价值在于解决实际问题。作为一个开发者，你需要思考：怎么把模型的"语言能力"封装成可以被其他程序调用的"功能模块"？

关键思路是：输入结构化，输出结构化。

人类喜欢自然语言的自由表达，但程序喜欢确定性的数据结构。如果你让模型"随便说说"，程序很难处理；但如果你让模型"严格按 JSON 格式返回"，程序就能直接解析、存储、展示。

另一个关键思路是角色分工。不同的任务需要不同的模型能力：文本摘要需要理解能力，代码生成需要逻辑能力，信息提取需要结构化能力。通过 system 提示和模型选择，你可以把同一个"大脑"调教成不同的"专家"。

最后一个思路是健壮性设计。模型不是确定性的程序，它偶尔会出错、会幻觉、会返回非法格式。你的程序必须假设"模型可能会搞砸"，并设计相应的错误处理机制。try/except、格式校验、Fallback 策略，都是生产环境中必不可少的。

原理核心原理

文本生成与摘要

最基础的应用：让模型帮你写东西。但"帮我写"这个指令太模糊。好的 prompt 要包含角色、任务、格式、约束四个要素。

prompt = """你是一位技术文档编辑。

任务：把下面的发布说明改写成面向普通用户的版本，用 3 句话概括核心变化。

原文：
本次更新重构了数据持久层，引入了连接池复用机制，将查询延迟降低了 40%。同时修复了并发场景下的竞态条件，升级了 ORM 依赖至 3.2 版本。

要求：
- 不要出现技术术语（如 ORM、连接池、竞态条件）
- 每句话不超过 20 个字
- 语气轻松自然
"""

response = ollama.generate(model="qwen2.5:7b", prompt=prompt, stream=False)
print(response["response"])

这个 prompt 为什么有效：

  • 角色：技术文档编辑（设定了专业视角）
  • 任务：改写发布说明（明确要做什么）
  • 格式：3 句话、每句不超过 20 字（限制了输出结构）
  • 约束：不出现技术术语、语气轻松（限定了内容边界）

代码辅助：让模型当你的编程搭档

模型最擅长的任务之一是代码相关的工作：写代码、解释代码、找 bug、优化性能。

prompt = """你是一位 Python 专家。请找出下面代码中的问题并给出修复版本。

代码：
def divide(a, b):
    return a / b

result = divide(10, 0)
print(result)

要求：
1. 指出问题所在
2. 给出修复后的完整代码
3. 解释为什么这样修复
"""

response = ollama.generate(model="deepseek-coder:6.7b", prompt=prompt)
print(response["response"])

代码任务的技巧：

  • 选择代码专用模型（如 deepseek-coder、codellama），比普通模型效果好很多
  • 在 prompt 里明确说"给出完整代码"，否则模型可能只给片段
  • 如果代码很长，先给关键部分，说"其他部分省略"

结构化输出：让模型返回 JSON

程序处理文本很困难，处理 JSON 很容易。如果能让模型直接返回 JSON，你的后续处理代码会简单很多。

Ollama 支持 JSON Mode：在请求中设置 format="json"，模型会强制输出合法的 JSON。

prompt = """从下面的用户评论中提取信息，返回 JSON 格式。

评论："这款手机电池很耐用，但相机在暗光下表现一般，价格 2999 元有点贵。"

返回格式：
{
  "sentiment": "positive/negative/mixed",
  "pros": ["优点1", "优点2"],
  "cons": ["缺点1", "缺点2"],
  "price_mentioned": true/false,
  "price": 数字或 null
}
"""

response = ollama.generate(
    model="qwen2.5:7b",
    prompt=prompt,
    format="json",  # 开启 JSON Mode
    stream=False
)

import json
data = json.loads(response["response"])
print(data["sentiment"])   # "mixed"
print(data["pros"])        # ["电池很耐用"]

JSON Mode 的使用要点：

  • 必须在 prompt 中给出期望的 JSON 结构示例，否则模型不知道你要什么字段
  • 设置 stream=False，流式输出不适合 JSON Mode
  • 返回的字符串用 json.loads() 解析，仍然需要 try/except 处理解析错误
  • 不是所有模型都完美支持 JSON Mode，如果输出格式不对，可以在 prompt 里加"必须严格按以上格式返回，不要添加额外文字"

嵌入向量：让文本变成数字

除了生成文字，模型还能把一段文本变成一串数字（向量）。这个向量的神奇之处在于：语义相近的文本，它们的向量距离也近。

import ollama

# 把两段话变成向量
vec1 = ollama.embeddings(model="nomic-embed-text", prompt="猫坐在垫子上")
vec2 = ollama.embeddings(model="nomic-embed-text", prompt="一只猫趴在毯子上")
vec3 = ollama.embeddings(model="nomic-embed-text", prompt="今天股市大涨")

# 计算向量相似度（余弦相似度）
import numpy as np

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

print(cosine_similarity(vec1["embedding"], vec2["embedding"]))  # ~0.95，很相似
print(cosine_similarity(vec1["embedding"], vec3["embedding"]))  # ~0.3，不相似

嵌入向量的应用场景：

  • 语义搜索：用户搜索"猫咪"，也能找到包含"小猫""喵星人"的文档
  • 文本分类：把文本向量聚类，自动分组
  • 推荐系统：找与用户历史兴趣向量最接近的内容

注意：生成嵌入需要专门的嵌入模型（如 nomic-embed-text、mxbai-embed-large），不要用生成文本的模型来做嵌入。

论证为什么"结构化输出"是工程化的关键？

在实际开发中，调用大模型的代码往往不是终点，而是整个数据流水线的中间环节。模型前面可能是用户输入、文件上传、数据库查询；模型后面可能是数据库存储、前端展示、下游分析。如果模型的输出是"自由文本"，这个流水线就会在每个接口处断裂。

论证 1：程序需要确定性

人类可以读懂"苹果的总部在加州库比蒂诺"，但程序需要 {"company": "Apple", "location": "Cupertino, CA"}。JSON Mode 的作用就是把"自然语言的不确定性"转化为"数据结构的可预测性"。

论证 2：多步骤流水线需要接口契约

假设你的应用流程是：用户上传文章 → 模型提取实体 → 程序去数据库查相关信息 → 模型生成回复 → 前端展示。如果"提取实体"这一步的输出格式每次都不一样，后续的"查数据库"步骤就会频繁报错。结构化输出相当于在模型和程序之间建立了一份"接口契约"。

论证 3：不同模型有不同专长

通用聊天模型（如 Qwen、Llama）擅长理解和生成自然语言，但它们的嵌入向量质量不如专门的嵌入模型（如 nomic-embed-text）。代码专用模型（如 DeepSeek Coder）在代码任务上表现远超通用模型。作为工程师，选择合适的工具做合适的事，是基本的职业素养。

核心论证：大模型不是魔法黑盒，而是可以被精确控制的工程组件。通过 prompt 工程、参数调节、格式约束和错误处理，你可以把模型的不确定性压缩到可接受的范围内，让它稳定地服务于生产环境。

要点核心要点

  • 好的 prompt = 角色 + 任务 + 格式 + 约束，四个要素越具体，输出越可控
  • 代码任务优先选代码专用模型（DeepSeek Coder、CodeLlama）
  • JSON Mode（format="json"）让模型返回结构化数据，但 prompt 里必须给格式示例
  • 嵌入向量把语义变成数字，是语义搜索和分类的基础
  • 不同任务选不同模型：聊天用通用模型、代码用代码模型、嵌入用嵌入模型

实验构建你的第一个实用工具

这个实验将引导你从零构建一个基于 Ollama 的实用工具，体验完整的开发流程。

实验 1：文本分类器

  1. 写一个 Python 脚本，定义三个类别："技术"、"娱乐"、"体育"。
  2. 使用 Ollama 的 chat 接口，构造 system 提示让模型扮演分类器：
import ollama

def classify(text):
    response = ollama.chat(
        model="qwen2.5:1.5b",
        messages=[
            {"role": "system", "content": "你是一个文本分类器。只输出类别名称：技术/娱乐/体育。不要解释。"},
            {"role": "user", "content": text}
        ],
        options={"temperature": 0.1}
    )
    return response["message"]["content"].strip()

# 测试
tests = [
    "苹果发布新款 iPhone",
    "某明星宣布结婚",
    "中国队获得世界杯冠军"
]
for t in tests:
    print(f"{t} -> {classify(t)}")
  1. 用 10 条不同领域的文本测试准确率，记录分类正确和错误的情况。

预期现象：对于界限清晰的文本，分类准确率通常很高（>80%）。但边界模糊的文本（如"某科技公司投资娱乐产业"）可能分类错误，这反映了语言本身的歧义性。

实验 2：JSON 信息提取

  1. 用本课示例中的 extract_info 函数处理 5 段不同的新闻文本。
  2. 检查返回的 JSON 是否合法，字段是否完整。
  3. 对于返回非法 JSON 的情况，打印原始输出，分析失败原因（通常是 prompt 不够明确或文本过于复杂）。

预期现象：简单、结构清晰的文本提取成功率高；复杂、含有多重嵌套信息的文本容易失败。低温度（0.1~0.3）能显著提升 JSON 格式的合规率。

实验 3：嵌入向量相似度搜索

  1. 拉取嵌入模型：ollama pull nomic-embed-text。
  2. 准备 5 个句子，涵盖不同主题。
  3. 计算两两之间的余弦相似度，找出最相似和最不相似的一对。
  4. 验证结果是否符合你的直觉。

预期现象：语义相近的句子（如"猫坐在垫子上"和"小猫趴在毯子上"）相似度通常在 0.8 以上；语义无关的句子相似度在 0.3 以下。这验证了嵌入向量能有效捕捉语义信息。

错误常见错误

错误 1：JSON Mode 返回了非法 JSON

即使开了 format="json"，模型仍可能输出带额外说明文字的 JSON。在 prompt 里强调"只返回 JSON，不要任何其他文字"。

错误 2：用生成模型做嵌入

qwen2.5:7b 是生成模型，不适合做嵌入。嵌入任务要用专门的嵌入模型，否则向量质量很差。

错误 3：prompt 太模糊导致输出不稳定

"帮我分析一下"这种 prompt，每次输出可能完全不同。加入角色、格式、约束后，输出会稳定很多。

示例代码示例

把前面学到的内容组合起来，做一个实际可用的信息提取工具：

{% raw %}import ollama
import json

def extract_info(text):
    """从任意文本中提取结构化信息"""
    prompt = f"""从下面的文本中提取关键信息，严格按 JSON 格式返回。

文本：{text}

返回格式：
{{
  "topic": "主题",
  "entities": ["实体1", "实体2"],
  "keywords": ["关键词1", "关键词2"],
  "summary": "一句话总结"
}}

只返回 JSON，不要任何其他文字。"""

    response = ollama.generate(
        model="qwen2.5:7b",
        prompt=prompt,
        format="json",
        options={"temperature": 0.2}  # 低温度，输出更稳定
    )

    try:
        return json.loads(response["response"])
    except json.JSONDecodeError:
        return {"error": "模型返回了非法 JSON", "raw": response["response"]}

# 使用
text = "苹果公司于 2024 年 9 月发布 iPhone 16，搭载 A18 芯片，起售价 799 美元。"
result = extract_info(text)
print(json.dumps(result, ensure_ascii=False, indent=2))
# {
#   "topic": "iPhone 16 发布",
#   "entities": ["苹果公司", "iPhone 16", "A18 芯片"],
#   "keywords": ["发布", "芯片", "售价"],
#   "summary": "苹果发布 iPhone 16，搭载 A18 芯片，售价 799 美元起"
# }
{% endraw %}

这个例子展示了如何把模型能力封装成一个可靠的函数：低温度保证稳定性、JSON Mode 保证格式、try/except 保证健壮性。
