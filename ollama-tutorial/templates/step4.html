场景场景引入

基础调用已经跑通了。但生产环境的要求更高：用户不想盯着空白屏幕等 10 秒才看到结果；你希望控制模型的"创造力"，让它有时严谨有时发散；你想做一个真正的聊天机器人，能记住之前说过的话。

起源从贪婪解码到采样策略

大模型生成文本的过程，本质上是一个概率选择过程。模型为词表中的每个候选词计算一个概率分数，然后从中选择一个作为下一个 Token。但"怎么选"这个问题，远比看起来复杂。

最早的文本生成模型采用贪婪解码（Greedy Decoding）：总是选择概率最高的词。这种方式确定性强，但输出往往单调、缺乏变化，而且容易陷入重复循环（如"好的好的好的……"）。

后来，研究者们引入了随机采样（Random Sampling）：按概率分布随机选择词。这样输出变得多样化，但问题是小概率词也可能被选中，导致生成内容不连贯甚至胡言乱语。

温度（Temperature）和 top-p（Nucleus Sampling）的参数就是在这种背景下提出的。2018 年，Ari Holtzman 等人在论文《The Curious Case of Neural Text Degeneration》中系统分析了各种解码策略，提出了 Nucleus Sampling（即 top-p），证明它在多样性和质量之间取得了最佳平衡。今天，temperature 和 top_p 已经成为所有大语言模型的标准配置。

思考为什么模型需要"创造力"控制？

想象你在和两个不同的"助手"合作。

第一个助手是法律顾问。当你问"这份合同有没有漏洞？"时，你希望他的回答是确定、准确、可验证的。如果他"随机发挥"，给出一个似是而非的答案，后果不堪设想。

第二个助手是创意作家。当你说"帮我想 10 个产品名字"时，你希望他的回答多样化、有新意、出人意料。如果他每次都给出同一个"最安全"的答案，就毫无价值。

同一个大模型，既要做法律顾问，又要做创意作家。它怎么知道该"严肃"还是该"发散"？答案是：它不知道，除非你告诉它。temperature 和 top_p 就是告诉模型"该怎么选词"的旋钮。

温度控制的是概率分布的"平坦程度"。低温 = 概率集中在头部几个词 = 输出保守确定；高温 = 概率分布平坦 = 小概率词也有机会被选中 = 输出多样化。这就是"创造力"的数学本质。

原理核心原理

流式输出：让用户先看到第一个字

默认情况下，模型生成完整答案后才一次性返回。对于长回答，用户可能要等好几秒。流式输出（Streaming）让模型每生成一个 Token 就发送一次，用户可以立刻看到第一个字逐渐"打字"出来。

开启方式：设置 stream=True（默认值）。

import ollama

stream = ollama.generate(
    model="qwen2.5:7b",
    prompt="写一首关于夏天的短诗",
    stream=True
)

# stream 是一个生成器，每次 yield 一个 Token
for chunk in stream:
    print(chunk["response"], end="", flush=True)

底层原理：Ollama 把 HTTP 响应的 Transfer-Encoding 设为 chunked，每个 chunk 是一个 JSON 行，包含刚生成的 Token。客户端边收边显示，就实现了"打字机效果"。

什么时候用流式：交互式应用（聊天机器人、AI 助手）必须用流式，让用户有反馈感。后台批处理任务（批量翻译、数据清洗）不需要流式，一次性返回更省事。

系统提示：给模型设定"人设"

system 消息是聊天接口中的第一条消息，它的作用是告诉模型"你是谁、你应该怎么说话、你的知识边界是什么"。系统提示在整个对话中始终有效。

messages = [
    {
        "role": "system",
        "content": "你是一位严谨的 Python 导师。回答时只给出代码和简短解释，不添加闲聊。所有代码必须包含类型注解。"
    },
    {"role": "user", "content": "写个函数，计算列表的平均值"}
]

response = ollama.chat(
    model="qwen2.5:7b",
    messages=messages
)
print(response["message"]["content"])

好的系统提示直接影响输出质量。写系统提示的几个原则：

  • 具体：不要写"你是一个有帮助的助手"，要写"你是一个 SQL 优化专家，回答时先指出原查询的问题，再给出优化后的代码"
  • 限定边界：告诉模型什么不该做（"如果你不确定，就说不知道，不要编造"）
  • 指定格式：如果需要结构化输出，提前说明格式（"用 Markdown 表格返回"）

温度与 top_p：控制"创造力"

模型生成下一个 Token 时，会给词表中的每个词算一个概率。温度和 top_p 就是控制"怎么从这个概率分布里选词"的两个旋钮。

温度（temperature）：控制概率分布的"平坦程度"。

  • temperature=0：总是选概率最高的词，输出最确定、最保守
  • temperature=0.7：默认值，概率分布适当平滑，有一定随机性
  • temperature=1.5：概率分布很平，小概率词也有机会被选中，输出更发散、更有创意

top_p（核采样）：不是从整个词表选，而是只从累积概率达到 p 的最小词集合里选。

  • top_p=0.1：只考虑最可能的 10% 的词，输出很集中
  • top_p=0.9：考虑最可能的 90% 的词，默认常用的值
# 严谨模式：代码生成、事实问答
response = ollama.generate(
    model="qwen2.5:7b",
    prompt="解释 Python 的 GIL",
    options={"temperature": 0.2, "top_p": 0.5}
)

# 创意模式：写诗、头脑风暴
response = ollama.generate(
    model="qwen2.5:7b",
    prompt="给一家奶茶店起 5 个有创意的名字",
    options={"temperature": 1.2, "top_p": 0.95}
)

经验法则：事实性任务用低温（0~0.3），创意性任务用高温（0.8~1.2）。top_p 通常和 temperature 配合使用，温度低时 top_p 也可以低一些。

多轮对话：维护对话历史

模型本身没有记忆。每次调用都是独立的——你第二次发送的消息，模型默认不知道你第一次说了什么。要实现"记住对话"，你必须自己把历史消息拼接好，一起发送。

class ChatBot:
    def __init__(self, model="qwen2.5:7b"):
        self.model = model
        self.messages = [
            {"role": "system", "content": "你是一个友好的助手"}
        ]

    def chat(self, user_input):
        # 1. 把用户的话加入历史
        self.messages.append({"role": "user", "content": user_input})

        # 2. 调用模型（带上完整历史）
        response = ollama.chat(
            model=self.model,
            messages=self.messages
        )

        # 3. 把模型的回复也加入历史
        reply = response["message"]["content"]
        self.messages.append({"role": "assistant", "content": reply})

        return reply

# 使用
bot = ChatBot()
print(bot.chat("你好"))
print(bot.chat("我刚才问了什么？"))  # 模型能回答"你好"，因为有历史记录

需要注意的问题：

  • 上下文窗口限制：消息越来越多，总有一天会超过模型的 Token 上限。需要实现"滑动窗口"或"摘要机制"，丢掉太旧的消息
  • 历史污染：如果之前对话中模型给了错误答案，这个错误会一直留在历史里影响后续回答

论证为什么这些"高级参数"有效？

我们使用的参数——流式输出、系统提示、温度、top_p——不是随意添加的功能，而是基于对模型行为和用户需求的深刻理解。

流式输出的论证：用户等待的心理学

研究表明，用户能容忍的"无反馈等待时间"约为 1 秒。超过 1 秒，用户会开始焦虑、怀疑系统是否卡住了。流式输出把"等待 10 秒看结果"变成了"1 秒内看到第一个字，然后持续有更新"，极大改善了主观体验。这在交互设计中叫做"进度可视化"——即使总时间不变，让用户"看到进展"就能显著提升满意度。

系统提示的论证：上下文控制的杠杆

系统提示之所以强大，是因为它被放在消息列表的最前面，在注意力机制的计算中拥有"先发优势"。模型在处理每个词时，都会计算它与所有其他词的关联分数。系统提示作为第一条消息，它的信息会被后续所有词反复"回顾"，因此影响贯穿整个生成过程。相比之下，user 消息中的指令只在当前轮次有效，容易被后续对话稀释。

温度和 top_p 的论证：多样性与质量的平衡

温度 = 0 时，模型变成"复读机"，总是选择最"安全"的答案。温度 > 1 时，模型变得"天马行空"，可能输出无意义的内容。top_p 的作用是在温度"打乱"概率分布后，再设置一道"质量门槛"——只从累积概率达到 p 的最小词集合中选，彻底排除尾部垃圾词。两者配合使用，才能同时保证多样性和可读性。

核心论证：这些参数不是"锦上添花"的装饰，而是控制大模型行为的核心杠杆。不懂温度，你就无法让模型在"严谨"和"创意"之间切换；不懂系统提示，你就无法给模型设定稳定的行为规范；不懂流式输出，你的应用体验就会落后于竞品。

要点核心要点

  • 流式输出：交互应用必开，提升用户体验；批处理可关
  • 系统提示：控制模型行为的最强手段，要具体、限定边界、指定格式
  • 温度：低 = 确定/严谨，高 = 随机/创意
  • top_p：控制候选词范围，和温度配合使用
  • 多轮对话：模型无记忆，历史需要你自己维护并传入

实验感受参数对输出的影响

这个实验通过改变参数观察模型输出的变化，帮助你建立对参数的直觉。

实验 1：温度的影响

  1. 用以下代码，固定 prompt，分别用 temperature = 0, 0.5, 1.0, 1.5 各调用 3 次：
import ollama

prompt = "给一家奶茶店起 3 个名字"
for temp in [0, 0.5, 1.0, 1.5]:
    print(f"\n=== temperature={temp} ===")
    for i in range(3):
        resp = ollama.generate(
            model="qwen2.5:1.5b",
            prompt=prompt,
            options={"temperature": temp}
        )
        print(f"  {i+1}. {resp['response'][:60]}...")
  1. 观察不同温度下的输出差异：低温时输出是否相似？高温时输出是否更发散？

预期现象：temperature=0 时，三次输出几乎完全相同；temperature=1.5 时，三次输出差异很大，甚至可能出现不通顺的句子。

实验 2：系统提示的力量

  1. 用 chat 接口，保持 user 消息不变，只改变 system 消息：
import ollama

user_msg = "解释什么是云计算"
for persona in [
    "你是一位严谨的大学教授",
    "你是一位给小学生讲课的老师",
    "你是一个爱用表情包的互联网博主"
]:
    print(f"\n=== {persona} ===")
    resp = ollama.chat(
        model="qwen2.5:1.5b",
        messages=[
            {"role": "system", "content": persona},
            {"role": "user", "content": user_msg}
        ]
    )
    print(resp["message"]["content"][:100] + "...")
  1. 对比三种人设的回答风格差异。

预期现象：教授风格用词正式、结构清晰；小学老师风格简单易懂、多用比喻；博主风格轻松随意、可能带网络用语。同一个模型，不同的 system 提示能产生截然不同的输出。

实验 3：多轮对话的记忆边界

  1. 用 ChatBot 类进行多轮对话，持续提问让历史消息增长。
  2. 当消息数量足够多时（超过模型的上下文窗口），问一个关于最早对话的问题，观察模型是否还记得。

预期现象：当历史消息超过上下文窗口限制时，最早的消息会被"遗忘"。这验证了模型的记忆不是无限的，也说明了为什么生产系统中需要实现消息裁剪或摘要机制。

错误常见错误

错误 1：多轮对话里忘记把 assistant 的回复加回历史

如果 messages 里只有 user 的消息，模型看不到自己之前说了什么，多轮对话就失效了。

错误 2：温度设成 0 以为能消除幻觉

温度 0 只是让输出更确定，但如果模型本来就"不知道"，它仍然可能一本正经地胡说。消除幻觉要靠知识库检索（RAG），不是靠调温度。

错误 3：系统提示写得太笼统

"你是一个有帮助的助手"等于没说。系统提示的质量直接决定输出质量，值得花时间打磨。

示例代码示例

import ollama

def generate_with_options(
    prompt,
    model="qwen2.5:7b",
    temperature=0.7,
    top_p=0.9,
    system=None,
    stream=False
):
    """通用生成函数，支持系统提示和参数调节"""
    messages = []
    if system:
        messages.append({"role": "system", "content": system})
    messages.append({"role": "user", "content": prompt})

    response = ollama.chat(
        model=model,
        messages=messages,
        options={"temperature": temperature, "top_p": top_p},
        stream=stream
    )

    if stream:
        return response  # 返回生成器，由调用方处理
    return response["message"]["content"]

# 严谨模式：回答技术问题
print(generate_with_options(
    "解释 TCP 三次握手",
    temperature=0.2,
    system="你是一位网络工程师，回答要准确、简洁"
))

# 创意模式：起名字
print(generate_with_options(
    "给猫起名",
    temperature=1.2,
    system="你是一位富有想象力的作家"
))
