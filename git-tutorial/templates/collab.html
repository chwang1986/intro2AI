07 多人协作

模拟两人协作：开发、推送、拉取、合并冲突

起源从 Linux 内核开发到 Git 的诞生

2005 年以前，Linux 内核开发团队一直使用一款商业版本控制软件 BitKeeper。这款软件是专有的，Linux 创始人 Linus Torvalds 和 BitKeeper 公司达成了一项特殊协议，允许内核开发者免费使用。

但是到了 2005 年 4 月，BitKeeper 公司决定停止向开源社区提供免费授权。这意味着 Linux 内核项目突然失去了版本控制工具。Linus 面临一个选择：要么回到旧的 CVS 或 SVN，要么自己写一个新的。

Linus 对当时的集中式版本控制系统很不满意。他认为这些系统要么太慢，要么不支持真正的并行开发。于是他花了大约两周时间，写出了 Git 的第一个版本。Git 的设计目标很明确：支持大规模分布式协作，速度快，数据完整性好。

两周后，Linux 内核代码就迁移到了 Git 上。这意味着 Git 从诞生第一天起，就必须解决「几百个开发者同时修改同一份代码」这个问题。

思考集中式 vs 分布式：该怎么协作？

在 Git 出现之前，大多数团队使用 CVS 或 SVN。这些系统是集中式的：只有一个中央服务器保存所有代码，每个人的电脑上只有当前工作的一个副本。你要提交代码，必须连接到服务器。

这种模式的缺点是显而易见的。如果服务器宕机，所有人都无法工作；如果两个开发者同时修改了同一个文件，SVN 会锁住文件，导致后面的人必须等待；最麻烦的是，如果没有网络，你连提交代码都做不到。

Linus 思考的问题是：为什么必须有一个「中心」？每个开发者的电脑都有完整的仓库历史，为什么不让他们在本地完成所有操作，只在需要同步时才交换代码？

基于这个思路，Git 采用了分布式模型。每个开发者的本地仓库都是完整的，可以独立提交、分支、合并。冲突不是在服务器上发生的，而是在每个人的本地解决。这彻底改变了协作的方式。

论证为什么分布式协作是更好的方案？

分布式协作之所以被现代开发团队普遍接受，不是因为「新」，而是因为它解决了集中式系统的根本问题。

论证一：离线工作能力。在分布式模式下，开发者可以在没有网络的情况下继续提交代码。你可以坐飞机时写代码，在咖啡馆断网时继续工作。每个提交都保存在本地，等有网络时再一次性推送。集中式系统（如 SVN）要求每次提交都必须连接服务器，这在移动办公场景下是致命的。

论证二：并行开发的自然支持。在 Git 中，创建分支和合并是日常操作，成本极低。每个功能都可以在独立分支上开发，互不干扰。SVN 也有分支，但实现方式是复制整个目录，操作成本高，团队往往避免使用。结果是大家都在 trunk 上直接开发，冲突频发。

论证三：数据安全与冗余。在集中式系统中，中央服务器是唯一的数据源，如果硬盘损坏且备份失败，历史记录就彻底丢失。在 Git 中，每个开发者的电脑上都有一份完整的历史。即使服务器挂了，从任意一台开发者电脑上都能恢复整个仓库。

论证四：工作流灵活性。Git 不强制任何特定的工作流。你可以用「集成管理者」模型（Linux 内核的方式），也可以用「功能分支」模型（大多数公司的方式），还可以用「Forking」模型（开源社区的方式）。这种灵活性让不同规模的团队都能找到适合自己的协作方式。

当然，分布式也有代价。你需要理解本地和远程的区别，需要学习 pull、push、fetch 等同步操作。但相比它带来的好处，这些学习成本是值得的。

场景两个人怎么一起工作

实际工作中，一个项目通常由多个人共同开发。每个人在本地开发自己的功能，然后把代码推送到远程仓库。Git 会自动合并不冲突的修改，如果两个人改了同一处代码，就需要手动解决冲突。

规范多人协作的日常约定

分支命名：团队应该约定统一的分支命名规则。常见的做法是：feature/登录功能、bugfix/修复空指针、hotfix/紧急修复。一眼就能看出分支的用途。不要在分支名里写中文拼音或缩写，比如 xg、xiugai，一个月后没人知道这是什么。

提交信息：提交信息不是写给自己看的，是给队友看的。好的提交信息应该说明「做了什么」和「为什么」。推荐格式：类型: 简短描述，比如 feat: 添加用户登录接口、fix: 修复订单金额计算错误、docs: 更新 API 文档。

先 pull 再 push：推代码前先拉取最新代码，这是最基本的礼貌。如果你直接 push，而远程已经有了新提交，Git 会拒绝。养成习惯：开始工作前 git pull，准备 push 前再 git pull 一次。

通过 Pull Request 合并：不要直接把代码 push 到 main 分支。应该推送到自己的 feature 分支，然后发起 Pull Request，让至少一个队友审查通过后再合并。这是防止 bug 进入主线的最后一道防线。

要点协作的核心流程

1. 各自开发：每个人从 main 切出自己的 feature 分支，在上面开发。不要直接在 main 上写代码，这是协作的基本纪律。开发完成后，把 feature 分支 push 到远程。

2. 获取更新：在开始工作前，先 git pull origin main 获取队友的最新代码。如果 main 有更新，建议先把 main 合并到自己的 feature 分支，解决冲突后再继续开发。这样可以避免最后合并时爆发大量冲突。

3. 解决冲突：如果两个人改了同一个文件的同一处代码，Git 会在文件中插入 <<<<<<< HEAD 标记。你需要打开文件，保留正确的代码，删掉标记，然后重新 add 和 commit。冲突不可怕，可怕的是不看清楚就随便选一个版本。

4. 代码审查：通过 Pull Request 让队友审查代码。审查时重点看逻辑是否正确、有没有明显的 bug、代码风格是否统一。审查通过后再点击合并按钮，把 feature 分支合并到 main。

错误协作中常见的坑

冲突解决不当：遇到冲突时，有人直接用 git checkout --ours 或 --theirs 强制选一边，结果把队友的功能删掉了。正确做法是打开冲突文件，理解两边的修改意图，手动合并出最好的方案。

代码被覆盖：A 改了文件 X 的第 10 行，B 也改了文件 X 的第 10 行，B 先 push，A pull 时产生冲突。A 解决冲突时没注意，把 B 的改动覆盖了。这种情况只能通过代码审查和测试来预防。

误推敏感信息：把密码、API 密钥、数据库连接字符串直接写进代码并 push 到远程。即使后续删除，敏感信息仍然留在历史记录中。正确做法是使用环境变量或配置文件（并把配置文件加入 .gitignore）。

大文件提交：把视频、图片数据集、编译产物等几百兆的文件提交到 Git。Git 不适合管理大文件，每次 clone 都要下载这些历史大文件，仓库会迅速膨胀到几个 G。应该使用 Git LFS 或云存储。

在 main 上直接开发：为了省事直接在 main 分支上写代码，结果把未完成的功能或带 bug 的代码推到了生产环境。main 分支应该始终保持可发布状态。

AI人与 AI 协作的额外注意

随着 AI 辅助编程的普及，团队中除了人类开发者，还多了 AI 这个「虚拟队友」。AI 可以快速生成代码、修复 bug、重构函数，但它也带来了一些人类协作中没有的新问题。

规则一：AI 不直接操作 main。AI 的所有修改都应该在 feature 分支上进行，经过人类审查后才能合并。不要给 AI 直接 push 到 main 的权限。

规则二：AI 的提交必须明确标注。commit message 中应该说明这是 AI 生成的代码，比如 feat(ai): 由 AI 生成用户登录接口。这样出了问题可以追溯到来源。

规则三：AI 修改的范围要受限。通过 .gitignore 和文件权限限制 AI 能修改的文件范围。配置文件、密钥文件、CI/CD 脚本等敏感文件不应该让 AI 触碰。

规则四：AI 代码必须经过测试。AI 生成的代码可能看起来正确，但实际运行时会有边界情况问题。合并前必须跑通自动化测试，核心功能还需要人工手动验证。

案例人-AI 协作的典型场景

案例一：AI 生成了新功能。你让 AI 写一个排序函数，AI 很快生成了代码。你不要直接复制到 main 分支，而是让 AI 把代码放到 feature/ai-sort 分支，然后你发起 Pull Request，检查逻辑是否正确、边界情况是否处理、代码风格是否统一，确认无误后合并。

案例二：AI 和人类改了同一个文件。你手动修改了 utils.py 的第 20 行，同时 AI 也在另一个分支修改了同一处。合并时产生冲突。你需要打开文件，比较人类代码和 AI 代码，保留更优的实现，或者把两者的优点结合起来。

案例三：AI 误改了配置文件。AI 在批量重构时，把 .env 文件中的数据库连接字符串改成了示例值，并把修改一起提交了。如果 push 到远程，敏感信息可能泄露（即使改回正确的，历史记录中仍有痕迹）。解决方案：用 .gitignore 忽略 .env，使用环境变量注入配置。

案例四：AI 生成了大量代码但测试不通过。AI 一次性生成了 500 行代码，看起来很完整，但跑测试时失败了 3 个用例。不要试图手动修复这 3 个用例然后把整段代码合并——应该先让 AI 定位失败原因，重新生成修复版本，再次测试通过后再合并。

实验亲手制造并解决一次冲突

冲突是协作中不可避免的，但真正理解冲突如何解决，最好的方式是亲手制造一次。下面我们用两个本地目录模拟两个开发者，让他们修改同一个文件的同一行，然后观察 Git 的行为。

按照下面的步骤操作，你会看到冲突标记 <<<<<<< 是怎么产生的，以及如何手动解决它。

# 步骤 1：创建模拟环境
mkdir collab-lab && cd collab-lab
git init --bare remote.git        # 创建「远程」裸仓库
git clone remote.git alice        # 开发者 Alice 克隆
git clone remote.git bob          # 开发者 Bob 克隆

# 步骤 2：Alice 创建初始文件并推送
cd alice
echo "print('hello')" > app.py
git add app.py && git commit -m "init"
git push origin main

# 步骤 3：Bob 拉取，然后修改同一行
cd ../bob
git pull origin main
echo "print('hello from bob')" > app.py
git add . && git commit -m "bob's change"
git push origin main

# 步骤 4：Alice 也修改同一行（先不 pull）
cd ../alice
echo "print('hello from alice')" > app.py
git add . && git commit -m "alice's change"

# 步骤 5：Alice 推送，会被拒绝，必须先 pull
git push origin main              # 失败！远程有 Bob 的新提交
git pull origin main              # 产生冲突！

# 步骤 6：查看冲突标记
cat app.py
# 你会看到类似下面的内容：
# <<<<<<< HEAD
# print('hello from alice')
# =======
# print('hello from bob')
# >>>>>>> 分支名

# 步骤 7：手动解决冲突，保留双方内容
echo "print('hello from alice and bob')" > app.py
git add app.py
git commit -m "resolve conflict: merge alice and bob changes"
git push origin main

完成上面的实验后，你应该理解了三个关键点：第一，Git 会在冲突处插入标记，而不是自动选择某一方；第二，解决冲突后必须重新 add 和 commit；第三，冲突的本质是「同一处代码有不同的意图」，解决冲突需要理解双方的修改目的，而不是随便选一个版本。

练习协作场景的设计与验证

验证题 1：在多人协作中，为什么「先 pull 再 push」是一个必须遵守的规则？如果不这样做会发生什么？

验证题 2：假设你和队友同时修改了同一个文件的不同行（比如你在第 10 行加代码，他在第 20 行加代码），Git 会怎么处理？会产生冲突吗？

设计题 1：你和另外两名同学要合作完成一个大作业项目。请设计一套分支命名规范、提交信息格式和代码审查流程，写成一份简短的团队约定（200 字以内）。

设计题 2：假设你的团队中有人习惯直接在 main 分支上开发，经常把未完成的功能 push 上去，导致其他同学 pull 下来后代码跑不通。请设计一个工作流或技术方案，从流程上防止这种情况发生。

设计题 3：在一次合并冲突中，你发现自己和队友的实现思路完全不同，你的方案效率更高但改动大，队友的方案改动小但效率一般。双方都有合理的理由。作为冲突的解决者，你应该怎么做？

测验协作知识自检

判断题 1：在 Git 中，如果两个人修改了同一个文件的不同行，Git 通常会自动合并，不需要手动解决冲突。

答案：正确。只有修改了同一行或相邻行时，Git 才无法自动判断意图，需要手动解决。

判断题 2：解决冲突时，使用 git checkout --ours 保留自己的版本是最安全的做法。

答案：错误。直接选择一方可能会丢弃另一方的合理修改，正确做法应该是理解双方意图后手动合并。

选择题 3：以下哪个操作最有助于减少最终合并时的冲突？

A. 每天开始工作前执行 git pull origin main

B. 等所有功能开发完再一次性 pull

C. 尽量避免和队友修改同一个文件

D. 直接在 main 上开发，不用分支

答案：A。频繁同步可以减少分支之间的差异，降低冲突概率。

选择题 4：把敏感信息（如密码）误提交到 Git 仓库后，以下哪种说法是正确的？

A. 立即删除文件并 commit，这样就不存在了

B. 删除文件后，历史记录中仍然会保留该信息

C. 只要文件不在当前工作目录，就是安全的

D. Git 会自动加密存储，所以不用担心

答案：B。Git 保存了完整历史，简单删除文件并不能从历史中移除敏感信息，需要使用 git filter-repo 等工具重写历史。

选择题 5：Pull Request 的主要作用是什么？

A. 把代码从本地推送到远程

B. 让队友审查代码，防止有问题的代码进入 main

C. 自动运行代码并部署到服务器

D. 替代 git merge 命令

答案：B。Pull Request 是代码审查的载体，是防止 bug 进入主线的最后一道防线。
