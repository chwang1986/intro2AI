06 远程仓库

git remote、git push、git pull、git clone

场景为什么需要远程仓库

本地仓库只能在你自己的电脑上访问。如果硬盘坏了、电脑丢了，或者你想在另一台电脑上继续工作，本地仓库就帮不上忙了。

远程仓库（如 GitHub、Gitee）就是放在云端的 Git 仓库。它既是备份，也是协作的桥梁。你可以把本地提交推送到远程，也可以从远程拉取别人的更新。

原理每个仓库都是完整的备份

Git 是分布式版本控制系统。这意味着每个开发者的本地仓库都包含完整的项目历史，不依赖中央服务器。远程仓库只是另一个本地仓库的镜像，大家通过 push 和 pull 来同步。

可以把远程仓库想象成一个公共图书馆。你把自己的书（commit）送过去（push），也可以把别人新上架的书借回来（pull）。如果图书馆暂时关门，你手里的书（本地仓库）依然是完整的，不影响你继续阅读和工作。

本地与远程同步

要点远程操作的核心命令

git remote add origin <地址> 关联远程仓库。origin 是远程仓库的默认别名，就像给地址起了一个短名字。

git push -u origin main 把本地 main 分支推送到远程。第一次推送时加 -u 参数，建立本地分支和远程分支的追踪关系，以后直接 git push 即可。

git pull 从远程拉取最新代码并自动合并。相当于 git fetch + git merge 的组合。

git clone <地址> 从远程下载完整的仓库到本地，自动设置好远程关联。这是获取别人项目的方式。

错误常见错误

push 被拒绝，提示「更新被拒绝，因为远程包含本地没有的工作」。这通常是因为有人在远程提交了新的内容。解决方案：先 git pull 拉取合并，再 push。

clone 时用了 HTTPS 地址，每次 push 都要输入密码。更好的方式是配置 SSH 密钥，或者使用个人访问令牌（PAT）。

在错误的分支上 push。比如你在 feature 分支上开发了新功能，但 push 到了远程的 main 分支，把未完成的功能发布到了生产环境。
