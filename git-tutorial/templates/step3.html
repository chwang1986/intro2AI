03 添加与提交

git add、git commit、提交信息规范

场景什么时候 add 和 commit

当你完成了一部分有意义的工作，比如写好了登录功能或修复了某个 bug，就应该 add 和 commit。这相当于给当前的工作拍了一张快照，以后随时可以回到这个状态。

不要等所有功能都做完才提交。一个好的习惯是每完成一个小功能就提交一次。这样如果你的新改动引入了 bug，可以精确回退到出问题之前的状态。

想象你在拼一幅巨大的拼图。每拼完一小块，就用手机拍张照片记录下来。如果后面拼错了，你可以根据照片找回之前正确的状态，而不需要从头开始。

起源Git 的对象模型是怎么来的

Linus Torvalds 在写 Git 的时候，借鉴了他在 Linux 内核开发中的经验。Linux 内核本身就是一个巨大的文件系统，Linus 对"如何高效地存储和索引文件"有着深刻的理解。

2005 年设计 Git 时，他明确提出：Git 不应该只是一个版本控制工具，它应该是一个"内容寻址的文件系统"。什么是内容寻址？简单来说，就是根据文件内容本身来命名和查找文件，而不是根据文件名或路径。只要两个文件内容一样，无论它们叫什么名字、放在哪里，在 Git 看来都是同一个对象。

这个想法直接催生了 Git 的三大核心对象：blob（存内容）、tree（存目录结构）、commit（存提交信息）。这三种对象层层嵌套，构成了 Git 的底层数据结构。当你执行 git add 和 git commit 时，本质上就是在创建这些对象，然后把它们链接起来。

思考为什么要把 add 和 commit 分成两步

很多同学第一次用 Git 时都会有这个疑问：为什么不能像保存文档一样，改完直接 commit，非要先 add 再 commit？这不是多此一举吗？

Linus 设计这两步分离，其实是经过深思熟虑的。想象你在写一篇期末论文：你可能同时修改了引言、方法论和结论三个部分。但等你准备保存的时候，你发现结论部分还有几个数据没核对完，暂时不想保存。这时候你怎么办？

git add 的作用就是让你"挑选"本次要保存的内容。你可以只 add 引言和方法论，把没写完的结论留到下次再提交。这种"暂存区"（也叫索引区）的设计，给了你一次"整理思绪"的机会，让你能把相关的改动打包成一次有意义的提交，而不是改一点就提交一点，最后历史记录乱成一团。

当然，如果你确定所有修改都要提交，也可以直接用 git commit -a 跳过显式 add 的步骤。但理解暂存区的存在，是掌握 Git 精细控制的关键。

原理Git 保存的是快照，不是差异

很多人以为 Git 像传统版本控制那样，只保存「这次改了什么」。实际上 Git 保存的是完整的快照——每次 git commit，Git 会把当前所有文件的内容打包成一个快照，用一个唯一的哈希值命名。

Git 内部用三种对象来组织数据：blob 存文件内容，tree 存目录结构，commit 存提交信息并指向一个 tree。这三个对象层层嵌套，构成了 Git 的「对象模型」。

对象模型图解

论证快照模型为什么比增量模型更好

传统的版本控制系统（比如早期的 SVN 和 CVS）使用的是"增量模型"：每次提交只保存与上一个版本的差异（diff）。想象一下，你有一本 100 页的书，每次修订只记录"第 5 页改了什么、第 20 页删了什么"。如果这本书经历了 1000 次修订，你想看第 500 个版本的内容，系统就得从第一个版本开始，依次应用 500 个差异补丁。

Git 使用的是"快照模型"：每次提交保存的是当时所有文件的完整内容。这听起来很浪费空间，但 Git 用了很多巧妙的技术来压缩（比如把内容相同的文件只存一份，用哈希值引用）。而且快照模型带来了一个巨大的好处：切换版本的速度极快。因为每个版本都是完整的快照，Git 不需要逐步应用补丁，直接取出对应的文件内容即可。

打个比方：增量模型就像一部只能快进的录像带，想跳到中间某一段，得从头慢慢放；快照模型就像一本相册，每一页都是完整的照片，翻到哪页就是哪页。对于需要频繁查看历史、回退版本的开发者来说，快照模型的体验好太多了。

另外，快照模型让分支和合并变得异常简单。因为每个分支本质上只是一个指向某个快照的指针，创建分支只需要创建一个新指针，不需要复制任何文件。这也是为什么 Git 的分支操作快到几乎无感知的原因。

要点add 和 commit 的区别

git add 把文件从工作区放到暂存区。你可以多次 add 不同的文件，也可以 add 同一个文件多次（如果中间又修改了）。

git commit 把暂存区的所有内容保存到本地仓库，生成一个永久的历史记录。commit 后暂存区清空，工作区保持现状。

提交信息（commit message）用 -m 参数指定。好的提交信息应该说明「做了什么」和「为什么做」，比如 "修复登录页面密码验证逻辑"。

实验观察 add 和 commit 的每一步变化

下面的实验会带你追踪一个文件从"未被追踪"到"被提交到仓库"的完整生命周期。请特别注意 git status 的输出变化，它会告诉你文件当前处于哪个区域。

# 前置：确保你在一个已初始化的 Git 仓库中
$ git status
# 应该显示 "On branch main" 且没有未跟踪文件（或有一些）

# 实验 1：创建一个新文件
$ echo "Hello Git" > hello.txt
$ git status
# 观察：hello.txt 出现在 "Untracked files" 区域

# 实验 2：将文件添加到暂存区
$ git add hello.txt
$ git status
# 观察：hello.txt 出现在 "Changes to be committed" 区域

# 实验 3：修改暂存后的文件（不重新 add）
$ echo "Hello World" >> hello.txt
$ git status
# 观察：同一个文件同时出现在两个区域！
# 一个是已暂存的版本，一个是未暂存的修改

# 实验 4：查看暂存区和工作区的差异
$ git diff
# 显示工作区与暂存区的差异
$ git diff --cached
# 显示暂存区与最新提交的差异

# 实验 5：提交暂存区的内容
$ git commit -m "添加 hello.txt"
$ git status
# 观察：暂存区清空，未暂存的修改还在工作区

# 实验 6：查看提交日志
$ git log --oneline
# 你应该能看到刚才的提交记录

这个实验最关键的一步是"实验 3"——当你 add 后又修改了文件，Git 会同时追踪两个版本：暂存区里的旧版本和工作区里的新版本。理解这一点，你就掌握了 Git 暂存机制的核心。

错误常见错误

修改了文件但忘记 add 就直接 commit。结果提交的是旧版本，新改动没有进入仓库。

提交信息写 "update"、"fix"、"改了东西"。一周后再看完全不知道这次提交改了什么。

一次性 add 所有文件（git add .）而不检查，把临时文件、日志文件也提交了。

用 git commit -a 跳过 add 步骤时，误以为所有修改都会被提交。实际上 -a 只自动 add 已跟踪的文件，新建的文件仍然需要手动 add。

提交了未完成或带有 bug 的代码，导致团队成员 pull 下来的代码无法运行。提交前应该在本地测试，确保代码至少能正常编译或运行。

验证题 1（动手操作）：创建一个文件并 add 它，然后修改文件但不重新 add，直接执行 git commit -m "test"。提交后查看文件内容，问：仓库里保存的是第一次 add 的版本，还是第二次修改后的版本？为什么？

设计题 1（开放思考）：假设你正在开发一个功能，同时修改了三个文件。但提交前你发现其中一个文件的改动引入了 bug，你只想提交另外两个文件。请设计一个操作流程，不需要撤销那个有 bug 的文件的修改，就能完成提交。

设计题 2（开放思考）：你和同学合作一个项目，你们约定每次提交信息都要遵循一定的格式。请你为你们的课程设计项目设计一套提交信息规范，要求至少能区分"新增功能"、"修复 bug"、"文档更新"三种类型。
