03 添加与提交

git add、git commit、提交信息规范

场景什么时候 add 和 commit

当你完成了一部分有意义的工作，比如「写好了登录功能」或「修复了某个 bug」，就应该 add 和 commit。这相当于给当前的工作拍了一张快照，以后随时可以回到这个状态。

不要等所有功能都做完才提交。一个好的习惯是：每完成一个小功能就提交一次。这样如果你的新改动引入了 bug，可以精确回退到出问题之前的状态。

要点add 和 commit 的区别

git add 把文件从工作区放到暂存区。你可以多次 add 不同的文件，也可以 add 同一个文件多次（如果中间又修改了）。

git commit 把暂存区的所有内容保存到本地仓库，生成一个永久的历史记录。commit 后暂存区清空，工作区保持现状。

提交信息（commit message）用 -m 参数指定。好的提交信息应该说明「做了什么」和「为什么做」，比如 "修复登录页面密码验证逻辑"。

错误常见错误

修改了文件但忘记 add 就直接 commit。结果提交的是旧版本，新改动没有进入仓库。

提交信息写 "update"、"fix"、"改了东西"。一周后再看完全不知道这次提交改了什么。

一次性 add 所有文件（git add .）而不检查，把临时文件、日志文件也提交了。
