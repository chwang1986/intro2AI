08 高级技巧

git stash、git rebase、git tag 的原理与应用

场景什么时候需要高级技巧

当你用 Git 一段时间之后，会遇到一些基础命令解决不了的问题。比如正在写代码时突然要修复紧急 bug，或者希望提交历史更整洁，又或者要给发布的版本打标记。

这些场景下，stash、rebase 和 tag 就派上用场了。它们像是 Git 工具箱里的专业工具，平时不一定天天用，但关键时刻能让你的工作效率大幅提升。

就像一位木匠，除了锤子和锯子，还需要卡尺和角尺来做精细活。基础命令是锤子和锯子，高级命令就是那些精细工具。

起源高级命令的诞生故事

git stash 的诞生源于一个日常困境：开发者正在写一个新功能，工作区里有大量未完成的改动，突然经理说「线上出了紧急 bug，需要你马上修复」。如果直接切换分支，Git 会阻止你，因为未提交的改动可能会和目标分支冲突。如果匆忙提交半成品，提交历史里就会多出一个「临时保存」的 commit，污染历史。

2007 年左右，Git 社区引入了 stash 命令，它可以把工作区和暂存区的改动打包成一个「草稿」，保存在一个独立的空间里。切换分支处理完紧急事务后，再把草稿恢复出来继续工作。

git rebase 的历史可以追溯到早期版本控制系统的「补丁队列」概念。在分布式协作中，如果每个人都用 merge 合并分支，提交历史会变成复杂的「意大利面条」。一些团队希望历史是干净、线性的，就像所有人在同一条时间线上依次提交一样。rebase 就是实现这个目标的工具——它把分支上的提交「摘下来」，在目标分支的最新位置重新「嫁接」上去。

git tag 的灵感来自传统的软件版本发布实践。软件开发到一定阶段会发布「1.0版」「2.0版」，开发者需要一个永不移动的标记来记住这些里程碑。branch 会随着新提交不断前进，而 tag 一旦被创建就固定指向某个 commit，永远不会变。

思考为什么要发明这些「额外」的工具

你已经学会了 add、commit、branch、merge、push、pull，这些命令足以完成日常开发。那为什么还需要 stash、rebase、tag 呢？因为真实世界的开发场景比教科书复杂得多。

关于 stash 的思考：工作区里的代码只有三种归宿——丢弃、提交、或保存。丢弃太可惜，提交会污染历史，所以需要一个「第四选项」：暂时存放但不进入正式历史。这就是 stash 的设计动机。

关于 rebase 的思考：merge 会把两条开发线「拧」在一起，形成一个带两个父节点的合并提交。如果团队频繁使用 feature 分支，历史图会变成一团乱麻。rebase 提供了一种替代方案：不「拧」在一起，而是把其中一条线「拉直」接在另一条线后面。代价是改写了历史——原来的提交被替换成了新的提交（hash 变了）。

关于 tag 的思考：commit hash（比如 a1b2c3d）对人类极不友好。开发者和用户交流时不会说「请使用 a1b2c3d 版本」，而是说「请使用 v1.2.0 版本」。tag 就是 hash 的人类可读别名，而且 tag 不会移动，适合标记发布节点。

原理stash、rebase、tag 的本质

git stash 的原理是：把工作区和暂存区的改动打包成几个 commit（默认存到 refs/stash），然后把工作区恢复到干净状态。这些 commit 不在正常的分支历史中，而是放在一个特殊的「储藏栈」里。你可以多次 stash，它们会按栈的结构堆叠，最后 pop 时按后进先出的顺序恢复。

git rebase 的原理是：找到当前分支和目标分支的共同祖先，计算当前分支上每个提交相对于祖先的「差异」（patch），然后把目标分支的最新提交作为新基底，依次重放这些差异。每个提交都会被重新创建，所以 commit hash 会改变。形象地说，rebase 就是「搬家」——把你的工作从旧地址搬到新地址。

git tag 的原理和分支几乎一样：它也是一个指向 commit 的引用。区别在于 branch 会随着新提交自动移动，而 tag 创建后就固定不动。tag 分为轻量标签（lightweight，只是一个引用）和附注标签（annotated，包含标签名、作者、日期、签名等元数据）。

论证这些工具为什么这样设计

stash 的论证：为什么 stash 不直接创建一个普通 commit？因为普通 commit 会进入当前分支的历史，而 stash 的意图是「临时存放」，不应该被未来的代码审查者看到。 stash 把草稿放在独立的空间里，既保留了改动，又不会污染任何分支的历史。这体现了「关注点分离」的设计思想。

rebase 的论证：rebase 的核心价值是历史可读性。一个线性的历史图（A→B→C→D）比一张分叉的网（A→B→C 和 A→D→E→merge）更容易理解。但 rebase 有一个重要前提：不要对已经 push 到公共仓库的提交执行 rebase。因为 rebase 改写了 commit hash，如果别人已经基于旧 hash 做了开发，你的 rebase 会让他们的工作基础「悬空」。所以 rebase 只适合清理本地尚未分享的历史。

tag 的论证：branch 和 tag 都是指向 commit 的引用，为什么需要两种机制？因为它们的语义完全不同：branch 表示「开发线的前端」，是动态的；tag 表示「某个特定的历史节点」，是静态的。用 tag 标记发布版本，几年后你仍然可以精确地回到那个版本。如果用 branch 标记，由于 branch 会前进，「v1.0 分支」几个月后可能指向的是 v1.0.17 的代码，失去了标记的精确性。

对比其他方案：Mercurial 也有类似的 queue/strip 机制对应 stash，有 rebase 扩展对应 git rebase，也有 tag 机制。这说明这些高级工具解决的是分布式版本控制的共性问题，不是 Git 独有的设计。

要点高级技巧的核心命令

git stash 把工作区和暂存区的改动保存起来，工作区恢复干净。git stash push -m "描述" 可以给 stash 加备注。git stash list 查看所有 stash。git stash pop 恢复最近一次 stash 并从列表中删除。git stash apply 恢复但不删除。

git rebase 目标分支 把当前分支的提交「嫁接」到目标分支的最新提交之后。git rebase -i HEAD~3 交互式 rebase，可以合并提交（squash）、修改提交信息、删除提交等。注意：已经 push 到远程的提交不要 rebase。

git tag 标签名 给当前 commit 打轻量标签。git tag -a v1.0 -m "版本 1.0 发布" 打附注标签。git push origin 标签名 把标签推送到远程。git push origin --tags 推送所有标签。git tag -d 标签名 删除本地标签。

实验探索 stash、rebase 和 tag 的内部机制

下面的实验将揭示这些高级命令在 Git 内部的真实行为。你会发现它们并没有那么神秘。

# ========== 实验 1：验证 stash 的存储位置 ==========
# 先修改一个文件但不提交
echo "临时改动" >> README.md
git stash push -m "我的临时改动"

# 观察 stash 引用
cat .git/refs/stash
# stash 本质上也是一个 commit 引用

# 查看 stash 列表
git stash list
# 恢复 stash
git stash pop

# ========== 实验 2：观察 rebase 如何改写历史 ==========
# 准备环境：创建 feature 分支并做两个提交
git branch feature-rebase && git switch feature-rebase
echo "feature 第一行" >> main.py && git add . && git commit -m "feat: 添加第一行"
echo "feature 第二行" >> main.py && git add . && git commit -m "feat: 添加第二行"

# 记录提交 hash
git log --oneline
# 记下这两个 feature 提交的 hash

# 切换回 main 并做一次新提交
git switch main
echo "main 新功能" >> main.py && git add . && git commit -m "main: 新功能"

# 执行 rebase
git switch feature-rebase
git rebase main

# 再次查看 hash
git log --oneline
# 你会发现 feature-rebase 上的两个提交 hash 变了！这就是 rebase 改写历史的证据

# ========== 实验 3：观察 tag 的本质 ==========
git switch main
git tag v1.0
git tag -a v2.0 -m "附注标签"

# 观察标签引用
ls .git/refs/tags/
cat .git/refs/tags/v1.0

# 附注标签不同，它是一个独立的 Git 对象
git cat-file -t refs/tags/v2.0
# 输出应该是 "tag" 类型，而不是 "commit"
git show v2.0
# 可以看到标签的作者、日期和附注信息

观察现象：stash 是一个特殊的引用（refs/stash），底层也是 commit 对象。rebase 会重新创建提交，hash 改变是最直接的证据。轻量标签直接指向 commit，附注标签则是一个独立的 tag 对象，再间接指向 commit。

错误高级命令的常见陷阱

stash 陷阱：stash 了很久之后忘记恢复， list 列表越来越长。建议给每个 stash 加描述（-m），定期清理不再需要的 stash。

rebase 陷阱一：在公共分支上 rebase 然后 force push，导致队友的代码基础「悬空」，他们 pull 时会产生大量冲突。黄金法则：只 rebase 本地未 push 的提交。

rebase 陷阱二：rebase 过程中遇到冲突，解决后忘记执行 git rebase --continue，导致 rebase 一直处于暂停状态。

tag 陷阱：打了标签但忘记 push 到远程，导致队友看不到标签。标签默认不会随 push 自动上传，需要显式推送。

混用陷阱：stash 里存放了未提交的改动，pop 出来后直接 rebase。如果 stash 的内容和 rebase 目标有冲突，会让情况变得更复杂。建议先 pop 确认工作区干净，再考虑 rebase。

练习高级技巧验证与设计

验证题 1：按照实验 2 的步骤，在 feature-rebase 分支上做两次提交，记录它们的 hash。然后回到 main 做一次新提交，再对 feature-rebase 执行 rebase。对比 rebase 前后 feature 提交的 hash，确认它们已经改变。思考：为什么 hash 会改变？

验证题 2：创建一个附注标签 git tag -a v0.1 -m "第一个测试版本"，然后用 git cat-file -p v0.1 查看它的内部结构。对比轻量标签和附注标签在 .git/refs/tags/ 下的区别。

验证题 3：在工作区修改两个文件，执行 git stash push -m "双文件改动"。用 git stash show -p 查看 stash 里保存的详细差异。然后执行 git stash pop 恢复，观察工作区是否完全恢复到 stash 前的状态。

设计题：假设你正在开发一个校园二手交易平台，代码已经部署到服务器上运行。现在有两个任务同时出现：任务 A 是添加「搜索过滤」功能（预计 3 天完成），任务 B 是修复一个导致订单丢失的紧急 bug（需要立即处理）。你已经在任务 A 上写了半天代码，工作区里有未提交的改动。请设计一个完整的工作流程：如何使用 stash 切换到任务 B？修复完成后如何恢复任务 A 的工作？任务 A 完成后，选择用 merge 还是 rebase 合并到 main？为什么？

设计题：你的项目准备发布 1.0 正式版。请设计一套标签管理策略：打什么类型的标签（轻量还是附注）？标签命名遵循什么规范？标签如何推送到远程？发布后如果发现严重 bug 需要发修复版，标签应该如何演进？
