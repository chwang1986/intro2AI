01 Git 是什么

版本控制的基本概念，为什么需要 Git

场景为什么需要版本控制

想象你在写一份重要的文档。你改了又改，保存了很多个版本：报告_v1.doc、报告_v2.doc、报告_最终版.doc、报告_最终版_真的.doc。最后你也不知道哪个才是真正的最终版。

当你和另一个人同时修改同一份文件时，问题更复杂。你改了 A 部分，他改了 B 部分，最后需要手动把两个人的修改合并到一起，一不小心就覆盖了对方的内容。

Git 就是来解决这些问题的。它自动记录每一次修改，你可以随时回到任意历史版本，也可以让多人同时工作并自动合并修改。

起源Linux 之父的两周杰作

2005 年，Linux 内核的开发团队遇到了一个大麻烦。他们一直免费使用的版本控制工具 BitKeeper，突然宣布不再对开源项目免费。这意味着全世界最大的开源项目——Linux 内核——失去了一直依赖的版本管理系统。

Linus Torvalds（Linux 的创始人）评估了当时市面上的替代方案：CVS 和 SVN。但他很快发现，这些集中式版本控制工具在处理像 Linux 这样庞大的项目时，速度太慢了。比如，SVN 在提交一个改动时，需要和远程服务器通信，而 Linux 内核每天有成百上千个改动。

Linus 决定自己动手写一个。他对新工具有三个硬性要求：第一，速度必须极快；第二，支持真正的分布式开发（每个开发者本地都有完整的仓库）；第三，数据绝对不能丢失。2005 年 4 月 3 日，Linus 发出了第一封关于 Git 的邮件。两周后，Git 已经可以自我管理自己的代码了。

思考为什么是"快照"而不是"增量"

当时主流的版本控制系统（CVS、SVN）采用的都是增量存储：它们只记录每次修改了哪些行。比如，你有一个 1000 行的文件，修改了第 50 行，系统就只保存"第 50 行从 A 改成了 B"。

这种设计在理论上很省空间，但有一个致命问题：回溯历史版本非常慢。如果你想回到 100 次提交之前的版本，系统必须依次应用 100 个增量补丁。对于 Linux 内核这种级别的项目，这可能需要几分钟。

Linus 的想法完全不同："我不在乎多占一点磁盘空间，我在乎的是速度。"他选择了快照存储：每次提交，Git 都会把整个项目的状态拍一张照片存起来。回到任意历史版本时，直接拿出那张照片即可——耗时几乎是零。

核心洞察：在 2005 年，磁盘空间已经便宜到可以忽略不计，但开发者的时间永远宝贵。Git 用空间换时间，这是一个典型的工程权衡。

原理Git 就像一台智能相机

想象 Git 是一台精密的相机系统。你在工作区编辑文件，就像在现实世界中布置场景。git add 相当于把想拍的物体放进取景框，你可以挑选哪些要拍、哪些不拍。git commit 则是按下快门，把取景框里的内容永久保存到相册——也就是本地仓库。每次快门都会生成一张带编号的照片，以后你可以随时翻出任何一张照片。

关键是：相册里保存的不是「变化」，而是完整的快照。这就是为什么 Git 能瞬间回到任意历史版本——它直接拿出那张照片即可，不需要逐条撤销。

三层模型图解

论证为什么三层模型是最佳设计

你可能会问：为什么 Git 不设计成"修改一次文件就自动保存"？为什么要多一个"暂存区"的环节？

这是一个非常好的问题。想象你在写一篇论文，同时修改了三个部分：标题、摘要、结论。这三个改动是相关的——它们共同构成了"完成摘要修订"这个逻辑单元。但如果 Git 每次保存文件都自动提交，你的历史记录里就会充斥着"改了标题"、"改了摘要"、"改了结论"这样的零散提交，而不是一个完整的"完成摘要修订"。

暂存区的存在，让你可以把多个文件的修改组合成一个有意义的提交。你可以先把标题和摘要的改动放进暂存区，确认它们构成了一个完整的逻辑单元后，再一次性提交。这就是"暂存区"的设计哲学：给开发者一个"整理包裹"的机会，而不是被迫每次改文件都发一个快递。

对比一下其他设计：如果 Git 只有工作区和仓库两层（没有暂存区），那么每次提交要么提交所有改动，要么只能提交单个文件。前者导致提交信息模糊，后者导致历史记录碎片化。暂存区是 Git 最精妙的设计之一。

要点三个核心概念

Git 的核心只需要理解三个东西：工作区、暂存区、仓库。它们之间的关系就像工厂的生产线。

概念图解
工作区
你实际编辑的文件夹，所有文件都在这里
→git add
暂存区
准备提交的快照，可以挑选要提交的文件
→git commit
本地仓库
历史版本记录，每次提交生成一个快照

错误初学者常犯的错误

以为 git add 就是「添加到仓库」。实际上 add 只是把文件放到暂存区，真正的保存是 git commit。

提交信息写得太随意，比如 改了点东西、更新。一个月后回头看，完全不知道这次提交做了什么。

直接在工作目录里复制文件做备份（备份_2024），而不是用 Git 管理版本。这样既占空间，又容易搞混。

实验亲手观察 Git 的三层模型

打开终端，跟着以下步骤操作，观察每个命令后 git status 的输出变化。这是理解 Git 三层模型最直接的方式。

实验步骤
# 第 1 步：创建一个测试目录并进入
mkdir git-lab && cd git-lab

# 第 2 步：初始化仓库，观察 .git 目录的诞生
git init
ls -la   # 你会看到一个 .git 目录

# 第 3 步：查看状态（此时应该是空的）
git status

# 第 4 步：创建一个文件，再次查看状态
# 注意：文件现在处于"工作区"（红色显示）
echo "hello world" > hello.txt
git status

# 第 5 步：把文件加入暂存区，观察状态变化
# 注意：文件现在处于"暂存区"（绿色显示）
git add hello.txt
git status

# 第 6 步：提交到仓库，观察状态变化
# 注意：工作区变干净了
git commit -m "添加 hello.txt"
git status

# 第 7 步：查看提交历史
git log --oneline

实验任务：在实验过程中，每次运行 git status 时，仔细观察输出中的文字提示。你能用自己的话解释"Untracked files"、"Changes to be committed" 和 "nothing to commit" 分别对应三层模型中的哪一层吗？

实例常用命令速查

核心命令
git init              # 初始化仓库
git status            # 查看当前状态
git add 文件名         # 将文件加入暂存区
git commit -m "说明"   # 提交到本地仓库
git log               # 查看提交历史
git diff              # 查看修改内容

练习巩固与思考

练习 1（验证）：按照"实验"部分的步骤，在本地创建一个 git-lab 目录并完成全部 7 步操作。截图或记录每次 git status 的输出，对比三层模型的对应关系。

练习 2（设计）：假设你要为班级课程项目设计一个版本控制流程。项目有 3 个同学参与，每人负责不同的章节。请设计一个"提交规范"，包括：什么时候提交、提交信息怎么写、什么时候把修改推送到共享仓库。写出至少 3 条规则。

练习 3（思考）：如果把 Git 的三层模型（工作区→暂存区→仓库）类比成生活中的某个流程，你会选择什么类比？（比如：购物车的"加入购物车→确认订单→支付"）请写出你的类比，并解释每个步骤的对应关系。
