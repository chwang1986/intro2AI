04 查看历史与回退

git log、diff、checkout —— 查看过去，回到过去

场景什么时候需要查看历史

代码出问题了，想知道是哪次提交引入的 bug。或者你想参考两周前某个功能的实现方式。又或者你不小心删了一段代码，想把它找回来。这些时候都需要查看历史。

Git 的每一次提交都是一个完整的快照，而不是增量补丁。这意味着你可以瞬间回到任意一个历史版本，不需要逐条撤销修改。

想象你有一台时光相机，每拍一张照片就永久保存那个时刻的全貌。当你想回顾某个时刻时，直接拿出那张照片即可，不需要一步步倒退时间。

起源Git 的历史链设计灵感来自哪里

Git 的核心数据结构——"有向无环图"（DAG）——并不是 Linus Torvalds 凭空发明的。在 BitKeeper 中，就已经有了类似的提交链概念。Linus 在设计 Git 时，吸收了 BitKeeper 中关于"每个提交都知道自己的父提交"这一思想，但做了重大改进。

在 BitKeeper 和早期的 CVS 中，历史虽然也是链式结构，但操作起来非常缓慢，因为每次回退或查看历史都需要与服务器交互。Linus 把整条历史链完整地搬到了本地，让每个开发者的电脑上都有一份完整的提交图。这样一来，git log、git diff、git checkout 这些操作都可以在本地瞬间完成。

另一个关键的设计来源是文件系统的"硬链接"概念。Git 中的分支本质上就是一个指向某次提交的"指针"，切换分支不需要复制任何文件，只需要移动指针。这种设计的巧妙之处在于：它让 Git 在支持复杂历史结构的同时，保持了极高的效率。

思考为什么回退操作需要多种方式

很多初学者会被 Git 的各种"回退"命令搞晕：git reset、git revert、git checkout……为什么一件事要有这么多不同的做法？

这其实是 Git 设计哲学的一个体现：不同场景下，"回退"的含义完全不同。有时候你想"撤销某次提交的效果，但保留历史记录"（比如公共仓库中你推错了代码）；有时候你想"彻底回到过去的某个状态，扔掉之后的所有改动"（比如本地实验搞砸了）；有时候你只是想"看看过去某个版本长什么样，但不打算长期停留"。

Linus 在设计这些命令时，刻意让它们保持独立，而不是用一个"万能回退"命令包揽所有场景。这样做的好处是：你必须明确知道自己的意图，才能选择合适的命令。虽然学习曲线更陡一些，但它极大地降低了误操作的风险。想想看，如果有一个"回退"按钮，你点下去却不知道它会删除历史还是保留历史，那该多危险。

原理历史是一条带箭头的链

Git 的历史不是一条简单的直线，而是一个有向无环图（DAG）。每个 commit 都指向上一个 commit，形成一条链。分支只是在某个 commit 上贴的标签，HEAD 是一个特殊指针，指向你当前所在的位置。

当你创建新分支时，Git 并没有复制任何文件，只是新建了一个指针。当你切换分支时，Git 把 HEAD 移过去，然后把工作区的文件换成那个 commit 对应的内容。这就是为什么 Git 的分支操作快到不可思议——它只是在移动指针。

提交链与 HEAD 指针

论证reset、revert 和 checkout 的本质区别

要理解这三个命令的区别，关键在于理解它们对"历史记录"和"工作区"的影响不同。

git reset --hard 是最激进的。它把当前分支的指针直接移到你指定的提交上，同时把工作区和暂存区都重置成那个提交的样子。之后的提交历史会从分支上"脱落"，虽然它们还暂时保存在 reflog 里，但如果不及时处理，就会被 Git 的垃圾回收机制清理掉。这就像你把一本书撕掉了后面几章——那些内容真的消失了。

git revert 是最安全的。它不会删除任何历史记录，而是创建一个新的提交，这个提交的内容恰好抵消你想要撤销的那次提交的修改。打个比方：reset 是擦掉黑板上的字，revert 是在黑板上再写一行字，说明"上一行字作废"。在团队协作中，revert 是首选，因为你已经推送到公共仓库的历史不应该被删除，否则其他协作者会混乱。

git checkout 是最温和的。它只是移动 HEAD 指针，让你可以查看某个历史版本的内容。默认情况下它不会修改工作区中已经提交的内容，但如果工作区有未提交的改动，checkout 可能会覆盖它们，所以需要小心。

总结一下：reset 改变历史（危险），revert 追加历史（安全），checkout 查看历史（温和）。理解了这三者的区别，你就不会再在回退时手忙脚乱了。

要点历史相关命令

git log 查看提交历史。加上 --oneline 参数可以显示简洁的单行格式，加上 -n 5 可以只显示最近 5 条。

git diff 查看修改内容。不带参数时对比工作区和暂存区，git diff HEAD 对比工作区和最新提交。

git checkout <commit> 切换到某个历史版本。注意这会进入「分离 HEAD」状态，适合查看旧代码，不适合继续开发。

git revert <commit> 撤销某次提交的修改，但保留提交历史（推荐）。git reset --hard 直接回退到某个版本，会丢失之后的提交（慎用）。

实验体验 diff、checkout 和 reset 的不同效果

这个实验需要你预先准备一个有多次提交的仓库。如果你手头没有，可以先按照下面的前置步骤快速建立一个实验环境。

# ===== 前置：建立实验环境 =====
$ mkdir diff-lab && cd diff-lab && git init
$ echo "v1" > app.txt && git add . && git commit -m "版本1：基础功能"
$ echo "v2" > app.txt && git add . && git commit -m "版本2：新增功能"
$ echo "v3" > app.txt && git add . && git commit -m "版本3：优化体验"
$ echo "v4" > app.txt && git add . && git commit -m "版本4：修复问题"

# 实验 1：查看历史，记录各版本的提交哈希
$ git log --oneline
# 记录下四个版本的哈希值（前 7 位即可），后面会用到

# 实验 2：用 diff 查看相邻版本的差异
$ git diff HEAD~1 HEAD
# 观察输出：它显示的是倒数第二个版本和最后一个版本的差异

# 实验 3：查看工作区与最新提交的差异
$ echo "未提交的修改" >> app.txt
$ git diff
# 观察输出：应该显示你刚添加的 "未提交的修改"

# 实验 4：checkout 到历史版本（分离 HEAD 状态）
$ git checkout HEAD~2
# 观察终端提示："You are in 'detached HEAD' state"
$ cat app.txt
# 问：文件内容显示的是哪个版本？

# 实验 5：回到主分支
$ git checkout main
# 或 git checkout master（取决于你的默认分支名）

# 实验 6：用 reset --hard 回退一个版本（危险操作！）
$ git reset --hard HEAD~1
$ git log --oneline
# 观察：最后一个提交"版本4"从历史中消失了
$ cat app.txt
# 问：文件内容变回了哪个版本？

# 实验 7：用 reflog 找回被 reset 掉的提交
$ git reflog
# 找到 "版本4" 对应的哈希值，然后执行：
# git reset --hard <版本4的哈希值>
# 观察："版本4" 被找回来了

这个实验最值得注意的是"实验 4"和"实验 6"的区别：checkout 让你临时查看历史，不影响分支走向；reset 则直接改写了分支历史。实验 7 则展示了 reflog 这个"时光机"——即使你用 reset 删掉了提交，只要没超过一定时间，都能找回来。

错误常见错误

用 git reset --hard 回退后发现删错了，但已经关掉了终端，找不到之前的 commit ID。实际上可以用 git reflog 找回。

在「分离 HEAD」状态下修改了代码并提交，然后切换回 main 分支，结果那些提交「消失」了。分离 HEAD 的提交不会被任何分支引用，切换后就找不到了。

git checkout 时没保存工作区的修改，导致未提交的改动丢失。checkout 前应该先 add/commit 或 stash。
