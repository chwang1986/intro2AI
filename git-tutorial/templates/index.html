如何使用 Git

从零开始掌握版本控制，让代码管理不再混乱

愿景为什么大学生需要学 Git

想象一下，你花了一晚上写大作业代码，眼看要交了，手一滑把关键文件删了，回收站也清空了——这种绝望，Git 可以完全避免。Git 就像一个无限撤销+自动存档神器，不仅能帮你找回任意时刻的代码，还能让你和同学同时改同一个文件而不互相覆盖。

学了 Git，你能做什么？

  • 课程项目：大作业、实验报告、课程设计都能版本化管理，再也不怕"昨天还能跑，今天怎么崩了"。
  • 毕业设计：从开题到答辩，每一次修改都有记录，导师随时能看到你的进展。
  • 实习工作：几乎所有公司的开发团队都用 Git，提前学会等于提前拿到入场券。
  • 开源贡献：想给喜欢的项目提 PR？Git 是你和全世界开发者协作的共同语言。

不学 Git 会怎样？代码说没就没，和同学组队只能用微信传文件、手动合并，看到 GitHub 上感兴趣的项目只能干瞪眼，软件工程课的小组作业更是灾难现场。

和大学课程的关系：不管是 C 语言、数据结构、面向对象，还是软件工程、数据库、Web 开发，只要涉及写代码，Git 都能派上用场。它不是某个语言的工具，而是你大学四年代码生涯的底层基础设施。

纲领这个教程要达成什么

最终目标：读完这个教程后，你能够独立使用 Git 管理自己的项目——从初始化仓库到推送到远程，从日常提交到分支合并，整个过程心中有数。

面向读者：完全零基础，或者只会用 GitHub 网页上传文件的人。不需要命令行基础，每一步都有解释。

学习路径：理解概念 → 掌握基本工作流 → 学会分支管理 → 掌握远程协作。

发展脉络版本控制简史：Git 是怎么来的

Git 不是凭空出现的。在 Git 诞生之前，程序员们已经折腾了 30 年的版本控制工具。了解这段历史，你会明白为什么 Git 长这样，以及它为什么能解决以前工具搞不定的问题。

  • 1975 - RCS：最早的版本控制工具，只能本地用，一次只能一个人改。就像图书馆"预约借书"——你借走了别人就得等着。
  • 1990 - CVS：第一个支持多人协作的工具，但分支难用、合并易错。多人协作时经常"代码冲突炸锅"。
  • 2000 - SVN：修正了 CVS 的一些问题，支持原子提交。但它是集中式的——没网就没法提交，服务器挂了全员停工。
  • 2005 - Git 诞生：Linux 内核社区和 BitKeeper 闹掰，Linus Torvalds 两周写出了 Git。核心设计是分布式：每人电脑都是完整仓库，离线也能提交。
  • 2008 - GitHub 上线：Git 从黑客工具变成全球开发者的社交平台。Pull Request、Fork 让开源协作变得前所未有的简单。
  • 现在：Git 是事实标准。VS Code 内置 Git、CI/CD 自动部署——Git 不只是版本控制，更是现代软件开发流程的基石。

怎么学给大学新生的学习建议

建议用时：每天 1 小时，约 2 周完成。不要一次性看完——Git 是工具，需要边学边用才能内化。

学习方法：

  • 第一步：通读"场景"和"起源"（10 分钟）——理解这一步要解决什么问题，为什么存在这个工具。
  • 第二步：动手做"实验"（20 分钟）——打开终端，跟着实验步骤一步步操作。不要只看，一定要敲命令。
  • 第三步：理解"原理"和"论证"（15 分钟）——思考"为什么这个设计是对的"，对比其他方案的优劣。
  • 第四步：完成"练习"和"测验"（15 分钟）——设计题没有标准答案，重要的是思考过程；测验题检验你是否真的理解了。

检验标准：如果你能向一个完全没用过 Git 的同学解释清楚"工作区、暂存区、仓库"的区别，并帮他初始化一个仓库完成第一次提交，说明这一步你掌握了。
