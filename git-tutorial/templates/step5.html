05 分支管理

git branch、git merge、冲突解决

场景为什么要用分支

想象你正在开发一个稳定运行的项目。突然想尝试一个新功能，但不确定能不能成功。如果你直接在主代码上改，万一改坏了，整个项目就崩了。

分支就是解决这个问题的方法。你从当前代码复制一条独立的时间线，在上面随意尝试。成功了就合并回主线，失败了就直接删掉，不影响原来的代码。

就像写小说时，作家会保存一份原稿，然后在副本上尝试不同的剧情走向。如果新剧情更好，就用副本替换原稿；如果不如原稿，直接扔掉副本即可。

起源分支的概念从哪来

2005 年，Linus Torvalds 在创建 Git 时，面临一个严峻的问题：Linux 内核有上千名开发者，有人正在修复紧急漏洞，有人在开发新功能，还有人在做实验性重构。这些工作必须同时进行，但不能互相干扰。

在 Git 之前，Linux 内核团队用过一款叫 BitKeeper 的商业版本控制工具。BitKeeper 支持轻量级的分支，但它不是开源软件，使用许可随时可能被收回。Linus 设计 Git 时，决心要做出一款开源的、分支操作极快的工具。

当时的 CVS 和 SVN 也有分支功能，但它们是「目录拷贝」式的分支——创建分支相当于把整个仓库复制一份到另一个目录，耗时几分钟甚至更久。开发者因为怕麻烦，很少使用分支。

Linus 的突破在于：他发现分支根本不需要复制任何代码。Git 的分支只是一个指向某个 commit 的轻量级指针（一个 41 字节的文件）。创建分支只需要写这个文件，切换分支只需要移动这个指针。这个设计彻底改变了开发者的工作方式。

思考当时面临的问题与选择

当时最核心的矛盾是：协作需要并行，但并行容易互相干扰。如果所有人都在同一条代码线上工作，一个开发者的实验性代码可能让整条线崩溃，所有人都受影响。

Linus 思考了三种可能的解决方案：

方案一：目录拷贝（SVN 的做法）。每次开新分支都复制整个项目目录。缺点是创建和切换都很慢，而且占用大量磁盘空间。开发者会因为成本太高而避免使用分支。

方案二：打标签标记版本。用标签记录稳定版本，开发者自己在本地管理不同版本。缺点是没有系统化的合并机制，两个人基于同一版本开发的代码很难整合到一起。

方案三：指针式分支（Git 的做法）。分支只是一个引用，指向某个 commit。创建工作瞬间完成，切换工作也只是更新工作区的文件。因为成本极低，开发者可以频繁创建和销毁分支。

Linus 选择方案三的原因是：Git 的底层存储结构是有向无环图（DAG），每个 commit 已经记录了其父 commit。既然历史本身就是一张网，那分支只需要在网的不同节点上放几个「书签」就可以了。这个设计与 Git 的底层数据结构天然契合。

原理分支只是可移动的标签

这是 Git 最优雅的设计之一：分支不是副本，只是一个指向某个 commit 的轻量级指针。创建分支时，Git 只花不到一毫秒写了一个 41 字节的文件。切换分支时，Git 把 HEAD 移过去，然后快速调整工作区的文件。

合并（merge）有三种情况：快进合并——目标分支没动，直接把指针移过去；三方合并——两个分支都有新提交，Git 找到共同祖先，把两边的改动合在一起；冲突——两边改了同一处，Git 标记出来让你手动决定。

分支与合并类型

论证为什么指针式分支是对的

你可能会问：分支只是一个指针，那不同的代码版本存在哪里？答案是：所有的代码版本都保存在 Git 的对象数据库中，分支只是帮你标记「当前应该看哪一个版本」。

这个方案的正确性可以从三个角度论证：

1. 数据结构层面：Git 的历史是一棵有向无环图，每个 commit 节点都包含指向父节点的指针。分支引用只是在这张图上增加了一个「入口点」。无论有多少个分支，底层数据都不会重复存储。这和 SVN 的目录拷贝有本质区别——SVN 的分支是 O(n) 操作，Git 是 O(1) 操作。

2. 合并算法层面：当两个分支需要合并时，Git 会自动找到它们的「最近共同祖先」（merge base）。这个祖先节点天然存在于 DAG 结构中。三方合并算法只需要比较「祖先版本」「分支 A 版本」「分支 B 版本」三个快照，就能自动合并不冲突的部分。这个算法的正确性建立在 DAG 结构之上，而指针式分支完美利用了这一点。

3. 工程实践层面：因为分支足够廉价，开发者养成了「一个任务一个分支」的习惯。功能开发、bug 修复、代码审查都在独立分支上进行，大大降低了代码互相干扰的概率。Linux 内核至今每天创建和合并数百个分支，这在 SVN 时代是不可想象的。

对比其他方案：Mercurial（hg）也采用了类似 Git 的指针式分支，这从侧面证明了这个设计的优越性。而 SVN 和 CVS 的目录拷贝式分支已经被现代版本控制系统淘汰。

要点分支操作的核心命令

git branch 查看所有分支，当前分支前面带 *。git branch 新分支名 创建新分支。git branch -d 分支名 删除已合并的分支。

git checkout 分支名 或 git switch 分支名 切换到指定分支。Git 2.23 之后推荐用 switch，语义更清晰。

git merge 分支名 把指定分支的修改合并到当前分支。如果两个分支修改了同一处代码，就会产生冲突，需要手动解决。

冲突解决：Git 会在冲突文件中插入 <<<<<<< 标记，显示两个版本的内容。你编辑文件保留想要的部分，然后 add 和 commit 即可。

实验验证分支的本质

下面的实验将带你亲眼看到「分支只是一个指针」这个事实。你会观察创建分支前后 Git 内部文件的变化。

# 实验 1：观察分支引用文件
# 先查看 .git/refs/heads 目录，这里存放着所有分支指针
cat .git/refs/heads/main

# 实验 2：创建新分支并观察变化
# 创建分支后，你会发现新增了一个 41 字节的文件
git branch test-branch
ls .git/refs/heads/
cat .git/refs/heads/test-branch
# 注意：test-branch 文件的内容和 main 完全一样，因为它们指向同一个 commit

# 实验 3：在新分支上提交后观察变化
git switch test-branch
echo "测试内容" >> README.md
git add . && git commit -m "test: 在 test-branch 上提交"
cat .git/refs/heads/test-branch
cat .git/refs/heads/main
# 你会发现 test-branch 的内容变了，而 main 保持不变

# 实验 4：快进合并的观察
git switch main
git merge test-branch
# 观察 git log，你会发现没有产生新的 merge commit，main 的指针直接移到了 test-branch 的位置

观察现象：创建分支时 Git 没有复制任何代码文件，只新建了一个引用文件。切换分支时工作区的文件内容变了，但 .git/objects 目录里的对象没有任何变化。这验证了分支的轻量性。

错误常见错误

在错误的分支上开发了很久，才发现自己不在 main 上。养成习惯：每次开始工作前先 git status 确认当前分支。

遇到冲突时直接用别人的版本或自己的版本，没有仔细对比。这可能导致功能丢失或 bug 引入。冲突时一定要理解两边的修改意图。

合并后忘记删除已经没用的分支，导致分支列表越来越长。定期清理已合并的分支是个好习惯。

练习动手验证与设计思考

验证题 1：按照上面的实验步骤，在终端中执行命令，确认 .git/refs/heads/ 目录下分支引用文件的内容变化。用 git log --oneline --graph --all 观察分支图，截图或记录你看到的结构。

验证题 2：创建两个分支 feature-A 和 feature-B，都从 main 切出。在 feature-A 上修改 main.py 的第一行，在 feature-B 上修改 main.py 的第二行。依次合并 feature-A 和 feature-B 到 main，观察是否产生冲突。思考：为什么修改不同行不会冲突？

验证题 3：在 feature-A 和 feature-B 上都修改 main.py 的同一行，然后尝试合并。观察冲突标记 <<<<<<< 的格式，手动解决冲突并提交。理解冲突文件中的 HEAD、分支名分别代表什么。
