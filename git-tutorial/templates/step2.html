02 初始化仓库

git init、工作区与暂存区

场景什么时候初始化仓库

每当你开始一个新项目，或者想让一个已有项目被 Git 管理时，第一步就是初始化仓库。这会在项目根目录创建一个隐藏的 .git 文件夹，Git 的所有记录都保存在这里。

想象你准备写一本日记。在开始记录之前，你需要先准备一个笔记本，确定页码格式和目录结构。git init 就是为你准备这个笔记本的过程。

如果你在没有初始化的情况下直接使用 git add 或 git commit，Git 会提示你当前目录不是一个仓库。这是 Git 在提醒你：先准备好档案室，再开始存档。

起源Git 为什么要被发明出来

2005 年，Linux 内核的开发遇到了一个大麻烦。当时全球有上千名开发者在为 Linux 写代码，他们需要一种工具来管理这些代码的变更。之前他们用的是一款叫 BitKeeper 的商业软件，但那一年 BitKeeper 的版权方收回了免费使用权。

Linus Torvalds——也就是 Linux 的创始人——面临两个选择：要么花钱买 BitKeeper 的授权，要么自己写一个。Linus 觉得现有的版本控制工具要么太慢，要么不够好使，于是他决定自己动手。短短两周内，他写出了 Git 的第一个版本。

Linus 设计 Git 时定了几个核心目标：第一，速度必须极快；第二，必须支持分布式协作，每个开发者都能在本地独立完成工作；第三，要能够可靠地管理超大型项目（比如 Linux 内核这种有数百万行代码的项目）。git init 就是这一切的起点——它在你本地搭建起一个完整的仓库。

思考为什么从"初始化仓库"开始

想象你要开始写日记。第一步是什么？不是直接拿起笔写，而是先准备一个本子。这个本子要有页码，要有目录，要有固定的格式，这样你以后翻阅才不会乱。

Git 也是同样的道理。在开始记录代码变更之前，Git 需要先在你的项目文件夹里搭建一个"档案室"——也就是 .git 目录。这个档案室有自己的组织架构：哪里放文件快照，哪里放分支记录，哪里放配置信息。没有这个档案室，Git 就无从知道该把数据存在哪里。

Linus 在设计时其实也可以让 Git 自动追踪所有文件，不需要显式初始化。但他选择了让用户主动执行 git init，因为这样可以明确区分"这是一个被 Git 管理的项目"和"这只是一个普通文件夹"。这种显式的边界感，避免了很多意外的误操作。

原理初始化就是建造档案室

执行 git init 时，Git 在你的项目文件夹里建造了一个精密的档案室——也就是 .git 目录。这个档案室有多个分区：对象库存放所有文件快照，引用区记录分支和标签，配置区保存仓库设置，钩子区存放自动化脚本。

初始化完成后，档案室是空的。它还没有收到任何文件，只是搭好了架子，等着你用 git add 和 git commit 往里面存东西。

.git 目录结构

Git 的底层指针机制

HEAD、分支引用与 Commit 的 DAG 关系

Git 的分支本质上是一个指向某个 commit 的轻量指针（文本文件），存储在 .git/refs/heads/ 目录下。例如 .git/refs/heads/main 的内容就是一个 40 位的 SHA-1 哈希值。

HEAD 是一个特殊的指针，它通常指向当前分支（如 ref: refs/heads/main），也可以直接指向某个 commit（detached HEAD 状态）。

所有 commit 对象通过 parent 字段链接，形成一棵有向无环图（DAG）：

A --- B --- C  <- main
       \
        D --- E  <- feature

每个 commit 节点包含：tree（快照）、parent（0~n 个）、作者信息、提交信息。

Merge 的三种策略

Fast-Forward

当目标分支（如 main）的历史是当前分支（如 feature）的直系祖先时，Git 只需将 main 的指针向前移动到 feature 的位置，无需创建新的 merge commit。

Before:  A --- B  <- main
                 \
                  C --- D  <- feature

After:   A --- B --- C --- D  <- main, feature

Three-Way Merge

当两个分支有分叉历史时，Git 找到它们的最近公共祖先（LCA, Merge Base），然后比较三个快照：LCA、当前分支末端、目标分支末端，自动合并差异。

Before:  A --- B --- C  <- main
            \
             D --- E  <- feature

After:   A --- B --- C --- M  <- main
            \            /
             D -------- E  <- feature

如果两个分支对同一文件的同一区域做了不同修改，Git 无法自动判断，就会产生合并冲突。

Octopus Merge

Git 支持一次合并多个分支（git merge branch1 branch2 branch3）。这需要一个以上的 parent，生成的 merge commit 会有多个 parent 指针。仅在没有冲突时才能使用。

Rebase vs Merge 的 DAG 对比

核心区别：Merge 保留分叉历史，Rebase 重写历史为线性。

Merge 结果：保留分叉历史

*   3a8f2e1 (main) Merge branch 'feature'
|\
| * 7b4c9d2 (feature) 新增功能 B
| * 2e5f8a1 新增功能 A
|/
*   9c1d4e5 基础版本

Rebase 结果：线性历史

* 7b4c9d2 (main, feature) 新增功能 B
* 2e5f8a1 新增功能 A
* 9c1d4e5 基础版本

Rebase 的原理是将 feature 分支上的每个 commit 逐个"移植"到 main 的最新 commit 之上，生成新的 commit SHA（因为 parent 变了）。

Merge 冲突的底层文件状态

冲突时 Git 如何标记文件

当 Three-Way Merge 遇到冲突时，Git 会在工作区文件中插入冲突标记，并创建三个暂存区版本：

  • stage 1：公共祖先（Base）版本
  • stage 2：当前分支（OURS）版本
  • stage 3：目标分支（THEIRS）版本
<<<<<<< HEAD
当前分支的代码
=======
目标分支的代码
>>>>>>> feature

使用 git ls-files -u 可以查看暂存区中这三个版本的 blob 对象哈希。

论证为什么 Git 选择在本地建"档案室"

在 Git 出现之前，最流行的版本控制工具是 CVS 和 SVN，它们都是"集中式"的。这意味着所有的版本历史都保存在一台中央服务器上，你的本地只保存当前版本的文件。每次查看历史、每次提交变更，都必须联网连接到服务器。

Linus 认为这种设计对于 Linux 内核开发来说是不可接受的。试想一下：如果你坐十几个小时的飞机去参加会议，在飞机上无法联网，你就不能提交代码、不能查看历史、不能回退版本——这对于一个全球协作的项目来说太脆弱了。

所以 Git 选择了"分布式"设计。执行 git init 后，你的本地就有了一个完整的仓库，包含了版本控制所需的全部基础设施。你可以离线提交、查看历史、创建分支，等联网后再把变更同步给其他人。这种设计的代价是每个开发者本地都有一份完整的数据副本，但在今天硬盘容量以 TB 计的时代，这点代价完全可以接受。

对比一下：SVN 的".svn"目录只保存了一些元数据，真正的历史在服务器上；而 Git 的".git"目录就是一个完整的数据库。这正是两者最根本的区别。

要点初始化后 Git 在做什么

执行 git init 后，Git 会在当前目录创建 .git 子目录。这个目录包含了所有版本历史、分支信息、配置等。你不应该手动修改里面的内容。

初始化后，所有文件都处于「未跟踪」状态。Git 知道你目录里有哪些文件，但还没有开始记录它们的变化。你需要用 git add 告诉 Git 哪些文件要纳入版本控制。

实验亲手揭开 .git 目录的面纱

下面这组操作会带你一步步观察 git init 到底做了什么。建议你在本地跟着做一遍，亲眼看看 .git 目录是怎么诞生的。

# 实验 1：创建一个新目录并进入
$ mkdir my-first-repo
$ cd my-first-repo

# 实验 2：确认此时没有 .git 目录
$ ls -la
# 观察输出，确认没有 .git

# 实验 3：执行初始化
$ git init
# 注意终端输出的 "Initialized empty Git repository"

# 实验 4：再次查看目录内容
$ ls -la
# 现在你应该能看到 .git 目录了

# 实验 5：查看 .git 内部的结构
$ ls -la .git/
# 你会看到 objects/、refs/、hooks/、config、HEAD 等文件和目录

# 实验 6：查看当前分支指针
$ cat .git/HEAD
# 输出应该是 "ref: refs/heads/main" 或 "ref: refs/heads/master"

# 实验 7：配置用户信息后查看配置文件
$ git config user.name "你的名字"
$ git config user.email "你的邮箱"
$ cat .git/config
# 观察 [user] 段落是否被写入了 config 文件

做完这组实验，你应该对"初始化仓库"有了更具体的理解。git init 不是做了什么神秘的操作，它只是在你的项目文件夹里建立了一套规范的文件结构。这套结构就是 Git 工作的基础。

错误常见错误

在错误的目录执行 git init，比如在「下载」或「桌面」这种包含很多无关文件的目录。结果 Git 把所有文件都标记为未跟踪，git status 输出一大串无关内容。

忘记配置用户名和邮箱就提交。Git 要求每次提交都记录作者信息，如果没有配置，提交会失败或显示匿名作者。

不小心删除或修改了 .git 目录里的内容。这个目录是 Git 的档案室，手动改动可能导致仓库损坏。如果 .git 被删除，整个版本历史就消失了。

在一个已有的 Git 仓库内部再执行 git init。这会在子目录里创建嵌套的仓库，导致外层仓库无法跟踪内层目录的文件，容易出现混乱。
