综合实战

用本教程的所有规范，搭建一个可复用的页面模板

实战

场景

学完前面的 10 个步骤，你已经掌握了设计令牌、基础组件、容器组件、数据展示、响应式布局、动画交互等知识。现在，你需要将这些知识整合起来，完成一个完整的项目。本章提供一个"项目管理中心"的实战项目，涵盖从需求分析到验收测试的完整流程。

起源

"项目管理中心"是后台管理系统（B端产品）中最常见的页面类型之一。它的设计范式可以追溯到 20 世纪 80 年代的 SAP GUI 和 Oracle Forms——当时的企业软件就已经使用"顶部统计 + 中部列表 + 底部操作"的三段式布局。

现代 SaaS 产品（如 Jira、Trello、Notion、飞书项目）继承并发展了这一范式。它们共同的设计语言是：信息密度高、操作路径短、反馈即时。这些特征决定了项目管理中心必须同时兼顾"快速概览"和"精细操作"两种用户意图。

思考

为什么项目管理中心是检验设计系统的最佳实战？

一个完整的项目管理中心页面几乎会用到设计系统中的所有组件和模式：

  • 色彩系统：统计卡片用品牌色，状态标签用功能色（成功/警告/错误）。
  • 排版系统：数字用大号加粗突出，辅助文字用小号灰色弱化。
  • 表单与反馈：新建项目需要表单，表单需要验证和错误提示。
  • 导航与布局：顶部导航、面包屑、页面标题构成信息架构。
  • 表格与数据展示：项目列表是典型的数据表格，需要排序、筛选、分页。
  • 响应式设计：统计卡片在手机端变为 2 列，表格支持横向滚动。
  • 设计令牌与主题：所有颜色、间距通过 CSS 自定义属性管理，支持主题切换。

如果你能独立完成这个项目管理中心的搭建，说明你已经具备了将设计系统规范应用到真实项目的能力。

原理

页面信息架构的"三段式"模型

B端页面的信息架构通常遵循"概览→详情→操作"的三段式模型：

  • 概览层（Overview）：用统计卡片、仪表盘图表告诉用户"整体情况如何"。这一层回答的是"有多少、完成了多少、还有多少"等量化问题。
  • 详情层（Detail）：用表格、列表展示具体数据。这一层回答的是"每一个项目的状态、负责人、进度是什么"。
  • 操作层（Action）：用按钮、表单提供创建、编辑、删除等操作入口。这一层回答的是"我能做什么"。

这三层从上往下排列，符合用户的认知顺序：先看整体，再关注细节，最后采取行动。任何打乱这一顺序的设计都会增加用户的认知负担。

数据到视图的映射原理

在实现项目管理中心时，前端需要将后端返回的原始数据映射为视觉呈现。这个映射过程涉及多个设计决策：

  • 状态映射：后端返回的状态码（如 0/1/2）需要映射为视觉标签（"进行中"/"已完成"/"已暂停"），并赋予不同的颜色语义。
  • 进度映射：0-100 的数字需要映射为进度条的长度，同时可以映射为颜色（如 <30% 红色、30-80% 蓝色、>80% 绿色）。
  • 优先级映射：高/中/低优先级可以映射为标签颜色、排序权重，甚至行背景色。

这些映射规则应当在设计系统中明确定义，避免每个开发者根据自己的理解随意实现。

论证

可复用模板对团队效率的影响

一项针对 50 个前端团队的调研表明，拥有标准化页面模板的团队，新功能开发速度比没有模板的团队快 35%，UI Bug 数量少 42%。这是因为模板消除了大量的"重复决策"——开发者不需要每次都想"这个页面的标题应该多大""表格和表单之间的间距应该是多少"。

设计系统的最终目标不是"限制创意"，而是"将低级决策自动化，让团队专注于高级决策"。页面模板就是这一理念的落地产物。

要点

  1. 每个页面有明确的"一页一事"：一个页面只做一件事（展示数据、填写表单、阅读文档）。不要把所有功能塞进一个页面。
  2. 页面结构保持一致：Header + Main + Footer 的大框架不变，具体内容在变。用户打开新页面时，不需要重新学习布局。
  3. 优先展示核心内容：页面加载后，用户第一眼应当看到最重要的信息，而不是被广告、提示、弹窗干扰。
  4. 面包屑用于深层级：如果页面嵌套超过两层（如 首页 / 教程 / 前端设计 / 色彩系统），用面包屑告诉用户当前位置。
  5. 预留扩展空间：模板不要写得太死，给未来可能增加的内容留位置。例如统计卡片行使用 Grid 或 Flexbox，新增卡片时自动适配。
  6. 空状态与加载状态不可少：模板中必须考虑数据为空、加载中、加载失败、权限不足等边界情况。

常见错误

❌ 每页结构都不一样

这个页面有侧边栏，那个没有，另一个侧边栏在右边。用户需要不断适应新布局，增加认知负担。同一产品内的页面结构应当保持一致。

❌ 模板写得太死

模板里所有尺寸、颜色都写死 px，复用时只能全盘复制，无法微调。应当用 CSS 变量和 class，方便覆盖和扩展。

❌ 忽略无数据状态

模板里放满了示例数据，但没有考虑真实场景中数据为空、加载失败、权限不足的情况。应当在开发时就为所有边界状态写好样式。

❌ 移动端完全未适配

项目管理中心的表格在手机上直接溢出屏幕。应当为表格添加横向滚动，或将关键信息以卡片形式重新组织。

实战项目：项目管理中心

需求清单

REQUIREMENTS

1. 统计卡片区域

  • 展示 4 个统计指标：总项目、进行中、已完成、待处理
  • 每个卡片包含指标名称和数值
  • 不同指标使用不同的品牌/功能色
  • 桌面端 4 列，手机端 2 列

2. 项目列表表格

  • 列：项目名称、状态、负责人、进度、操作
  • 状态用标签组件展示（进行中/已完成/已暂停）
  • 进度用进度条展示
  • 表格在小屏幕下支持横向滚动

3. 新建项目表单

  • 字段：项目名称（必填）、项目描述、优先级（高/中/低）
  • 提交前验证项目名称不为空
  • 提交成功后显示 Toast 提示，并清空表单

4. 空状态

  • 当项目列表为空时，显示空状态插画和"创建第一个项目"按钮
  • 空状态在不同主题下都能正常显示

验收标准

ACCEPTANCE CRITERIA

视觉规范

  • 所有颜色使用 CSS 自定义属性（--color-*），不硬编码色值
  • 间距使用 8px 网格系统（4/8/16/24/32px）
  • 字体大小遵循层级：标题 24px、卡片标题 16px、正文 14px、辅助 12px
  • 阴影使用统一的 shadow token（--shadow-1/2/3）

交互规范

  • 按钮 hover 时有 0.15s 的颜色过渡
  • 表格行 hover 时背景变色
  • 表单验证失败时，输入框边框变红并显示错误提示
  • 提交按钮在提交过程中显示加载状态并禁用

响应式规范

  • 桌面端（>1200px）：统计卡片 4 列，表格完整展示
  • 平板端（768-1200px）：统计卡片 2 列，表格完整展示
  • 手机端（<768px）：统计卡片 2 列，表格支持横向滚动，表单全宽

无障碍规范

  • 所有图片有 alt 文本
  • 表单输入框有关联的 label
  • 颜色不是传达信息的唯一方式（如错误状态同时有边框变色和文字提示）
  • 支持键盘 Tab 导航

常见问题排查

TROUBLESHOOTING

问题 1：表格在小屏幕上被挤压，文字重叠

解决方案：给表格外层容器添加 overflow-x: auto; 和 white-space: nowrap;。确保表格最小宽度大于容器宽度时会出现横向滚动条，而不是压缩列宽。

问题 2：统计卡片在手机上换行不整齐

解决方案：使用 CSS Grid 的 grid-template-columns: repeat(2, 1fr); 而不是 Flexbox。Grid 能确保两列严格等宽，而 Flexbox 的 flex: 1 在内容长度不同时可能导致列宽不一致。

问题 3：主题切换时部分组件颜色不更新

排查步骤：1) 检查该组件是否使用了硬编码颜色；2) 检查 CSS 自定义属性是否定义在正确的作用域（:root 或 [data-theme="dark"]）；3) 检查是否有内联 style 覆盖了 CSS 变量。

问题 4：表单提交后页面刷新，数据丢失

解决方案：在 form 的 submit 事件中调用 event.preventDefault()。如果使用原生 HTML form，未阻止默认行为会导致页面刷新。

问题 5：进度条动画在部分浏览器不生效

解决方案：避免使用 CSS transition 动画改变 width，这在低性能设备上可能卡顿。改用静态进度条，或通过 transform: scaleX() 实现硬件加速的动画。

扩展练习

扩展 1：为项目管理中心添加排序和筛选功能。点击表头按状态或进度排序，在表格上方添加搜索框和状态下拉筛选器。

扩展 2：实现项目的增删改查（CRUD）。点击"新建项目"弹出模态框表单，提交后将新项目添加到表格顶部。点击"删除"需要确认对话框。

扩展 3：将项目管理中心的数据持久化到 localStorage，刷新页面后数据不丢失。同时添加数据导出功能（导出为 CSV 或 JSON）。

扩展 4：为整个页面添加暗色模式支持。使用 prefers-color-scheme 检测系统主题，并提供手动切换按钮。

扩展 5：将项目管理中心改造为单页应用（SPA）。使用原生 JavaScript 实现路由切换：仪表盘 / 项目列表 / 新建项目 / 项目详情。

QUIZ

1. B端页面的信息架构通常遵循什么模型？

  • A. 漏斗模型
  • B. 概览→详情→操作的三段式模型
  • C. 瀑布流模型
  • D. 随机访问模型

答案：B。B端页面通常先展示整体概览（统计卡片），再展示具体详情（表格），最后提供操作入口（按钮/表单）。

2. 为什么项目管理中心是检验设计系统的最佳实战？

  • A. 因为它最难实现
  • B. 因为它几乎用到了设计系统中的所有组件和模式
  • C. 因为所有公司都需要项目管理中心
  • D. 因为它不需要后端接口

答案：B。项目管理中心涉及色彩、排版、表单、导航、表格、响应式、设计令牌等几乎所有设计系统要素。

3. 模板应当如何设计才能方便复用？

  • A. 所有尺寸和颜色都写死，确保一致性
  • B. 使用 CSS 变量和 class，预留覆盖和扩展的空间
  • C. 每个模板只适用于一个具体页面
  • D. 将所有样式内联到 HTML 中，减少文件依赖

答案：B。模板应当使用 CSS 变量和可复用的 class，在保持一致性的同时允许针对具体场景微调。
