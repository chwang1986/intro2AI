Step 8 表格与数据展示

信息密度、可读性、排序筛选与交互设计

组件层

场景

后台管理系统、数据报表、商品列表、用户管理……这些页面都要展示大量结构化数据。表格是最古老也是最有效的数据展示形式之一。与卡片、时间轴、图表等替代方案相比，表格在信息密度和数据对比方面具有无可替代的优势。

起源

表格的历史可以追溯到公元前 2000 年的美索不达米亚泥板，当时人们用行列结构记录粮食交易。 double-entry bookkeeping（复式记账法）在 15 世纪的意大利文艺复兴时期成熟，卢卡·帕乔利（Luca Pacioli）在 1494 年的《算术、几何、比及比例概要》中系统阐述了借贷记账法，表格成为商业世界的基础设施。

1979 年，Dan Bricklin 发明了 VisiCalc——第一款电子表格软件，被 Steve Jobs 称为"个人电脑的杀手级应用"。VisiCalc 让普通人也能在屏幕上创建和计算表格，彻底改变了商业数据处理的方式。

网页中的表格最早只是用来展示静态数据。随着 Web 应用的发展，表格逐渐具备了排序、筛选、分页、行内编辑等交互能力。今天，Ant Design、AG Grid 等组件库提供了功能完备的数据表格，但底层的设计原则仍然延续了纸质时代形成的共识：行列对齐、表头固定、数据右对齐。

思考

为什么表格比卡片更适合数据对比？

当用户需要对比多个对象的多项属性时，表格的效率远高于卡片。假设有 5 个用户，每个用户有姓名、邮箱、角色、状态、注册时间 5 个属性：

  • 卡片布局：每个用户一张卡片，5 张卡片垂直排列。用户需要在卡片之间来回扫视，才能对比同一属性（如"状态"）。视线移动路径呈"之"字形，认知负荷高。
  • 表格布局：5 个用户排成 5 行，5 个属性排成 5 列。用户只需水平扫视一行，就能对比同一属性在所有用户中的值。视线移动路径是直线，认知负荷低。

这是因为表格利用了"格式塔心理学"中的"接近律"和"连续律"：同一列的数据在垂直方向上连续排列，用户的大脑会自动将它们归为一组，从而快速发现模式和异常。

为什么数字要右对齐？

西方文字从左到右阅读，但数字的对比是从右到左进行的（个位→十位→百位）。当一列数字右对齐时，个位数在同一条垂直线上，用户扫视这一列就能快速比较大小。如果左对齐，个位数位置参差不齐，对比效率大幅降低。

原理

信息密度的科学

信息密度（Information Density）指单位面积内展示的数据量。表格的信息密度由三个因素决定：

  • 行高：标准行高 48px（Ant Design 默认），紧凑型 40px，宽松型 56px。行高越小，信息密度越高，但误点击风险增加；行高越大，可读性越好，但一屏展示的数据量减少。
  • 列宽：列宽应按内容类型分配。ID 列 80px、日期列 120px、名称列 200px+。过宽的列会浪费水平空间，过窄的列会导致内容换行或截断。
  • 内边距：单元格左右内边距通常为 16px，既保证文字不贴边，又不浪费空间。

信息密度与可读性是一对矛盾。高密度让用户看到更多数据，但增加了视觉疲劳；低密度让阅读更轻松，但需要更多滚动。设计目标是找到"最佳平衡点"——在典型屏幕上展示刚好足够决策的数据量。

视觉层级与扫描效率

人眼在浏览表格时并非逐字阅读，而是"扫描"。良好的视觉层级能引导扫描路径：

  • 表头区分：表头用较深的背景色或底部边框与数据行区分，让用户知道"这是列标题，不是数据"。
  • 斑马纹（Zebra Striping）：交替行背景色能帮助用户保持横向扫描时的位置感。但现代设计更倾向于 hover 高亮（悬停时整行变色），因为斑马纹在数据行较少时反而增加视觉噪音。
  • 悬停高亮：鼠标移到某一行时，该行背景变色。这让用户能明确追踪"当前关注的行"，尤其在宽表格中至关重要。

排序与筛选的交互模型

  • 排序（Sorting）：点击表头按该列排序。第一次点击升序，第二次降序，第三次取消排序。排序状态必须在表头用箭头图标明确指示。
  • 筛选（Filtering）：在表头或表格上方提供筛选控件。筛选条件应当即时生效，并在界面上显示当前生效的筛选条件（方便用户知道"数据为什么变少了"）。
  • 分页 vs 无限滚动：分页适合需要定位到特定页码的场景（如"我记得在第 3 页"）；无限滚动适合浏览型场景（如信息流）。后台管理系统通常使用分页，因为用户需要知道数据总量和当前位置。

论证

信息密度与可读性的平衡

MIT 的研究表明，在标准桌面显示器（1920×1080）上，一个 48px 行高的表格每屏可展示约 18 行数据。如果行高增加到 64px，每屏只剩 13 行，信息量减少 28%；如果行高压缩到 32px，每屏可展示 27 行，但用户更容易看错行、点错按钮。

Ant Design 选择 48px 作为默认行高，是基于以下考量：

  • 在 14px 字体大小下，48px 行高提供了约 2.5 倍的行间距，足以区分相邻行。
  • 48px 是 8px 网格系统的整数倍（48 = 8 × 6），符合设计系统的网格规范。
  • 48px 的高度为操作按钮（28-32px）留出了足够的垂直边距，防止误触。

固定表头的必要性

当表格数据超过一屏时，用户向下滚动后就会看不到表头。此时他们需要记住每一列的含义，这直接挑战了工作记忆的容量限制。固定表头（sticky header）将表头始终固定在视口顶部，让用户随时知道每一列的含义。研究表明，固定表头可以将大表格的任务完成时间缩短 15-20%。

要点

  1. 文本左对齐，数字右对齐，操作居中：这是表格对齐的黄金法则。日期、金额、百分比等可比较的数据必须右对齐，方便纵向对比。
  2. 表头固定：数据行超过一屏时，表头应当使用 position: sticky 固定在顶部。
  3. 长文本截断显示：用 text-overflow: ellipsis 截断过长内容，鼠标悬停时通过 tooltip 显示完整内容。
  4. 空状态要统一：所有"无数据"场景用同一个空状态组件，保持图标、文案、按钮风格一致。空状态要告诉用户"为什么没有数据"以及"可以做什么"。
  5. 加载状态分级：局部加载（表格内显示 spinner）、骨架屏（初次加载时模拟内容轮廓）、全屏加载（整页刷新时使用）。
  6. 操作列要醒目但不过度：编辑、删除等操作链接用品牌色，但不要用大按钮，避免与主按钮混淆。

    const rawData = [
      { name: '前端设计系统', status: '进行中', progress: 75, budget: 50000 },
      { name: 'Git 教程', status: '已完成', progress: 100, budget: 20000 },
      { name: 'Python 教程', status: '进行中', progress: 45, budget: 80000 },

      const tbody = document.getElementById('tableBody');
      tbody.innerHTML = data.map(row => `
        <td>${row.name}</td>
        <td><span class="tag tag-${row.status === '已完成' ? 'success' : row.status === '已暂停' ? 'error' : 'warning'}">${row.status}</span></td>

    function sortTable(col) {
      if (sortState.col === col) {
        sortState.dir = sortState.dir === 'asc' ? 'desc' : 'asc';
      } else {
        sortState = { col, dir: 'asc' };
      }
      currentData.sort((a, b) => {
        if (a[col] < b[col]) return sortState.dir === 'asc' ? -1 : 1;
        if (a[col] > b[col]) return sortState.dir === 'asc' ? 1 : -1;
        return 0;
      });
      document.querySelectorAll('th').forEach(th => th.classList.remove('sort-asc', 'sort-desc'));
      const th = document.querySelector(`th[data-col="${col}"]`);
      if (th) th.classList.add(sortState.dir === 'asc' ? 'sort-asc' : 'sort-desc');
      render(currentData);
    }

    function filterData() {
      const keyword = document.getElementById('searchInput').value.trim().toLowerCase();
      const status = document.getElementById('statusFilter').value;
      currentData = rawData.filter(row => {
        const matchName = row.name.toLowerCase().includes(keyword);
        const matchStatus = !status || row.status === status;
        return matchName && matchStatus;
      });
      render(currentData);
    }

    document.getElementById('searchInput').addEventListener('input', filterData);
    document.getElementById('statusFilter').addEventListener('change', filterData);
    render(currentData);

常见错误

❌ 表格没有横向滚动

列太多时在小屏幕上被挤压变形，文字重叠。给表格容器加 overflow-x: auto，确保小屏幕也能正常浏览。

❌ 数字左对齐

金额、百分比、ID 等数字左对齐后，个位、十位、百位不在同一垂直线上，用户无法快速比较大小。数字必须右对齐。

❌ 空状态没有引导操作

只写一句"暂无数据"，用户不知道接下来该做什么。应当加一个创建按钮或引导链接，帮助用户脱离空状态。

❌ 排序状态不明显

用户点击表头排序后，没有任何视觉反馈表明当前按哪列排序、是升序还是降序。应当在表头显示排序箭头图标。

示例

设计一个订单管理表格

假设你要为后台管理系统设计一个订单管理页面，运营人员每天需要处理 200+ 笔订单。如何从零开始设计一个既信息丰富又操作高效的表格？

1. 需求拆解

与产品经理沟通后，梳理出用户的核心诉求：

  • 浏览：一屏内看到尽可能多的订单，快速识别异常订单（超时未发货、退款中）。
  • 筛选：按状态（待付款/待发货/已发货/已完成）、时间范围、金额区间查找订单。
  • 排序：按下单时间、金额排序，找出最近的大额订单。
  • 操作：对单条订单执行"查看详情""发货""取消"等操作。

由此确定表格列：订单号、下单时间、客户信息、商品摘要、金额、状态、操作，共 7 列。

2. 信息密度决策

运营人员的显示器为 1920×1080，表格可用宽度约 1400px。如何分配空间？

  • 订单号：120px，等宽字体（font-variant-numeric: tabular-nums），方便视觉扫描。
  • 下单时间：160px，格式统一为 2024-01-15 14:32，避免换行。
  • 客户信息：200px，显示昵称 + 手机号后四位，超出截断。
  • 商品摘要：弹性列（min-width: 240px），用省略号截断，hover 显示 tooltip。
  • 金额：120px，右对齐，保留两位小数，千分位分隔。
  • 状态：100px，用带底色标签展示，颜色语义：待付款（橙）、待发货（蓝）、已发货（绿）、已完成（灰）。
  • 操作：140px，居中，放置"查看""发货"文字链接，避免大按钮争抢视觉焦点。

行高选择 48px（Ant Design 默认），在 1080p 屏幕上可展示约 16 行，配合分页每页 20 条，刚好覆盖一屏。

  1. 排序状态可视化：默认所有表头显示灰色 ↕，点击后变为蓝色 ↑ 或 ↓，其他列恢复灰色。

测验

QUIZ

1. 表格中的数字列应当如何对齐？

  • A. 左对齐，与文本保持一致
  • B. 居中对齐
  • C. 右对齐，方便纵向比较个位数
  • D. 两端对齐

答案：C。数字右对齐时，个位、十位、百位在同一垂直线上，用户可以快速比较大小。

2. 当表格数据超过一屏时，为什么要固定表头？

  • A. 为了美观
  • B. 用户滚动后仍能知道每一列的含义，减少记忆负担
  • C. 为了节省页面高度
  • D. 表头固定是 CSS 的默认行为，不需要理由

答案：B。固定表头让用户在滚动时始终看到列标题，不需要记忆列的顺序和含义。

3. 下列哪种场景更适合用卡片而不是表格？

  • C. 电商网站的商品橱窗浏览
