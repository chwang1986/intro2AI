Step 9 响应式设计

移动优先、流式布局、断点策略与媒体查询

进阶

场景

同一个页面，在 27 寸显示器、13 寸笔记本、iPad、手机上都要能正常显示。响应式设计不是"把 PC 页面缩小到手机上"，而是根据不同屏幕尺寸重新排列内容，保证每个设备上的体验都最佳。

起源

2000 年代初期，网页设计采用"固定宽度"模式——最常见的宽度是 960px，因为当时的显示器分辨率主要是 1024×768，960px 能留出适当的浏览器边框和滚动条空间。设计师会为 800×600、1024×768、1280×1024 等几种常见分辨率分别出图。

2010 年，Ethan Marcotte 在 A List Apart 上发表文章《Responsive Web Design》，正式提出了"响应式网页设计"的概念。他整合了三种已有技术：流式网格（fluid grids）、弹性图片（flexible images）和媒体查询（media queries），创造了一种能够适应任意屏幕尺寸的设计方法论。

同一时期，智能手机迅速普及。2010 年全球智能手机出货量超过 3 亿部，移动流量开始爆发。开发者面临一个选择：为 PC 和移动端分别维护两套代码（m.example.com），还是用一套代码适配所有设备？响应式设计提供了后者的技术方案，大大降低了维护成本。

思考

为什么移动优先？

响应式设计有两种策略："桌面优先（Desktop First）"和"移动优先（Mobile First）"。桌面优先先设计大屏布局，再用 max-width 媒体查询适配小屏；移动优先则相反，先设计小屏布局，再用 min-width 媒体查询扩展到大屏。

移动优先的优势在于：

  • 强迫聚焦：手机屏幕只有 375px 宽，你必须决定"什么是最重要的"。这种约束反而有助于理清内容优先级。
  • 渐进增强：小屏布局是功能的最小可用版本，大屏上再添加增强功能（如侧边栏、多列布局）。这符合"为所有人提供核心功能"的无障碍原则。
  • 性能优化：移动设备通常网络较慢、性能较弱。移动优先迫使开发者从最开始就优化性能，而不是事后补救。
  • CSS 更简洁：min-width 媒体查询更符合直觉——"当屏幕大于 X 时，添加这些样式"，而不是"当屏幕小于 X 时，移除这些样式"。

为什么断点不能随意选择？

很多初学者会根据具体设备的宽度来设断点："iPhone 是 375px，iPad 是 768px，所以我设两个断点就够了。"这种做法的问题在于：设备尺寸是连续分布的，每年都有新尺寸的设备出现。基于设备的断点很快会过时。

正确的做法是"基于内容选择断点"——不断缩小浏览器窗口，当布局开始"崩坏"时（如文字换行、元素重叠），那个宽度就是你的断点。这样选择的断点是"内容驱动"的，不依赖于具体设备。

原理

视口（Viewport）的底层机制

移动浏览器的默认行为是模拟桌面浏览器的宽度（通常是 980px），然后将页面缩小到手机屏幕大小。这就是为什么如果不加 viewport meta 标签，手机上的网页会显示为微缩的桌面版。

<meta name="viewport" content="width=device-width, initial-scale=1.0"> 告诉浏览器："这个页面的宽度等于设备宽度，不要缩放。"这是响应式设计的基石。

媒体查询的解析机制

CSS 媒体查询（Media Query）不是"条件编译"，而是浏览器在解析样式时动态匹配的。当窗口大小改变时，浏览器会重新计算匹配哪些媒体查询，并应用对应的样式。这个过程是实时的，不需要刷新页面。

媒体查询的匹配顺序很重要：CSS 的层叠规则（Cascading）意味着后匹配的样式会覆盖先匹配的样式（如果优先级相同）。因此，移动优先的媒体查询应当按断点从小到大排列：

/* 基础样式（手机） */
.container { padding: 16px; }

/* 平板 */
@media (min-width: 768px) {
  .container { padding: 24px; }
}

/* 桌面 */
@media (min-width: 1200px) {
  .container { max-width: 1200px; margin: 0 auto; }
}

弹性单位的选择策略

响应式设计中常用的弹性单位：

  • 百分比（%）：相对于父元素。适合宽度、高度等需要随容器缩放的场景。缺点是无法确定最终像素值，嵌套时容易混淆。
  • em/rem：相对于字体大小。em 相对于当前元素的字体大小，rem 相对于根元素（html）的字体大小。适合间距、边距等需要随文字缩放的比例关系。
  • vw/vh：相对于视口宽/高的 1%。适合需要与视口直接关联的场景，如全屏背景、大字标题。
  • fr：Grid 布局中的分数单位，表示剩余空间的分配比例。适合网格布局中的列宽定义。

论证

断点选择的科学依据

Ant Design 的断点系统（xs: <576px, sm: 576-768px, md: 768-992px, lg: 992-1200px, xl: >1200px）并非随意设定，而是基于设备尺寸的统计学分布：

  • 576px：大屏手机（如 iPhone Plus 系列）的宽度边界。
  • 768px：iPad 竖屏宽度，是手机和平板的分界线。
  • 992px：iPad 横屏和较小笔记本的宽度，是平板和桌面的分界线。
  • 1200px：主流桌面显示器的常用内容区宽度，超过此宽度后内容不应无限拉伸（否则阅读距离过长）。

实际项目中，2-3 个断点足够覆盖 95% 的场景。断点越多，代码越复杂，维护成本越高。

触摸目标的最小尺寸

MIT Touch Lab 的研究发现，成年人食指的接触面积平均为 10-14mm（约 40-55px）。Apple 的 Human Interface Guidelines 建议触摸目标至少为 44×44px，Google 的 Material Design 建议 48×48px。小于这个尺寸，用户的误操作率会显著上升。

这意味着：移动端的按钮、链接、表单输入框必须有足够的高度。Ant Design 的移动端按钮最小高度为 36px，但在核心操作按钮上建议达到 40px 以上。

要点

  1. 用 Flexbox 和 Grid 做布局：它们天生支持响应式，配合 flex-wrap 和 gap 可以轻松实现多列变单列。
  2. 图片要自适应：max-width: 100%; height: auto;，防止图片撑破容器。
  3. 触摸目标不小于 44×44px：按钮、链接在手机上要足够大，方便手指点击。
  4. 隐藏非核心内容：手机端可以把侧边栏收起、把次要信息折叠，优先展示核心内容。
  5. 测试真实设备：Chrome 的开发者工具只能模拟尺寸，不能模拟真实的触摸、滚动性能。必须在真机上测试。
  6. 使用相对单位：字体大小用 rem，间距用 px 或 rem，宽度用 % 或 fr。避免在布局中使用固定 px 值定义宽度。

实验

将以下代码复制到你的编辑器中，保存为 responsive-experiment.html，用浏览器打开。调整窗口大小，观察三栏布局如何变为单栏。

<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>响应式布局实验</title>
  <style>
    * { margin: 0; padding: 0; box-sizing: border-box; }
    body { font-family: system-ui, sans-serif; background: #F5F7FA; }

    /* 移动优先：单列 */
    .container { padding: 16px; }
    .card {
      background: white; border-radius: 8px; padding: 24px;
      box-shadow: 0 1px 2px rgba(0,0,0,0.05); margin-bottom: 16px;
    }
    .card h3 { margin-bottom: 8px; color: #1677FF; }
    .card p { color: #595959; font-size: 14px; line-height: 1.6; }

    /* 断点 1：平板，两列 */
    @media (min-width: 768px) {
      .container { padding: 24px; max-width: 1200px; margin: 0 auto; }
      .grid { display: grid; grid-template-columns: 1fr 1fr; gap: 16px; }
      .card { margin-bottom: 0; }
      .card-full { grid-column: 1 / -1; }
    }

    /* 断点 2：桌面，三列 */
    @media (min-width: 1200px) {
      .grid { grid-template-columns: 1fr 1fr 1fr; }
    }

    /* 断点指示器 */
    .breakpoint {
      position: fixed; bottom: 16px; right: 16px;
      background: #1677FF; color: white; padding: 8px 16px;
      border-radius: 4px; font-size: 14px; font-weight: 600;
    }
    @media (min-width: 768px) { .breakpoint::after { content: '平板 (≥768px)'; } }
    @media (min-width: 1200px) { .breakpoint::after { content: '桌面 (≥1200px)'; } }
    @media (max-width: 767px) { .breakpoint::after { content: '手机 (<768px)'; } }
  </style>
</head>
<body>
  <div class="container">
    <div class="grid">
      <div class="card">
        <h3>卡片 1</h3>
        <p>这是第一张卡片的内容。在手机端，卡片垂直排列；在平板端变为两列；在桌面端变为三列。</p>
      </div>
      <div class="card">
        <h3>卡片 2</h3>
        <p>这是第二张卡片的内容。观察 Grid 布局如何根据屏幕宽度自动调整列数。</p>
      </div>
      <div class="card">
        <h3>卡片 3</h3>
        <p>这是第三张卡片的内容。不需要写任何 JavaScript，纯 CSS 媒体查询就能实现响应式。</p>
      </div>
      <div class="card card-full">
        <h3>通栏卡片</h3>
        <p>这张卡片在平板上会占据两列的宽度（grid-column: 1 / -1），用来展示需要更多空间的内容，比如表格或图表。</p>
      </div>
    </div>
  </div>
  <div class="breakpoint">当前断点：</div>
</body>
</html>

实验任务：

  1. 在手机宽度（< 768px）下观察：卡片垂直排列，内边距为 16px。
  2. 将窗口拖到 768px-1200px 之间，观察：卡片变为两列，内边距增加为 24px，容器居中。
  3. 将窗口拖到 > 1200px，观察：卡片变为三列。
  4. 注意右下角的断点指示器，它随窗口宽度变化而更新。
  5. 修改代码，在 768px 断点下将"通栏卡片"改为只占据一列（删除 .card-full 的样式），观察布局变化。
  6. 尝试将 Grid 改为 Flexbox 实现（提示：display: flex; flex-wrap: wrap;），对比两者在响应式场景下的差异。

常见错误

❌ 用 px 定义所有宽度

一个 400px 宽的卡片在手机上会撑出屏幕。应当用 max-width 限制最大宽度，用百分比或 flex 分配空间。

❌ 断点太多太细

定义了 7 个断点，每个都要单独写样式，维护困难。2-3 个断点足够覆盖 95% 的场景。基于内容选择断点，而不是基于设备型号。

❌ 忽略横屏模式

手机横过来后，高度变小，原本竖屏适配的布局可能会出问题。要测试横屏，尤其是固定定位的元素（如底部按钮栏）。

❌ 只在 Chrome DevTools 中测试

DevTools 的设备模拟只能模拟视口尺寸和 UA，无法模拟真实设备的字体渲染、触摸事件、滚动惯性和性能特征。必须在真机上验证。

完整代码

/* 基础样式（移动端优先） */
.container {
  width: 100%;
  padding-left: 16px;
  padding-right: 16px;
}

.card-grid {
  display: flex;
  flex-direction: column;
  gap: 16px;
}

/* 平板 */
@media (min-width: 768px) {
  .container {
    padding-left: 24px;
    padding-right: 24px;
  }
  .card-grid {
    flex-direction: row;
    flex-wrap: wrap;
  }
  .card-grid > * {
    flex: 1 1 calc(50% - 8px);
  }
}

/* 桌面 */
@media (min-width: 1200px) {
  .container {
    max-width: 1200px;
    margin-left: auto;
    margin-right: auto;
  }
  .card-grid > * {
    flex: 1 1 calc(33.333% - 11px);
  }
}

练习

练习 1：用 Flexbox 实现一个三列卡片布局，要求在手机上一列、平板两列、桌面三列。不借助任何 CSS 框架。

练习 2：打开你的项目，在 Chrome DevTools 的手机模拟模式下测试每个页面，记录布局崩坏的地方，并制定修复计划。

练习 3：为前端设计系统教程的侧边栏添加响应式逻辑：PC 显示侧边栏，手机隐藏侧边栏并添加一个汉堡菜单按钮。

QUIZ

1. 响应式设计中，为什么推荐"移动优先"策略？

  • A. 移动端用户比桌面端多
  • B. 手机屏幕小，能强迫设计师聚焦核心内容，且 CSS 用 min-width 更直观
  • C. 移动端开发更简单
  • D. 搜索引擎要求移动优先

答案：B。移动优先强迫聚焦核心内容，且 min-width 媒体查询符合渐进增强的逻辑。

2. 断点应该基于什么来选择？

  • A. 具体设备的宽度（如 iPhone 375px、iPad 768px）
  • B. 内容开始崩坏的宽度
  • C. 设计师的个人喜好
  • D. 行业平均值 960px

答案：B。断点应当基于内容——当布局开始崩坏时，就是需要断点的时候。基于设备的断点会很快过时。

3. 移动端的触摸目标最小尺寸建议是多少？

  • A. 24×24px
  • B. 32×32px
  • C. 44×44px（Apple）或 48×48px（Google）
  • D. 60×60px

答案：C。Apple HIG 建议 44×44px，Material Design 建议 48×48px，这是基于成年人食指平均接触面积的研究数据。
