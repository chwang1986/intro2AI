Step 10 设计令牌与主题

CSS 自定义属性、令牌化系统、亮暗模式切换与可维护性

实战

场景

一个产品需要支持亮色模式和暗色模式；另一个产品需要将品牌色从蓝色改为紫色；还有一个产品需要在不同子品牌之间共享组件但使用不同的主题。如果没有系统化的颜色、间距、字体管理，这些需求会导致大量的"全局搜索替换"，极易出错。设计令牌（Design Tokens）就是解决这一问题的核心机制。

起源

设计令牌的概念由 Salesforce 的设计团队于 2014 年正式提出。他们在维护 Lightning Design System 时发现：设计师用 Sketch 的样式定义颜色，开发者用 Sass 变量定义颜色，产品经理用文档记录颜色——三个来源很容易不同步。当需要更新品牌色时，设计师改了一个地方，开发者改了另一个地方，导致上线后颜色不一致。

Salesforce 的解决方案是将颜色、字体、间距等设计决策抽象为平台无关的"令牌"（Tokens）。这些令牌以 JSON 格式存储，作为单一事实来源（Single Source of Truth）。设计师、开发者、产品经理都引用同一个令牌名，而不是具体的色值。

CSS 自定义属性（Custom Properties，又称 CSS 变量）在 2015 年随着 Chrome 49 和 Firefox 31 的普及，成为在 Web 端实现设计令牌的最佳技术方案。与 Sass/Less 等预处理器变量不同，CSS 自定义属性在运行时生效，可以被 JavaScript 动态修改，从而实现主题切换、无障碍适配等动态需求。

思考

为什么设计令牌是设计系统的核心？

设计系统的本质是"一致性"。一致性的敌人是"硬编码"——当每个组件都直接使用 #1677FF 这个色值时，一旦品牌色改变，你需要找到所有使用这个色值的地方并修改。在大中型项目中，这可能涉及数百个文件、数千个位置。

设计令牌引入了"间接层"。你不再说"这个按钮用 #1677FF"，而是说"这个按钮用 --color-primary"。按钮不关心 --color-primary 到底是蓝色还是紫色，它只关心这是"主色"。这种解耦带来了三个好处：

  • 一处修改，全局生效：改变 --color-primary 的定义，所有使用它的组件自动更新。
  • 语义化管理：--color-primary 比 #1677FF 更有语义，新成员一看就知道这是"主品牌色"。
  • 多主题支持：同一套组件可以加载不同的令牌值，实现亮/暗模式或子品牌定制。

为什么 CSS 自定义属性比 Sass 变量更适合主题切换？

Sass 变量在编译时确定值，生成的是静态 CSS。如果你想在运行时切换主题，需要为每个主题生成一套完整的 CSS 文件，然后通过加载不同文件来切换。

CSS 自定义属性在运行时生效。你可以在一个 :root 选择器中定义亮色主题的令牌值，在 [data-theme="dark"] 选择器中定义暗色主题的令牌值。切换主题时，只需要修改 HTML 元素的 data-theme 属性，所有使用这些令牌的组件会即时更新，无需重新加载 CSS 文件。

原理

CSS 自定义属性的底层机制

CSS 自定义属性的底层实现是浏览器渲染引擎中的"属性级联"。与普通的 CSS 属性不同，自定义属性的值不是立即解析的，而是在使用时（通过 var() 函数）才进行计算。这种"惰性求值"机制带来了几个重要特性：

  • 作用域继承：自定义属性遵循 CSS 的层叠和继承规则。子元素可以继承父元素的自定义属性值，也可以在自己的选择器中覆盖。
  • 运行时修改：通过 JavaScript 的 element.style.setProperty('--name', 'value') 可以实时修改自定义属性，所有引用它的样式会立即重绘。
  • 回退机制：var(--name, fallback) 允许在变量未定义时使用默认值，增强了代码的健壮性。

令牌的层级结构

成熟的设计令牌系统通常分为三个层级（Salesforce 的命名方式）：

  • 全局令牌（Global Tokens）：最基础的设计决策，如 --blue-500: #1677FF; --spacing-1: 4px;。这些令牌对应设计工具的原始值，通常不直接在组件中使用。
  • 别名令牌（Alias Tokens）：将全局令牌映射到语义化名称，如 --color-primary: var(--blue-500); --color-error: var(--red-500);。别名令牌表达的是"这个颜色用来做什么"。
  • 组件令牌（Component Tokens）：针对特定组件的令牌，如 --button-bg: var(--color-primary); --button-bg-hover: var(--color-primary-hover);。组件令牌将别名令牌绑定到具体组件的具体状态。

三层结构的好处是：全局令牌改变不影响语义（蓝色换成另一种蓝色），别名令牌改变不影响组件逻辑（主色从蓝变紫），组件令牌改变只影响特定组件。

暗色模式的实现原理

暗色模式不仅仅是"把白色背景换成黑色"。人眼在暗光环境下对对比度的感知会发生变化——同样的灰度差在暗背景下看起来更不明显。因此，暗色模式需要重新调整整个色板：

  • 背景色：从浅色（#FFFFFF → #F5F7FA）变为深色（#141414 → #1F1F1F）。
  • 文字色：从黑色系（rgba(0,0,0,0.88)）变为白色系（rgba(255,255,255,0.85)）。
  • 边框色：在暗色模式下，边框应当比背景色稍亮，而不是稍暗。
  • 阴影：暗色模式下的阴影应当用"发光"（高亮背景）替代，因为深色上的黑色阴影几乎不可见。

论证

令牌化对可维护性的量化影响

假设一个中型项目有 200 个组件，每个组件平均使用 10 个颜色值。如果没有令牌化，项目中会有 2000 个硬编码的颜色引用。当品牌色改变时，需要修改 2000 个位置，即使使用全局搜索替换，也存在误改风险（如将非品牌色但色值相同的颜色也替换掉）。

引入令牌化后，同样的项目可能只有 50 个全局/别名令牌。品牌色改变只需要修改 1 个令牌的定义。维护工作量从 O(n) 降到了 O(1)。

更关键的是"心理成本"：开发者在写样式时，不需要记住"这个按钮的 hover 色是 #4096FF 还是 #3b82f6"，只需要记住"用 --color-primary-hover"。这减少了认知负荷，也降低了出错概率。

暗色模式的用户价值

Google 的研究表明，在暗光环境下使用暗色模式可以将屏幕能耗降低最高 60%（对 OLED 屏幕而言，因为黑色像素不发光）。此外，暗色模式在低光环境下可以减少眼睛疲劳，这也是 iOS 和 Android 都引入系统级暗色模式的原因。作为一个前端设计系统，支持暗色模式已经不再是"加分项"，而是"基本项"。

要点

  1. 令牌命名要有语义：用 --color-primary 而不是 --color-blue，用 --spacing-lg 而不是 --spacing-16px。语义化命名让令牌的目的清晰，不受具体值变化的影响。
  2. 区分全局、别名和组件令牌：不要直接在组件中引用全局色值，而是通过别名令牌过渡。这保持了系统的灵活性和可维护性。
  3. 暗色模式不是简单的反色：需要为暗色模式单独定义一套色板，确保对比度、可读性和视觉层次在暗背景下仍然成立。
  4. 使用 CSS 自定义属性而非预处理器变量：只有 CSS 自定义属性支持运行时修改，这是实现主题切换的前提。
  5. 提供系统级主题适配：用 @media (prefers-color-scheme: dark) 检测用户的系统主题偏好，自动匹配。
  6. 令牌值要有文档：每个令牌应当说明其用途、使用场景和变更历史，帮助团队成员正确使用。

  <script>
    function setTheme(mode) {
      if (mode === 'auto') {
        const prefersDark = window.matchMedia('(prefers-color-scheme: dark)').matches;
        document.documentElement.setAttribute('data-theme', prefersDark ? 'dark' : 'light');
      } else {
        document.documentElement.setAttribute('data-theme', mode);
      }
      localStorage.setItem('theme', mode);
    }

    // 页面加载时恢复主题
    const saved = localStorage.getItem('theme') || 'auto';
    setTheme(saved);

    // 监听系统主题变化
    window.matchMedia('(prefers-color-scheme: dark)').addEventListener('change', e => {
      if (localStorage.getItem('theme') === 'auto') {
        document.documentElement.setAttribute('data-theme', e.matches ? 'dark' : 'light');
      }
    });
  </script>
</body>

常见错误

❌ 直接在组件中硬编码颜色

组件中写死 #1677FF，导致需要主题切换时无法适配。所有颜色都应当通过 var() 引用令牌。

❌ 暗色模式只是反色

将白色背景换成黑色，黑色文字换成白色，结果很多元素的对比度严重不足。暗色模式需要重新设计整个色板，尤其是边框、阴影和次要文字。

❌ 令牌命名过于具体

用 --blue-500 作为按钮背景色令牌名，当品牌色从蓝色变为紫色时，令牌名就失去了语义。应当使用 --color-primary 这样的语义化名称。

❌ 忘记处理过渡动画

主题切换时，颜色瞬间跳变，用户体验生硬。应当在 body 和常用元素上添加 transition: background 0.3s, color 0.3s，让切换更平滑。

  --blue-500: #1677FF;
  --color-primary: var(--blue-500);
  --bg-page: #F5F7FA;
  --bg-card: #FFFFFF;
  --text-primary: rgba(0, 0, 0, 0.88);

  --bg-page: #141414;
  --bg-card: #1F1F1F;
  --text-primary: rgba(255, 255, 255, 0.85);
  --shadow-card: none;
}

/* 组件使用令牌 */
.btn-primary {
  background: var(--button-bg);
  color: #fff;
}

/* ===== 设计令牌定义 ===== */
:root {
  /* 全局令牌 */
  --blue-500: #1677FF;
  --green-500: #52C41A;
  --red-500: #F5222D;

  /* 别名令牌 */
  --color-primary: var(--blue-500);
  --color-success: var(--green-500);
  --color-error: var(--red-500);

  /* 组件令牌 */
  --button-bg: var(--color-primary);
  --button-bg-hover: var(--color-primary-hover);
  --input-border-error: var(--color-error);
}

/* ===== 暗色主题 ===== */
[data-theme="dark"] {
  --blue-500: #3B82F6;
  --color-primary: var(--blue-500);
  --bg-page: #141414;
  --bg-card: #1F1F1F;
  --text-primary: rgba(255, 255, 255, 0.85);
}

/* ===== 组件使用令牌 ===== */
.btn-primary {
  background: var(--button-bg);
}
.btn-primary:hover {
  background: var(--button-bg-hover);
}

练习

练习 1：为你正在开发的项目创建一套基础设计令牌，至少包含 5 个颜色令牌（主色、成功、警告、错误、背景）和 3 个间距令牌（sm、md、lg）。

练习 2：用 CSS 自定义属性实现一个支持亮/暗模式切换的个人博客页面。页面包含导航栏、文章列表和页脚，所有元素的颜色都要随主题变化。

练习 3：将本教程的所有组件样式整理为令牌化形式。找出所有硬编码的颜色值，替换为 CSS 自定义属性引用。

测验

QUIZ

1. CSS 自定义属性与 Sass 变量的核心区别是什么？

  • A. 自定义属性只能在 :root 中定义
  • B. 自定义属性在运行时生效，可以被 JavaScript 动态修改；Sass 变量在编译时确定
  • C. Sass 变量支持回退值，自定义属性不支持
  • D. 自定义属性不支持嵌套作用域

答案：B。CSS 自定义属性的运行时特性使其成为主题切换的理想选择，而 Sass 变量生成的是静态 CSS。

2. 设计令牌通常分为哪三个层级？

  • A. 颜色、字体、间距
  • B. 全局令牌、别名令牌、组件令牌
  • C. HTML、CSS、JavaScript
  • D. 原子、分子、有机体

答案：B。全局令牌存储原始值，别名令牌赋予语义，组件令牌绑定到具体组件状态。

3. 暗色模式的背景色为什么不使用纯黑色（#000000）？

  • A. 纯黑色在 OLED 屏幕上太省电了
  • B. 纯黑色会导致文字对比度过高，长时间阅读容易疲劳；且会丢失阴影和层级的表达
  • C. 浏览器不支持纯黑色渲染
  • D. 纯黑色文件体积更大

答案：B。暗色模式通常使用深灰色（如 #141414）作为背景，因为纯黑的对比度过高，且无法通过阴影表达层级。
