Prompt 工程

学会「精准提问」——让 AI 一次性听懂你的需求

场景为什么 AI 总是答非所问？

想象你走进一家餐厅，对服务员说："给我来点吃的。"结果上来了一份你不爱吃的菜。这不是服务员的问题，而是你的指令太模糊了。

和 AI 对话也是如此。很多人抱怨"AI 很蠢"，其实是因为Prompt（提示词）写得像随手涂鸦，而不是清晰的施工图纸。Prompt 工程就是一门「如何精准表达需求」的艺术——让 AI 在第一次尝试时，就给出你想要的答案。

例如，与其说"写个 Python 函数"，不如说"你是一个 Python 专家，请写一个函数，接收一个字符串列表，返回按长度排序后的列表，要求使用类型注解并处理空列表的边界情况。"后者得到的代码质量天差地别。

起源从填词游戏到 Prompt 工程

要理解 Prompt 工程，我们先得理解 AI 模型是怎么"思考"的。现代大语言模型的核心技术是 Transformer，由 Google 研究团队在 2017 年的论文《Attention Is All You Need》中提出。它本质上是一个超级强大的"填词游戏玩家"——你给它一段文字，它预测下一个最可能出现的词，然后把这个词接上去，再预测下一个，如此循环。

OpenAI 的 GPT 系列就是基于 Transformer 的杰出代表：2018 年的 GPT-1 有 1.17 亿参数，2019 年的 GPT-2 有 15 亿参数，2020 年的 GPT-3 突然飙升到 1750 亿参数。参数越多，模型从训练数据中学到的模式就越丰富，"填词"能力就越强。

但模型本身只是一个"自动补全机器"，它并不真正"理解"你的意图。Prompt 工程这个概念大约在 2020~2021 年随着 GPT-3 的商用而兴起。人们发现：同样一个模型，用不同的方式提问，得到的答案质量天差地别。2022 年，Google Research 提出了 Chain-of-Thought（思维链）提示技术，让模型"一步一步想"，大幅提升了推理能力。Prompt 工程从此从"玄学"变成了一门"科学"。

思考为什么 AI "听不懂人话"？

很多人第一次用 AI 时都会有这种挫败感："我已经说得很清楚了啊，为什么它还是给不出我要的答案？"要回答这个问题，我们需要换个角度——不是从"人怎么理解语言"的角度，而是从"模型怎么生成文本"的角度。

想象你正在玩一个填词游戏。主持人说："请写一个____。"你的答案会是什么？可能是"故事"、"程序"、"报告"——你根本不知道对方想要什么。但如果主持人说："你是一位资深 Python 工程师，请写一个____，功能是接收列表返回排序结果。"这时你填"函数"的概率就大大增加了。

AI 模型面临的困境就是这样：它接收到的每一个 Prompt 都是在激活训练数据中的某些"模式"。Prompt 太模糊 = 激活太多无关模式 = 输出混乱。

早期的 NLP 研究者试图通过"微调模型"（Fine-tuning）来解决这个问题——用特定领域的数据重新训练模型。但微调成本高、技术门槛高。Prompt 工程则提供了一种更廉价的替代方案：不改变模型，只改变输入。就像你不需要重新训练一个翻译官，只需要把话说得更清楚。

原理好 Prompt = 角色 + 任务 + 约束 + 格式

AI 模型的「大脑」是一个巨大的概率机器，它根据你提供的上下文来预测下一个最可能出现的词。Prompt 就是你给这台机器设定的上下文窗口——窗口里的信息越完整、结构越清晰，AI 的「猜测」就越准确。

一个高质量的 Prompt 就像一份完整的建筑设计图：先说明角色（谁来做）、再描述任务（做什么）、然后设定约束（边界条件）、最后指定格式（输出样式）。四个要素缺一不可。

论证为什么结构化 Prompt 比随意提问有效？

有些同学可能会质疑："我随便写个问题，AI 也能回答啊，为什么要费心思设计 Prompt？"让我们从认知科学和概率模型的角度，论证结构化 Prompt 的价值。

1. 角色设定的原理：激活特定知识区域

大语言模型在训练时读了互联网上的海量文本，包括技术文档、小说、论坛帖子、论文等等。当你说"你是一位 Python 专家"时，你实际上是在提高模型激活"技术文档风格"和"Python 相关知识"的概率，同时抑制"网络段子"和"口语文体"的概率。这类似于给搜索引擎加了精准的关键词过滤。

2. Few-shot vs Zero-shot：示例的力量

认知心理学中有一个概念叫"格式塔原理"——人类擅长从例子中归纳模式。AI 模型也是如此。Zero-shot（零示例）是"直接问"，模型只能依赖训练时的泛化能力。Few-shot（给几个示例）则是"做给你看"，模型可以从示例中捕捉到输出格式、风格、深度等细粒度信息。研究表明，在某些分类任务上，Few-shot Prompt 的准确率可以比 Zero-shot 高出 20% 以上。

3. 约束条件：缩小概率空间

模型生成每一个词时，都在一个巨大的词表上做概率选择。Prompt 中的约束条件（如"用 JSON 格式""不超过 100 字"）相当于在概率空间中画了一个圈，让模型只在圈内采样。约束越明确，"跑偏"的概率越低。

4. 反面论证：为什么过度复杂的 Prompt 也不好？

Prompt 不是越长越好。模型的上下文窗口有限（虽然现在已经很大了），且过长的 Prompt 会稀释关键信息。研究发现，模型对 Prompt 开头和结尾的信息最敏感，中间部分容易被"遗忘"。所以好的 Prompt 应该像好文章：结构清晰、重点突出、没有废话。

要点五个技巧让 Prompt 脱胎换骨

下面五个技巧是 Prompt 工程的「黄金法则」，掌握它们，你的 AI 调用成功率将大幅提升：

1. 角色设定：给 AI 一个明确的身份，它会自动调动相关知识领域。"你是一位资深 Python 后端工程师" 比 "请写代码" 效果好十倍。

2. 明确任务：把大任务拆解为小步骤。不要说"帮我做一个网站"，而要说"第一步，设计用户登录的数据模型；第二步，写登录 API 的视图函数"。

3. 提供示例（Few-shot）：给 AI 看几个正确示例，它就能「举一反三」。这是让 AI 理解你期望风格的最快方式。

4. 约束输出格式：明确要求 JSON、Markdown 表格、代码块等格式，避免解析噩梦。

5. 迭代优化：第一次结果不完美很正常，把问题反馈给 AI，像打磨钻石一样反复雕琢。

代码示例
import os
import requests
import json

API_KEY = os.getenv("OPENAI_API_KEY")
API_URL = "https://api.openai.com/v1/chat/completions"

def build_structured_prompt(user_request):
    """
    构建一个结构化的 Prompt，让 AI 一次性输出高质量结果。
    就像给厨师一张完整的订单：角色 + 任务 + 示例 + 约束 + 格式。
    """
    
    system_prompt = """你是一位经验丰富的 Python 代码审查专家。
你的任务是分析用户提供的 Python 代码，找出潜在的 Bug 和性能问题。
请严格按照以下 JSON 格式输出，不要添加任何额外解释。"""

    # Few-shot 示例：给 AI 看一个正确输出的样板
    few_shot_example = """
示例输入：
def get_user(user_id):
    return database.query("SELECT * FROM users WHERE id = " + user_id)

示例输出：
{
  "issues": [
    {
      "severity": "high",
      "line": 2,
      "description": "SQL 注入风险：直接拼接用户输入到 SQL 语句中",
      "fix": "使用参数化查询：database.query('SELECT * FROM users WHERE id = ?', (user_id,))"
    }
  ],
  "overall_score": 3,
  "summary": "存在严重的 SQL 注入安全漏洞，需立即修复"
}
"""

    user_prompt = f"""请分析以下代码：

{user_request}

要求：
1. 找出所有 Bug、安全漏洞和性能问题
2. 每个问题标注严重程度（high / medium / low）
3. 给出具体的修复代码
4. 返回有效的 JSON，不要 Markdown 代码块包裹
5. 必须包含字段：issues（列表）、overall_score（1-10）、summary（字符串）"""

    headers = {
        "Authorization": f"Bearer {API_KEY}",
        "Content-Type": "application/json"
    }
    
    payload = {
        "model": "gpt-4o-mini",
        "messages": [
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": few_shot_example},
            {"role": "user", "content": user_prompt}
        ],
        "temperature": 0.2,  # 低温度让输出更稳定、更可预测
        "response_format": {"type": "json_object"}  # 强制 JSON 输出
    }
    
    response = requests.post(API_URL, json=payload, headers=headers, timeout=30)
    response.raise_for_status()
    
    data = response.json()
    result_text = data["choices"][0]["message"]["content"]
    
    # 解析并验证 JSON 结构
    result_json = json.loads(result_text)
    assert "issues" in result_json, "响应缺少 issues 字段"
    assert "overall_score" in result_json, "响应缺少 overall_score 字段"
    
    return result_json


# 使用示例：让 AI 审查一段有问题的代码
code_to_review = '''
def calculate_average(numbers):
    total = 0
    for n in numbers:
        total += n
    return total / len(numbers)
'''

result = build_structured_prompt(code_to_review)
print(json.dumps(result, indent=2, ensure_ascii=False))

实验亲手对比不同 Prompt 的效果

下面三个实验可以用任何大模型工具完成（如 ChatGPT、Claude、文心一言等）。重点是观察同一个模型在不同 Prompt 下的表现差异。

实验 1：角色设定的威力

# 第一轮：不加角色
"请解释什么是递归。"

# 第二轮：加角色
"你是一位有 20 年教龄的计算机科学教授，擅长用生活中的类比解释复杂概念。
请向大一新生解释什么是递归，要求：
1. 用一个生活中的类比
2. 说明递归的两个必要条件（基准情况和递归步骤）
3. 举一个 Python 代码例子"

# 观察对比：
# - 第一轮的回答是否更干巴巴、更像维基百科？
# - 第二轮的回答是否有更生动的例子、更清晰的结构？

实验结论：角色设定不仅改变语言风格，还会改变内容深度和组织方式。

实验 2：Few-shot 的魔力

# 第一轮：Zero-shot（直接问）
"请将以下文本按情感分类为正面或负面：
这家餐厅的服务太糟糕了。"

# 第二轮：Few-shot（给示例）
"请将以下文本按情感分类为正面或负面。

示例 1：
文本：这部电影真的太精彩了！
情感：正面

示例 2：
文本：完全浪费时间，剧情毫无逻辑。
情感：负面

现在请分类：
文本：这家餐厅的服务太糟糕了。
情感："

# 观察：Few-shot 是否让输出格式更统一？是否减少了"废话"？

实验结论：示例就像"模板"，让模型知道你想要什么格式、什么风格、什么深度。

实验 3：Chain-of-Thought（思维链）

# 第一轮：直接问
"15 * 24 等于多少？"

# 第二轮：让模型一步步想
"15 * 24 等于多少？请一步一步计算，展示中间过程。"

# 尝试更难的题目：
"一个篮子里有 5 个苹果和 3 个橙子。小明拿走了 2 个水果，其中至少一个是苹果。
问：篮子里还剩下几个橙子？"

# 对比"直接回答"和"请一步一步推理"两种 Prompt 的准确率。

实验结论：对于数学和逻辑推理题，要求模型"一步步想"能显著提升准确率。这被称为 Chain-of-Thought Prompting。

错误Prompt 编写的四大雷区

即使掌握了技巧，这些常见错误仍然会让你的 Prompt 功亏一篑：

1. 指令太模糊 —— "帮我优化一下""写个好点的版本"。AI 不知道"好"的标准是什么。就像你对理发师说"剪短一点"，结果可能完全不是你想的。要量化、要具体："将函数行数减少到 20 行以内，并添加类型注解"。

2. 不给上下文 —— 直接丢给 AI 一段代码说"修 Bug"，却不解释这段代码是干什么的、在什么环境下运行。AI 没有读心术。提供背景信息："这是一个处理用户登录的 Flask 视图函数，使用的是 PostgreSQL 数据库"。

3. 要求一次做太多事 —— "设计数据库、写后端 API、做前端页面、部署到服务器"。AI 的上下文窗口有限，一次性塞太多任务会导致每个部分都做得敷衍。分步骤进行，每轮对话聚焦一个明确目标。

4. 不验证输出 —— 盲目相信 AI 的每一行代码，直接复制粘贴到生产环境。AI 会「幻觉」——生成看起来对但实际错误的代码。永远把 AI 当作「草稿生成器」，最终代码必须经过人工审查和测试。

def build_code_explainer():
    return """你是一位资深的 Python 技术专家。

任务：解释以下 Python 代码的核心逻辑。

约束：
- 用中文解释
- 不超过 100 字
- 重点说明代码做了什么，而非逐行翻译语法
- 如果代码有潜在问题，一并指出

代码：
{code}
"""

prompt = build_code_explainer().format(code="sorted([3,1,2])")

def build_code_review_prompt(code):
    return f"""你是一位拥有 15 年经验的资深代码审查员。

任务：请对以下 Python 代码进行全面审查，重点关注 5 个维度。

审查维度：
1. 可读性（命名、注释、结构）
2. 性能（时间/空间复杂度、是否有优化空间）

5. 边界情况（空输入、超大输入、异常值）

输出格式：请用 Markdown 表格输出，每行一个维度，列包括：维度、评分（1-5）、问题描述、修改建议。

代码：
```python
{code}
```"""

测验快速检验一下

1. 单选题：Prompt 工程中"Few-shot"指的是什么？

A. 用很少的 token 调用 API

B. 在 Prompt 中提供几个示例，让模型模仿输出格式和风格

C. 只让模型回答一次，不允许迭代

D. 用很短的 Prompt 提问

点击查看答案

正确答案：B。Few-shot 是在 Prompt 中嵌入少量输入-输出示例，引导模型理解任务格式和期望风格。

2. 判断题：Prompt 越长，AI 的输出质量一定越高。

A. 正确，信息越多越好

B. 错误，过长的 Prompt 会稀释关键信息，且模型对中间部分容易"遗忘"

点击查看答案

正确答案：B。Prompt 的质量取决于信息密度和结构清晰度，而非长度。冗长的 Prompt 可能让模型抓不住重点。

3. 单选题：Chain-of-Thought（思维链）技术的核心思想是？

A. 让模型同时思考多个问题

B. 让模型逐步展示推理过程，而不是直接给最终答案

C. 把 Prompt 链式连接起来发送

D. 用思维导图的形式组织 Prompt

点击查看答案

正确答案：B。Chain-of-Thought 通过在 Prompt 中要求"一步一步想"或在示例中展示推理过程，让模型生成中间推理步骤，从而提高复杂任务的准确率。

4. 单选题：为什么给 AI 设定角色（如"你是一位 Python 专家"）通常能提升回答质量？

A. 模型内部真的有一个"Python 专家"模块被激活

B. 角色描述会改变模型生成下一个词时的概率分布，激活相关知识领域

C. API 会根据角色分配不同的服务器

D. 这只是一种心理安慰，没有实际效果

点击查看答案

正确答案：B。角色设定通过上下文影响模型的条件概率分布，提高技术术语、专业表达和相关知识的激活概率。

5. 判断题：AI 模型真正"理解"了 Prompt 的含义，就像人类一样。

A. 正确，大模型已经具备了真正的理解能力

B. 错误，模型本质上是基于统计模式预测下一个词，并不具备人类的理解和意识

点击查看答案

正确答案：B。尽管大模型的输出看起来很有理解力，但从底层机制看，它只是一个复杂的概率模型，通过模式匹配生成文本，没有意识、意图或真正的理解。
