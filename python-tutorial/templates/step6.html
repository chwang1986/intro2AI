文件与异常

让程序与外部世界安全地交互

场景程序不只是计算器

想象你正在写一份报告：你需要从文件夹里读取参考资料，写完后保存到硬盘。如果笔没墨了、文件被误删了，你不能直接崩溃大哭，而得优雅地处理这些意外。

程序也一样：它需要读写文件来持久化数据，也需要在出错时不至于"炸掉"，而是给用户一个友好的提示。文件操作和异常处理就是解决这两个问题的关键技能。

起源从错误码到异常处理

文件操作的历史可以追溯到 1960 年代。当时计算机科学家们在设计操作系统时，需要一个统一的方式来管理磁盘上的数据。1969 年，Unix 操作系统诞生，提出了著名的"一切皆文件"哲学——无论是键盘输入、屏幕输出还是磁盘数据，都用统一的接口读写。这种设计极大简化了程序与外部世界的交互。

异常处理的概念最早出现在 1960 年代的 PL/I 语言中。1980 年代，C++ 引入了 try/catch 机制，让异常处理成为主流。1995 年 Java 发布时进一步完善了受检异常（Checked Exception）体系。Python 在 1991 年诞生时就内置了异常机制，并且奉行EAFP（Easier to Ask for Forgiveness than Permission，先执行再请求原谅）的哲学——与其每次操作前都做繁琐的检查，不如直接尝试，出错再处理。

Python 的 with 语句则是在 2006 年（Python 2.5）引入的，它源自 C++ 的 RAII（资源获取即初始化）思想，目的是解决程序员总是忘记关闭文件这个老大难问题。

思考没有异常处理的时代，程序员怎么活？

想象一下，你正在用 C 语言写一个读取配置文件的程序。每次调用文件操作函数，你都要检查返回值：

// C 语言风格的错误处理（想象一下这种痛苦）
FILE *f = fopen("config.txt", "r");
if (f == NULL) {
    printf("打开文件失败\n");
    return -1;
}
char buffer[256];
if (fgets(buffer, sizeof(buffer), f) == NULL) {
    printf("读取失败\n");
    fclose(f);  // 别忘了关闭！
    return -1;
}
if (fclose(f) != 0) {
    printf("关闭文件失败\n");
    return -1;
}

这种错误码方式的痛点在于：

1. 容易遗漏：程序员经常忘记检查返回值，错误被静默忽略，程序带着错误数据继续运行。

2. 代码臃肿：业务逻辑和错误处理代码交织在一起，读代码像在读一本被脚注淹没的书。

3. 资源泄漏：一旦某一步出错，后续的资源释放代码（如 fclose）可能被跳过，导致内存泄漏或文件锁死。

异常机制就是为解决这些问题而生的：它让错误无法被忽略（不捕获就崩溃），让错误处理代码与业务逻辑分离（try 里写正常逻辑，except 里处理错误），并且通过 finally 保证清理代码一定执行。

原理文件指针与异常传播链

打开文件时，操作系统会返回一个文件指针，像书签一样标记当前读写到哪个位置。read() 从头读到尾，write() 从当前位置写入。用完文件必须关闭，否则数据可能丢失，内存也会泄漏。

异常是程序运行时的"警报"。当错误发生时，Python 会创建一个异常对象并沿着调用链向上抛，直到被某个 try/except 捕获，否则会终止程序。finally 块则保证无论是否出错，清理代码都会执行。

论证为什么 try/except 比错误码更靠谱？

你可能会问：C 语言用了几十年的错误码不也活得好好的吗？为什么 Python 非要搞异常？让我们从几个维度对比：

1. 错误是否会被静默忽略

错误码方式下，如果你忘了检查返回值，程序会继续运行，可能产生级联错误。而异常如果不被捕获，程序会立刻终止并打印堆栈信息——失败要趁早、要响亮，这是 Python 的设计哲学。

2. 代码可读性

错误码让每一行正常代码都被 if (error) 包围，而 try/except 把"正常路径"和"异常路径"分开，代码结构像小说一样有主线和支线。

3. 跨层传播

假设 A 函数调用 B，B 调用 C，C 出错了。用错误码的话，C 要返回错误，B 要检查并返回，A 也要检查——错误码要层层传递。而异常会自动沿着调用栈向上冒泡，直到被捕获，中间各层无需做任何额外工作。

4. with 语句 vs 手动关闭

手动管理资源就像租了房子后自己记着要交还钥匙；with 语句则像自动门——你走进去，它自动打开；你走出来，它自动关上。无论中间发生什么（甚至发生火灾/异常），门一定会关上。

要点文件操作与异常处理

open() 函数接收文件名和模式（r 读/w 写/a 追加）；with 语句是最安全的文件操作方式，自动处理关闭。try/except/else/finally 构成完整的异常处理结构：try 放可能出错的代码，except 捕获特定异常，else 放成功后的逻辑，finally 放必须执行的清理。

代码示例
# 1. 基本文件读写（不推荐，容易忘记关闭）
f = open("data.txt", "r", encoding="utf-8")
content = f.read()
print(content)
f.close()   # 千万别忘！


# 2. with 语句（推荐）：自动关闭文件
with open("data.txt", "r", encoding="utf-8") as f:
    content = f.read()
    # 离开 with 块，文件自动关闭，即使出错也会关闭


# 3. 逐行读取（适合大文件，不占内存）
with open("data.txt", "r", encoding="utf-8") as f:
    for line in f:
        print(line.strip())   # strip() 去掉换行符


# 4. 写入文件
with open("output.txt", "w", encoding="utf-8") as f:
    f.write("第一行\n")
    f.write("第二行\n")

# 追加模式
with open("output.txt", "a", encoding="utf-8") as f:
    f.write("第三行（追加）\n")


# 5. 完整的异常处理结构
try:
    with open("config.json", "r", encoding="utf-8") as f:
        data = f.read()
        number = int(data)   # 可能抛出 ValueError
except FileNotFoundError:
    print("错误：文件不存在，请检查路径")
except ValueError:
    print("错误：文件内容不是有效的数字")
except Exception as e:
    print(f"未知错误：{e}")
else:
    print(f"读取成功，数值为：{number}")   # 没异常时执行
finally:
    print("清理完成（无论是否出错都会执行）")

# 先创建一个测试文件

    f = open("test.txt", "w")

    # 故意不调用 f.close()
    # 在 Windows 上，这个文件会被 Python 进程锁定，其他程序打不开

manual_open()

# 再试试 with 语句：
with open("test2.txt", "w") as f:
    f.write("world")
# 退出 with 块后，文件一定已经正确关闭，内容一定已经落盘

实验 3：制造并捕获多种异常

# 实验 3a：文件不存在
open("不存在的文件.txt", "r")
# 观察：FileNotFoundError

# 实验 3b：类型错误
int("abc")
# 观察：ValueError

# 实验 3c：捕获异常后继续执行
try:
    result = int("abc")

    print(f"捕获到异常：{e}")
    result = 0
print(f"result = {result}")  # 程序没有崩溃，继续运行！

错误常见陷阱

1. 忘记关闭文件：打开文件不关闭会导致资源泄漏，在 Windows 上还可能锁定文件，其他程序无法访问。

# 错误
f = open("data.txt", "r")
data = f.read()
# 没有 f.close()！

# 正确：使用 with 语句
with open("data.txt", "r") as f:
    data = f.read()

2. 捕获所有异常却不处理：裸 except 会吞掉所有错误，包括系统退出信号，导致程序"死机"式卡死。

# 错误：捕获所有异常但什么都不做
try:
    risky_operation()
except:           # 不要这样写！
    pass          # 静默吞掉所有错误，调试时想死

# 正确：捕获具体异常并记录
except FileNotFoundError as e:
    print(f"文件未找到：{e}")
    raise           # 或重新抛出，让上层处理

3. 用 except 吞掉关键错误：不要在不理解错误的情况下静默处理，至少记录日志或重新抛出。

# 危险：让程序带着错误数据继续运行
try:
    result = calculate()
except Exception:
    result = 0   # 错误的默认值可能导致更大灾难

# 正确：要么处理，要么抛出
except ZeroDivisionError:
    result = float('inf')   # 合理的回退值
    logging.warning("除零，已设为正无穷")

练习 2 参考答案
def safe_divide(a, b):
    try:
        result = a / b
        return result
    except ZeroDivisionError:
        return "除数不能为 0"
    except TypeError:
        return "请输入数字"

print(safe_divide(10, 2))   # 5.0
print(safe_divide(10, 0))   # 除数不能为 0
print(safe_divide(10, "a")) # 请输入数字
练习 4 思路提示
import os

def copy_file(src, dst):
    try:
        with open(src, "r", encoding="utf-8") as f:
            content = f.read()
    except FileNotFoundError:
        return "源文件不存在"
    
    if os.path.exists(dst):
        print(f"警告：{dst} 已存在，将覆盖")
    
    try:
        with open(dst, "w", encoding="utf-8") as f:
            f.write(content)
        return "复制成功"
    except PermissionError:
        return "权限不足，无法写入"
    finally:
        print("复制操作结束")

测验快速检验一下

1. 单选题：with open("a.txt", "r") as f: 的主要作用是？

A. 让代码看起来更高大上

B. 自动关闭文件，即使发生异常也会关闭

C. 让文件读取速度更快

D. 自动创建不存在的文件

点击查看答案

正确答案：B。with 语句通过上下文管理器确保文件在离开代码块时被正确关闭。

2. 判断题：以下代码能捕获所有类型的异常：except:（不带任何异常类型）。

A. 正确，这是捕获所有异常的最佳实践

B. 错误，这样做会吞掉系统退出信号，非常危险

点击查看答案

正确答案：B。裸 except 会捕获包括 KeyboardInterrupt 在内的所有异常，导致程序无法被 Ctrl+C 中断，应避免使用。

3. 单选题：在 try/except/else/finally 结构中，else 块什么时候执行？

A. 无论是否发生异常都会执行

B. 仅当 try 块中没有发生异常时执行

C. 仅当捕获到异常时执行

D. 永远不会执行

点击查看答案

正确答案：B。else 块在 try 块成功完成且没有异常时执行，通常用于放置"成功后"的代码。

4. 单选题：Python 的异常处理机制属于哪种哲学？

A. LBYL（Look Before You Leap，三思而后行）

B. EAFP（Easier to Ask for Forgiveness than Permission，先斩后奏）

C. DRY（Don't Repeat Yourself，不要重复自己）

D. KISS（Keep It Simple, Stupid，保持简单）

点击查看答案

正确答案：B。Python 鼓励先尝试操作，如果失败再捕获异常处理，而不是在操作前做大量条件检查。

5. 判断题：以下代码可以正确读取文件并按行打印：for line in open("data.txt"): print(line)。

A. 正确，这是最简洁的写法

B. 错误，虽然能运行，但没有关闭文件，可能造成资源泄漏

点击查看答案

正确答案：B。虽然语法上能运行，但文件对象没有被关闭。应该使用 with 语句：with open("data.txt") as f: for line in f: print(line)。
