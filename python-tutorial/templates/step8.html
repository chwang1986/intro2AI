调用 AI API

让你的程序学会「开口说话」——与人工智能对话

场景程序也需要「打电话」

想象你是一位忙碌的总经理，每天需要处理大量信息：总结会议记录、翻译外文邮件、分析客户反馈。你不可能事事亲为，于是你拨通了一位超级助手的电话——这位助手就是 AI 模型。

在编程世界里，调用 AI API 就像给这位助手打电话。你的程序拨通号码（发送 HTTP 请求），告诉助手要做什么（传递 Prompt），助手思考后给出回答（返回 JSON 响应）。你只需要学会「打电话」的正确姿势，就能让程序拥有理解语言、生成内容、分析情感的超能力。

例如，你写了一个自动总结文章的小工具：程序读取一篇长文，「打给」AI 说"请用三句话总结以下内容"，AI 立刻返回精炼的摘要——全程无需人工干预。

起源从远程过程调用到 AI 即服务

API（Application Programming Interface）的概念可以追溯到 1960 年代。当时大型机厂商们发现，不同程序之间需要一种标准化的"对话方式"。到了 1990 年代，随着互联网兴起，远程过程调用（RPC）技术让一台计算机可以像调用本地函数一样调用另一台计算机上的功能。

2000 年，Roy Fielding 在他的博士论文中提出了 REST（Representational State Transfer）架构风格，定义了现代 Web API 的设计原则：无状态、资源导向、统一接口。同年，Douglas Crockford 推广了 JSON 格式，让它成为数据交换的事实标准——比 XML 轻量得多。

AI API 的故事则要近得多。2012 年深度学习在 ImageNet 竞赛中取得突破后，AI 模型越来越大、越来越耗算力。2020 年，OpenAI 发布了 GPT-3 并提供 API 访问，开创了"AI 即服务（AIaaS）"的新模式——你不需要购买昂贵的 GPU，只需要一个 API Key 和几行代码，就能调用世界上最强大的语言模型。

思考为什么不让 AI 住在我家电脑里？

刚接触 AI 编程时，很多同学会问："我能不能把模型下载到本地运行？为什么要通过网络'打电话'给一个遥远的服务器？"这确实是个值得思考的问题。

假设你想在本地运行一个 GPT-4 级别的模型，你需要：

1. 硬件成本：一张高端 GPU（如 NVIDIA A100）售价数万元，而运行大模型通常需要多张。作为学生或初创开发者，这笔投入几乎不可承受。

2. 维护成本：模型文件几十到几百 GB，安装配置环境（CUDA、驱动、依赖库）可能要折腾几天。而 API 提供商会帮你搞定这一切。

3. 更新迭代：OpenAI 每隔几个月就发布新版本模型。如果本地部署，你要重新下载、测试、上线。而 API 用户什么都不用做，模型升级是透明的。

4. 弹性伸缩：你的应用突然上了热门，访问量暴增 100 倍。本地服务器会直接崩溃，而云 API 可以自动扩展（当然账单也会自动扩展，这是另一回事）。

当然，本地部署也有优势：数据隐私更好、延迟更低、没有网络依赖、没有按量计费。所以很多企业会在本地部署开源模型（如 Llama、Qwen），而把云端 API 作为备选。作为初学者，从 API 入手是最务实的选择。

论证为什么是 REST + JSON，而不是其他方案？

在调用 AI API 时，你可能没意识到背后是一场"技术路线的胜利"。历史上曾有过多种 competing 的方案，REST + JSON 能在今天成为主流，有其必然性。

1. REST vs SOAP vs GraphQL

2000 年代初期，企业级 Web 服务主要用 SOAP（Simple Object Access Protocol）。SOAP 基于 XML，每条消息都包裹在复杂的信封里，像一个层层包装的俄罗斯套娃。它很严谨，但也很臃肿——一个简单的请求可能要几百行 XML。

REST 则像一张明信片：简单、直接、无状态。你用 HTTP 的 GET/POST/PUT/DELETE 对应增删改查，用 URL 定位资源，用 HTTP 状态码表示结果。对于 AI API 这种"发一个问题，收一个回答"的场景，REST 的简洁恰到好处。

GraphQL（2015 年由 Facebook 开源）允许客户端精确指定需要哪些字段，避免了过度获取数据。但 AI API 的响应结构通常是固定的，GraphQL 的优势不明显，而增加的复杂度却不小。

2. JSON vs XML

XML 是人类可读的，但标签冗余严重。一个简单对象 {"name": "Tom", "age": 20} 用 XML 要写：

<person>
  <name>Tom</name>
  <age>20</age>
</person>

JSON 的体积通常只有 XML 的 30%~40%，解析速度更快，而且和 JavaScript/Python 的数据结构天然对应。对于按 token 计费的 AI API 来说，省下的每一字节都是钱。

3. HTTP 为什么能胜任？

HTTP 是互联网上最成熟、最普及的协议。防火墙普遍放行 80/443 端口，CDN 和缓存基础设施完善，调试工具（curl、Postman、浏览器开发者工具）一应俱全。用 HTTP 作为 AI API 的传输层，就像是把商店开在繁华的商业街——客流（开发者）自然多。

原理一次 API 调用的旅程

当你调用 AI API 时，数据经历了怎样的旅程？就像寄一封特快专递：

你的程序把请求写好（信）→ HTTP 协议负责运输（快递车）→ API 网关检查身份并分拣（邮局）→ AI 模型处理思考（收件人读信写回信）→ 原路返回 JSON 响应（回信送达）。

API Key 相当于你的「身份证」，证明你有权使用这项服务。JSON 则是通用的「信封格式」，无论里面装的是文本、数字还是列表，都能被双方理解。

要点如何正确「拨打电话」

调用 AI API 的核心技能包括：使用 requests 库发送 HTTP 请求、安全地管理 API Key、构造正确的请求体、解析 JSON 响应，以及优雅地处理各种异常情况。

下面的示例展示了如何安全地调用 OpenAI API 进行文本总结。注意我们把 API Key 存放在环境变量中，而不是直接写在代码里——就像你不会把家门钥匙贴在门牌上一样。

代码示例
import os
import requests
from dotenv import load_dotenv

# 1. 从环境变量加载 API Key（安全第一！）
load_dotenv()
API_KEY = os.getenv("OPENAI_API_KEY")
API_URL = "https://api.openai.com/v1/chat/completions"

def summarize_article(article_text):
    """调用 AI API 总结文章，像打电话一样礼貌而清晰"""
    
    # 2. 构造请求头：带上你的「身份证」
    headers = {
        "Authorization": f"Bearer {API_KEY}",
        "Content-Type": "application/json"
    }
    
    # 3. 构造请求体：告诉 AI 要做什么
    payload = {
        "model": "gpt-4o-mini",
        "messages": [
            {"role": "system", "content": "你是一位擅长总结的编辑。"},
            {"role": "user", "content": f"请用三句话总结以下文章：\n\n{article_text}"}
        ],
        "temperature": 0.7,
        "max_tokens": 150
    }
    
    try:
        # 4. 发送 POST 请求（拨号）
        response = requests.post(API_URL, json=payload, headers=headers, timeout=30)
        
        # 5. 检查 HTTP 状态码
        response.raise_for_status()
        
        # 6. 解析 JSON 响应（读取回信）
        data = response.json()
        summary = data["choices"][0]["message"]["content"]
        return summary
        
    except requests.exceptions.Timeout:
        return "错误：请求超时，AI 助手可能太忙了，请稍后再试。"
    except requests.exceptions.ConnectionError:
        return "错误：网络连接失败，请检查你的网络。"
    except requests.exceptions.HTTPError as e:
        if response.status_code == 429:
            return "错误：请求太频繁了，请放慢速度（API 限流）。"
        return f"错误：HTTP {response.status_code} - {e}"
    except KeyError:
        return "错误：响应格式异常，无法提取总结内容。"
    except Exception as e:
        return f"错误：未知异常 - {e}"

# 使用示例
article = """
Python 是一种高级编程语言，由 Guido van Rossum 于 1991 年创建。
它以简洁、易读的语法著称，广泛应用于数据分析、人工智能、Web 开发等领域。
Python 拥有庞大的第三方库生态系统，让开发者能够快速构建复杂应用。
"""

result = summarize_article(article)
print("文章总结：")
print(result)

错误这些坑千万别踩

新手调用 AI API 时，总会犯一些「低级错误」。下面三个是最常见、也最危险的：

1. 硬编码 API Key —— 把密钥直接写在代码里，然后不小心提交到了 GitHub。这就像把银行卡密码写在名片上到处发。攻击者拿到你的 Key 后，可以疯狂调用 API，让你一夜之间欠下巨额账单。务必使用环境变量或专用密钥管理服务。

2. 不处理网络异常 —— 以为网络永远畅通无阻。现实中，WiFi 会断、服务器会挂、DNS 会失效。如果不加 try-except 和超时设置，程序遇到一点网络波动就会崩溃，用户体验极差。永远为网络请求加上异常处理和超时参数。

3. 忽略 API 限流（Rate Limit） —— 像机关枪一样连续发送请求，结果被服务商封禁。API 提供商通常限制每分钟/每小时的调用次数。不处理 429 状态码，不仅会导致请求失败，还可能触发账号风控。实现指数退避重试机制，遇到限流时主动等待。

下面三个实验不需要你有真实的 AI API Key，我们用公开的服务或模拟数据来理解 HTTP 请求的本质。

实验 1：用 curl 发送一个 HTTP 请求（观察原始数据）

# 打开终端，输入以下命令（不需要 API Key）
curl -s "https://httpbin.org/get" | python3 -m json.tool

# 观察输出：
# - "url" 字段显示了你请求的地址
# - "headers" 显示了你发送的请求头（如 User-Agent）
# - "origin" 显示了你的公网 IP

# 试试 POST 请求：
curl -s -X POST "https://httpbin.org/post" \
  -H "Content-Type: application/json" \
  -d '{"message": "hello"}' | python3 -m json.tool

# 观察：服务器把请求体原封不动地返回给你了（echo 服务）

实验结论：HTTP 请求本质上是"发送一段文本，接收一段文本"。headers 是信封上的信息，body 是信的内容。

练习 2 参考答案
import time

def ask_ai_with_retry(question, max_retries=3):
    for i in range(max_retries):
        try:
            return ask_ai(question)  # 假设 ask_ai 是已有的函数
        except Exception as e:
            print(f"第 {i+1} 次尝试失败: {e}")
            if i < max_retries - 1:
                time.sleep(1)
    return "请求失败，请稍后重试"

result = ask_ai_with_retry("你好")
print(result)
练习 4 思路提示
import time
import random

class MockAIClient:
    def __init__(self):
        self.call_count = 0
        self.total_tokens = 0
        self.responses = {
            "你好": "你好！有什么可以帮你的？",
            "天气": "今天天气不错，适合出门。",
        }
    
    def chat(self, prompt):
        self.call_count += 1
        time.sleep(0.1)  # 模拟网络延迟
        
        if random.random() < 0.1:
            raise Exception("模拟网络故障")
        
        response = self.responses.get(prompt, f"这是对'{prompt}'的模拟回复")
        tokens = len(prompt) + len(response)
        self.total_tokens += tokens
        return response
    
    def get_stats(self):
        return {
            "calls": self.call_count,
            "tokens": self.total_tokens
        }

测验快速检验一下

1. 单选题：为什么把 API Key 放在环境变量中，而不是直接写在代码里？

A. 环境变量里的 Key 会自动加密

B. 避免密钥随代码一起提交到版本控制系统，防止泄露

C. 环境变量比代码执行得更快

D. 因为 API 文档要求必须这样做

点击查看答案

正确答案：B。代码仓库可能被公开或共享，硬编码密钥极易泄露。环境变量将配置与代码分离，是安全最佳实践。

2. 判断题：HTTP 状态码 429 表示请求成功，只是响应比较慢。

A. 正确

B. 错误，429 表示请求过于频繁，触发了限流

点击查看答案

正确答案：B。429 Too Many Requests 是标准的限流状态码，意味着你需要降低请求频率或实现退避重试。

3. 单选题：REST API 的设计核心思想是什么？

A. 使用复杂的 XML 信封包装所有请求

B. 资源导向，用 HTTP 方法和 URL 操作资源，无状态通信

C. 每个请求必须包含完整的用户会话状态

D. 只支持 POST 方法

点击查看答案

正确答案：B。REST 的核心是资源导向和无状态。HTTP 方法（GET/POST/PUT/DELETE）对应资源的增删改查操作。

4. 单选题：在 AI API 调用中，temperature 参数的作用最接近什么？

A. 控制服务器的 CPU 温度

B. 控制模型输出的随机性和创造性程度

C. 控制请求的超时时间

D. 控制返回结果的最大长度

点击查看答案

正确答案：B。temperature 越高，模型采样时越倾向于选择概率较低的词，输出越有创意但也越不稳定。

5. 判断题：API 返回的 JSON 数据可以直接使用，不需要做任何错误检查。

A. 正确，API 的响应格式是固定的，永远可靠

B. 错误，网络问题、服务端变更都可能导致响应格式异常或字段缺失

点击查看答案

正确答案：B。生产代码必须对 API 响应做校验，使用 try/except 捕获解析异常，检查关键字段是否存在。
